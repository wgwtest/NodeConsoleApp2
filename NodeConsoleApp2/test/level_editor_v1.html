正式关卡编辑器 v1

本页是当前工程的正式关卡编辑器主入口，负责在浏览器内直接加载 `levels.json`，按“关卡列表 + 详情表单 + 波次 / 敌人池”方式编辑关卡内容，并把结果导出成运行时可消费的包产物。

它不是主流程页面，也不是故事关卡选择页。它的职责是编辑关卡内容、生成 JSON 产物、把产物写入 runtime override。真正验证运行时是否消费了这份产物，应跳转到 `level_runtime_probe.html` 或刷新 `mock_ui_v11.html`。

本页用来验证什么

  1. 能否直接从默认 `levels.json` 加载出完整关卡包。
  2. 能否修改关卡基本字段、章节 fallback 元数据、解锁规则、胜败条件、波次、敌人池成员。
  3. 能否导入现有 JSON，再继续编辑并导出新的关卡包 JSON。
  4. 能否把导出产物写入 runtime override，供独立运行时消费。

与其他页面的关系

  • `level_editor_v1.html`：正式编辑器主入口。
  • `level_runtime_probe.html`：独立运行时消费验证页。
  • `mock_ui_v11.html`：主流程烟测页，验证 override 是否不破坏战斗流程。
  • 旧 `level_editor_io_test.html`：历史 I/O 注入页，用来对照说明导入导出链，但不再作为正式入口。

关键字段怎么填

  • `unlockRules`：`after_levels_cleared` 表示需要满足前置关卡；多个 ID 用逗号或换行分隔。
  • `victoryCondition`：当前支持 `defeat_all_enemies` 和 `survive_rounds`。
  • `failureCondition`：当前支持 `player_hp_zero` 和 `body_part_broken`。
  • `background`：这里填写的是战斗场景背景，不是章节地图背景；地图背景在地图编辑器维护。
  • `敌人风格标签`：用逗号或换行分隔多个标签。
  • `敌人池成员`：每行一条，格式为 `templateId@position`。
  • `关卡 ID`：当前版本只读展示，不支持直接改 ID。
按钮 会做什么 为什么这能证明
`重新加载默认关卡包` 重新从 `../assets/data/levels.json` 生成一个全新的编辑工作区。 证明编辑器依赖的是关卡包文件，而不是某次测试残留的内存对象。
`保存当前关卡` / `保存波次与敌人池` 把详情表单和波次表单写回工作区，再刷新列表与校验面板。 证明编辑页本身具备结构化修改能力，而不是单纯展示 JSON；同时也证明关卡页只维护运行时关卡与章节展示元数据，不直接维护地图图结构。
`导出 JSON` / `下载 JSON` 把当前工作区重新封装成 wrapped `levels.json` 结构。 证明编辑结果已经具备产物化能力，可交给运行时或其他工具消费。
`应用 Runtime Override` 把当前导出的关卡包写入浏览器 localStorage override。 这是“编辑器产物 -> 运行时消费”的桥接步骤；之后由 runtime probe 读取并实例化。
`导入现有 JSON` 读取外部 JSON 文件，重建关卡编辑工作区。 证明正式编辑器可以从已有产物继续编辑，而不是只能从默认样例起步。
