Buff Runtime Probe

Local runtime probe for the Buff pack bridge. This page reloads DataManagerV2.js, resolves buff_counter, and verifies that Buff runtime consumes the active pack artifact instead of relying on editor memory state.

本页用途

运行时消费页本页是“运行时消费页”，会重新加载 DataManagerV2.js、重建 BuffRegistry / BuffSystem / Probe Actor，然后验证运行时是否真的消费了当前 Buff pack。

它和 buff_editor_io_test.html 的职责不同：I/O 页负责写入 override，本页负责在一个独立运行时里读出 override 并真正应用 Buff。

字段怎么读

  • Pack Source：`file` 表示走默认文件，`override` 表示运行时已经读到 override 存储。
  • Resolved Name：运行时真正解析出来的 Buff 名称。
  • Active Buffs：把该 Buff 实际加到 Probe Actor 后，运行时对象里最终生效的结果。

你在本页应该看什么

  1. 点 Reload Runtime 后，页面会重新读取当前 Buff pack。
  2. 若前一页已写入 override，则 Pack Source 应显示为 override。
  3. Resolved Name 应变成样本名 反击·运行时覆写样本。
  4. 点 Apply Probe Buff 后，Active Buffs 里应出现同样的名字。

为什么这能证明

本页会重新加载 DataManagerV2.js，不共享 I/O 页内存。如果这里仍能读到 override 名称，说明运行时消费的是“存储里的产物”而不是“上一页的内存对象”。再把 Buff 真正加到 Active Buffs 里，就证明运行时消费链闭环成立。

按钮说明

按钮 点击后会做什么 为什么这能证明
Reload Runtime 重新加载 DataManagerV2.js 和 Buff 运行时对象。 确保当前显示的是一次全新的运行时解析结果，而不是旧页面残留。
Apply Probe Buff 把 buff_counter 真正加到 Probe Actor 身上。 证明“解析出来的定义”不仅能看，还能被运行时 BuffManager 真正消费。
Clear Active Buffs 重建 Probe Actor，清空当前运行时 Buff 列表。 便于重复验证，但不会清掉 override 存储本身。
Open buff_editor_io_test.html 回到产物注入页继续写入或清除 override。 用来和上一页配合，形成“写入 - 读取 - 应用”的完整验证链路。
