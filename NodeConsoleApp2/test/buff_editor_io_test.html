Buff Editor I/O Test Module

用于验证 Buff 数据导入、导出，以及“当前页临时写入一份 Buff 数据后，其他页面刷新后是否也能用上这份数据”。

本页用途

产物注入页本页是“产物注入页”，负责把 buffs_v2_7.json 这份 Buff 配置数据载入当前页面，并在你点击确认后，把当前页里的 Buff 数据另存为一份浏览器本地临时数据。

它不直接证明运行时已经用上这份数据。真正负责证明“运行时已经读到这份临时 Buff 数据”的页面是 buff_runtime_probe.html。

验证链路

  1. 先从文件加载默认 Buff 数据。
  2. 用 Inject Runtime Probe Sample 给 buff_counter 打上明显的覆写样本名称。
  3. 用 Apply Runtime Override 把当前页里的 Buff 数据写入浏览器本地临时存储。
  4. 再去 buff_runtime_probe.html 刷新运行时，看另一页是否也读到这个名称。

你在本页应该看什么

  1. 默认加载后，列表应显示 19 条 Buff。
  2. 注入样本后，buff_counter 的名称应变成 反击·运行时覆写样本。
  3. 应用 override 后，底部应显示 Runtime override active。

为什么这能证明

Inject Runtime Probe Sample 只修改当前页面内存，不会自动改别的页面。只有点击 Apply Runtime Override 后，另一张页面在刷新或重新打开后仍能读到同样的样本名称，并且能把这个 Buff 真正加到运行时列表里，才说明这份临时 Buff 数据已经被运行时用上。

几个关键词是什么意思

  • 这里说的 Buff 数据，就是当前页载入的 buffs_v2_7.json 这份 Buff 配置数据。
  • Apply Runtime Override 的意思不是“重写 buffs_v2_7.json 文件”，而是“把当前页里的 Buff 数据另存为一份临时数据”。
  • 这里说的本地 override 存储，不是本地文件，也不是几个页面共用的一段内存；它实际是浏览器 localStorage 里的一段 JSON 数据。
  • 这里的“本地”是指当前浏览器对这个站点保存的数据，不是工程目录里的文件。

与其他页面关系

  • buff_editor_io_test.html 本身是一个单页工具。点击加载、注入样本这类操作，只会修改当前页面内存。
  • 它不会直接修改 buff_editor_v4.html。因为 buff_editor_v4.html 每次都是重新读取默认文件 buffs_v2_7.json，不会读取这份浏览器临时数据。
  • 只有点击 Apply Runtime Override 后，才会把当前页里的 Buff 数据写入浏览器本地临时存储。此后只会影响那些会重新加载 Buff 数据的页面，例如 buff_runtime_probe.html，以及刷新后重新进入运行时的 mock_ui_v11.html。
  • 同一浏览器里打开的其他页面，在刷新或重新打开后可以重新读到它；已经打开但没有刷新的页面，不会自动同步。
  • 这就是为什么验收步骤会在不同页面之间跳转。目的不是跨页拼操作，而是证明“这一页写入浏览器临时 Buff 数据后，另一页刷新后真的会用上它”。

按钮说明

按钮 点击后会做什么 为什么这能证明
Load from ../assets/data/buffs_v2_7.json 从默认文件加载 Buff pack 到本页内存。 证明默认 pack 可被工具页正确读取，是后续注入和导出的基线。
Inject Runtime Probe Sample 把当前内存中的 buff_counter 改成一个易辨识的样本名。 给后续验证制造一个“只有 override 才会出现”的明显标记。
Apply Runtime Override 把当前页里的 Buff 数据写入浏览器 localStorage 里的临时 JSON 数据槽，不会改磁盘上的 buffs_v2_7.json 文件。 这是把“当前页里的 Buff 数据”变成“其他页面刷新后也能重新读到的数据”的关键一步。
Clear Runtime Override 清空浏览器里的这份临时 Buff 数据，恢复默认文件来源。 证明这条验证链可重复执行，不会永久污染默认数据。
Open buff_runtime_probe.html 打开独立运行时消费页。 下一页会重新加载运行时，而不是沿用本页内存，这样才能验证“别的页面刷新后也能读到这份临时 Buff 数据”。
