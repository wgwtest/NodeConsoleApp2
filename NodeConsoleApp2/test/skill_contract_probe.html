本页用途

批量校核页本页是“批量校核页”，职责不是试玩主流程，而是把当前运行时真正能读到的技能契约一次性摊开，帮助快速判断“技能说明写了什么、配置声明了什么、运行时可直接推导出什么”。

这里的批量校核基于 DataManagerV2.js 的 getSkillContractCatalog() 与 getSkillContractIssues()。它会批量列出技能契约摘要与异常，但不会直接修改技能配置，也不会替代 mock_ui_v11.html 的主流程观察。

你在本页应该先看什么

  1. 先看“技能状态总表”：它直接回答“哪些技能正常、哪些已修复、哪些还没修”。
  2. 再看“WBS-3.3.3 异常分批闭环”：确认当前节点负责哪些批次、哪些已关闭、哪些留到后续。
  3. 最后再看“仍有问题的技能”和“技能契约矩阵”，定位具体 issue code 与运行时结构。

为什么这能证明

因为本页不复用技能编辑器内存，也不靠手工猜测，而是直接走运行时的数据装载边界。如果这里能稳定列出相同的异常，就说明问题已经从“主观感觉”变成了“可重复读取的结构事实”。

与其他页面关系

  • mock_ui_v11.html：验证玩家在主流程里是否真的能看到效果。
  • codex_regression_runner.html：验证自动回归是否持续通过。
  • 本页：批量收口“哪些技能天然存在契约冲突”。

为什么它不能单独证明数值平衡正确

本页能证明的是“契约是否一致、异常是否可归类”，不能单独证明战斗数值一定合理，更不能替代真实战斗观察。所以它是结构校核页，不是最终平衡结论页。

问题码说明

Issue Code 含义 为什么值得先看
selection_count_exceeds_candidates 选择数量超过了可选部位数量。 这类问题通常意味着技能配置本身已越界，运行时再怎么消费都存在歧义。
single_selection_count_invalid single 模式却给了不等于 1 的数量。 这会直接破坏 UI 选择语义与规划阶段的理解。
tag_scope_mismatch 标签里的 SCOPE_* 与真实 target.scope 不一致。 说明文档、检索、平衡统计都会被带偏。
unexpected_effect_tag 标签声明了某种效果，但 actions / buffRefs 无法直接推导出来。 这类问题常见于说明漂移或旧标签残留。
missing_effect_tag actions / buffRefs 已经体现某种效果，但 tags 没写出来。 这会让后续检索、设计分析和异常归类失真。

技能状态总表

先看这里如果你只关心“哪些技能有问题、哪些已经修过、为什么还没修完”，先看下表，不需要先理解批次或 issue code。

状态（正常，已修复，未修复）的判定口径：正常 表示当前未发现契约异常；已修复 表示当前节点已正式修过该技能的结构问题，若仍有后续批次问题，会继续写在“问题原因 / 修改方法”列；未修复 表示当前仍有未收口问题。
