Level Editor I/O Test Module

Tests loading wrapped level pack JSON, exporting as pack, and verifying the runtime override bridge independently from battle UI memory state.

本页用途

产物注入页作者样本工具页本页是“产物注入页”，负责把 关卡 pack 载入当前工具页，并在你确认后把当前内存里的 pack 写入本地 runtime override 存储。

它不直接证明运行时实例化成功。真正验证“运行时是否吃到这份关卡配置并能实例化敌人”的页面是 level_runtime_probe.html。

它不属于故事推进关卡，也不属于验收样本入口；它只服务于作者验证“编辑结果如何变成运行时可消费产物”。

验证链路

  1. 先从文件加载默认关卡 pack。
  2. 用 Inject Runtime Probe Sample 把 level_1_1 改成一个明显的样本名，并把首个敌人池改成 pool_runtime_probe。
  3. 用 Apply Runtime Override 把当前内存 pack 写入本地 override 存储。
  4. 再去 level_runtime_probe.html 重新加载运行时，看独立页面是否也解析出同样的关卡名和敌人池成员。

你在本页应该看什么

  1. 默认加载后，列表应显示 Loaded Levels (8)。
  2. 注入样本后，level_1_1 的名称应变成 幽暗森林边缘·运行时覆写样本。
  3. 同一条目的敌人池应变成 pool_runtime_probe。
  4. 应用 override 后，底部应显示 Runtime override active。

为什么这能证明

Inject Runtime Probe Sample 只修改本页内存，不会自动影响运行时。只有点击 Apply Runtime Override 后，另一张独立页面重新加载运行时还能读到相同的关卡名和敌人池成员，才说明“关卡产物 -> override 存储 -> 运行时实例化”链路真的成立。

与其他页面关系

  • level_editor_io_test.html 本身是一个单页工具。点击加载、注入样本这类操作，只会修改当前页面内存。
  • 它不会直接修改 mock_ui_v11.html 当前已打开页面的内存状态。因为已经打开的主流程页不会自动同步本页内存。
  • 只有点击 Apply Runtime Override 后，才会写入本地 override 存储。此后只会影响通过 DataManager 重新装载关卡 pack 的页面，例如 level_runtime_probe.html，以及刷新或重新进入运行时后的 mock_ui_v11.html。
  • 它不属于验收样本入口。验收样本入口仍在 mock_ui_v11.html 的运行时菜单里。
  • 这就是为什么验收步骤会在不同页面之间跳转。目的不是跨页拼操作，而是证明“这一页写入，另一页独立重载后仍能实例化”。

按钮说明

按钮 点击后会做什么 为什么这能证明
Load from ../assets/data/levels.json 从默认文件加载关卡 pack 到本页内存。 证明默认 pack 可被工具页正确读取，是后续注入和导出的基线。
Inject Runtime Probe Sample 把当前内存中的 level_1_1 改成样本名，并把首个敌人池替换为 pool_runtime_probe。 给后续验证制造一个“只有 override 才会出现”的明显标记。
Apply Runtime Override 把当前内存 pack 写入本地 override 存储。 这是把“编辑结果”转成“运行时可消费产物”的关键一步。
Clear Runtime Override 清空本地 override 存储，恢复默认文件来源。 证明这条验证链可重复执行，不会永久污染默认数据。
Open level_runtime_probe.html 打开独立运行时消费页。 下一页会重新加载运行时，而不是沿用本页内存，这样才能验证真正的消费边界。
