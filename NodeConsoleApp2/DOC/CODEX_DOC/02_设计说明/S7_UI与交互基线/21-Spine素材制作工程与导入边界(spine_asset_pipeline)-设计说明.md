# Spine素材制作工程与导入边界设计说明

创建时间：2026-04-07

最后整理时间：2026-04-07

状态：`当前有效 / 本文同时承担可行性分析报告`

## 1. 结论摘要

针对当前工程，如果目标已经明确为 `Spine`，那么把素材制作、Rig、动画预览、导出校验从主游戏工程中拆出去，作为相对独立的工作区或独立工程，是合理且推荐的做法。

核心结论如下：

1. `NodeConsoleApp2` 只需要消费素材工程的正式产物，不应直接承担 `Spine Editor` 工作流本身。
2. 最低建议是建立独立工作区；中长期建议是独立仓库或独立工程。
3. 当前游戏侧只需接入战斗渲染适配层，不应让 `CoreEngine / Timeline / DataManager / Save` 直接感知 `Spine` 细节。
4. 当前工程已经具备把 `Spine` 限定在战斗展示层的技术前提，但还不具备直接从现有 PNG 自动拆成生产级纸娃娃的素材前提。
5. 单靠 `Codex` 可以完成大量“技术链路、目录契约、导出校验、预览页、导入适配”工作，但无法可靠替代正式美术生产中的“单张立绘拆解、补绘、关节遮挡修复、风格统一”。

## 2. 当前工程现状判断

### 2.1 展示层边界已经比较适合隔离

当前战斗界面中的角色容器已经独立于规则层存在：

1. `mock_ui_v11.html` 中角色展示区为：
   - `.fighter`
   - `.character-sprite-container`
   - `.sprite-layer.body / armor / weapon`
2. 当前展示层控制器已经独立于规则层：
   - `BattlePresentationController`
   - `FighterPresenter`
   - `BattleAnimationDriver`
3. 这意味着后续把 DOM 图片层替换成 `Spine runtime`，理论上只需要替换“战斗展示适配器”，不必回改结算主链。

### 2.2 当前素材并不是正式纸娃娃素材

当前工程中已存在角色 PNG：

1. `source/character/主角-001-状态001-正常状态.png`
2. `source/character/敌人-001-状态001-正常状态.png`
3. `source/character/敌人-001-状态002-丢失盾牌.png`
4. `source/character/敌人-001-状态003-盾牌破损.png`

但这些素材当前属于“平面成品图 / 状态图”，不是“分件图”：

1. 角色关节背后的隐藏区域并不存在于原图中
2. 武器、前臂、盾牌、披风、护甲等未按 Rig 友好的粒度拆层
3. 现有 `armor / weapon` 层在主界面里主要还是展示占位层，不是正式装备部件库

所以：

1. 当前系统“结构上接近纸娃娃”
2. 当前素材“还不是纸娃娃生产资产”

## 3. 为什么推荐拆成独立工程

如果最终方向是 `Spine`，继续把这部分工作直接塞进主工程，会很快遇到边界污染问题。

### 3.1 与主游戏工程的职责不同

主游戏工程负责：

1. 规则
2. 战斗状态
3. 存档
4. UI 主流程
5. 展示层接入

素材工程负责：

1. 角色分件
2. Rig
3. 动画
4. 导出
5. 导出校验
6. 预览

这两类职责天然不是同一层。

### 3.2 工具链依赖不同

`Spine` 方案会引入新的依赖面：

1. `Spine Editor`
2. `Spine` 导出格式
3. `atlas / texture pages`
4. 预览运行时
5. 角色资产版本管理

这些内容与当前 `NodeConsoleApp2` 的 HTML/CSS/JS 主流程不是同一种工程对象。

### 3.3 可以更严格地限制影响面

如果素材工程独立，主游戏侧只要遵守输入输出契约：

1. 主游戏只读取导出物
2. 主游戏不读 `.spine` 工程源文件
3. 主游戏不依赖 `Spine Editor`
4. 只有战斗渲染层知道 `Spine`

那么你最关心的约束就容易成立：

1. 不影响战斗渲染以外的其他功能
2. 后续即使素材工程反复迭代，也不会直接冲击主流程、关卡、Buff、技能和存档

## 4. 推荐工程划分

## 4.1 推荐的两工程模型

推荐长期采用以下模型：

### 工程 A：主游戏工程

`NodeConsoleApp2`

职责：

1. 维护战斗规则与 UI 主流程
2. 提供可供素材工程消费的角色清单 / 配置清单
3. 读取导出后的 `Spine bundle`
4. 在战斗展示层中按事件驱动播放动画

### 工程 B：素材制作工程

建议名称示例：

1. `NodeConsoleApp2-SpineAssets`
2. 或 `NodeConsoleApp2-BattlePresentationLab`

职责：

1. 读取角色目录
2. 管理 `Spine` 源工程
3. 预览动画
4. 导出 `Spine bundle`
5. 做导出产物校验
6. 生成供主工程读取的 manifest

## 4.2 过渡期建议

如果现在还不想立刻新开完整仓库，建议先做：

1. 独立工作区
2. 独立目录
3. 独立文档根
4. 独立运行时预览页

也就是先达到“逻辑独立”，再决定是否提升为“仓库独立”。

推荐过渡期目录模型：

1. `NodeConsoleApp2/`
2. `NodeConsoleApp2-SpineLab/`

等输入输出契约稳定后，再看是否独立成仓库。

## 5. 输入输出契约建议

## 5.1 主工程给素材工程的输入

你提出“素材制作工程能够加载 enemy 列表和 player 配置列表”，这个方向是对的。

当前工程已经有统一内容注册入口：

1. `assets/data/config.json`
2. `player.json`
3. `enemies.json`

因此素材工程可以读取：

1. 玩家基础配置列表
2. 敌人基础配置列表
3. 可选的关卡列表，用于做战斗预览编排

但要注意：

1. 当前 `player / enemy` 配置主要是玩法数据
2. 还不是正式的视觉数据目录

所以推荐新增一个单独的视觉目录层，而不是把视觉信息直接混进玩法字段。

建议增加的视觉元数据示例：

1. `presentationId`
2. `spineSkeletonId`
3. `defaultSkin`
4. `weaponSocket`
5. `shieldSocket`
6. `bodyVariant`
7. `factionVisualTheme`
8. `scaleProfile`
9. `anchorProfile`

推荐原则：

1. 玩法数据负责“这个角色是谁、怎么打”
2. 视觉数据负责“这个角色长什么样、读哪个骨骼包”

## 5.2 素材工程输出给主工程的产物

主工程不应消费 `Spine Editor` 源文件本身，而应消费导出物。

推荐正式导出物：

1. `skeleton.skel` 或 `skeleton.json`
2. `skeleton.atlas`
3. `texture pages (.png)`
4. `character_manifest.json`
5. `bundle_index.json`

其中 `character_manifest.json` 建议包含：

1. 角色 ID
2. 骨骼包 ID
3. 默认皮肤
4. 可用动画名
5. 关键插槽名
6. 推荐缩放
7. 推荐站位偏移
8. 可替换部件清单

主工程建议只读取：

1. `bundle_index.json`
2. 单角色 `manifest`
3. 运行时导出文件

而不反向依赖素材工程目录结构。

## 6. 当前工程为什么适合做“只消费产物”的接入

当前 `NodeConsoleApp2` 已经具备如下条件：

1. 战斗展示层已经独立存在
2. 动画可以关闭，不阻断主流程
3. probe 页已经能单独验证展示层

这意味着后续接入可以继续沿同一原则推进：

1. `Spine` 只在 battle render backend 中存在
2. `BattlePresentationController` 继续只管事件语义，不直接碰 `Spine API`
3. 再包一层 `BattleSpinePresenter / SpineRenderAdapter`
4. 不允许把 `Spine` 细节反向泄漏到 `CoreEngine`

## 7. Spine 方案的技术依据

根据官方资料，`Spine` 本身就是围绕“骨骼 + 插槽 + 附件 + 皮肤 + 运行时”设计的，这与“纸娃娃、换装、武器替换、动作复用”的需求高度吻合。

官方可直接支撑当前判断的点包括：

1. `Bones` 用于层级变换与动作驱动
2. `Skins` 用于按部件混装整角色
3. `Images` 与 `Import PSD` 支持从图层工作流进入编辑器
4. 官方维护 `spine-pixi-v8` 运行时，可接入 Web 渲染

官方参考：

1. Spine Bones:
   - <https://esotericsoftware.com/spine-bones>
2. Spine Skins:
   - <https://esotericsoftware.com/spine-skins>
3. Spine Images:
   - <https://esotericsoftware.com/spine-images>
4. Spine Import PSD:
   - <https://esotericsoftware.com/spine-import-psd>
5. Spine Pixi Runtime:
   - <https://esotericsoftware.com/spine-pixi>

## 8. 成本分析

## 8.1 技术成本

如果只做“单独素材工程 + 主工程消费导出物 + 战斗页独立接入”，纯技术成本是中等，不是最高风险项。

粗估单人技术投入：

1. 独立素材工程脚手架、目录和导出契约：
   - `2 ~ 4` 个工作日
2. 预览页、导出校验、bundle manifest：
   - `2 ~ 4` 个工作日
3. 主工程中的 `Spine` 战斗展示适配器与回退机制：
   - `2 ~ 5` 个工作日

如果只算工程和代码，不算美术生产，技术侧是可控的。

## 8.2 素材成本

真正高成本的是素材拆解，不是代码接入。

分两种情况：

### 情况 A：你有 PSD / 分层源文件

这是可控方案。

粗估：

1. 一个玩家样本 + 一个敌人样本
2. 分层整理、Rig、4 个基础动作、导出、修正
3. `5 ~ 10` 个工作日内可以做出首版可验证样本

### 情况 B：你只有当前这些平面 PNG

这是高风险方案。

因为需要额外做：

1. 人工拆层
2. 遮挡补绘
3. 关节转动补边
4. 武器 / 护甲重新分件
5. 风格和轮廓修正

粗估：

1. 一个玩家样本 + 一个敌人样本
2. 只靠现有平面图重建可用部件
3. 需要 `10 ~ 20+` 个工作日，且质量不稳定

换句话说：

1. 技术接入不是最贵的
2. 资产重建才是最贵的

## 8.3 工具与许可证成本

`Spine` 还会带来额外工具成本：

1. `Spine Editor` 许可证
2. 可能的图像编辑软件成本
3. 导出格式与运行时版本对齐的维护成本

因此从投入结构上看：

1. 代码成本：中
2. 文档和契约成本：中
3. 许可证和工具成本：中
4. 美术拆解与修正成本：高

## 9. 单靠 Codex 能否完成全部流程

结论是：

1. `Codex` 可以完成“很大一部分”
2. 但不能被视为“可完全替代正式美术生产的唯一工具”

## 9.1 Codex 可以高把握完成的部分

1. 独立素材工程脚手架
2. 输入输出契约设计
3. 目录规范
4. 导出清单与校验脚本
5. 预览页
6. 主工程导入适配器
7. 回退开关与探针页
8. 文档与版本管理
9. 与 `player / enemy` 数据目录的同步逻辑

## 9.2 Codex 可以辅助但不能完全保证质量的部分

1. 单张 PNG 的自动分割
2. 关节遮挡区补绘
3. 前后层关系修复
4. 复杂装备拼装
5. 动作审美与节奏微调
6. 大量角色的风格一致性维护

这些环节不是“写脚本”就能解决，而是强依赖美术判断。

## 9.3 最现实的判断

如果只有当前平面 PNG：

1. `Codex` 可以做原型级拆解辅助
2. `Codex` 不能承诺产出“可交付级最终美术”

如果你有 PSD / 图层源文件：

1. `Codex` 的可完成度会大幅提升
2. 因为这时主要工作变成“命名规范、导出流程、适配和验证”
3. 而不是“从平面图逆向重建隐藏部件”

所以最务实的判断是：

1. `Codex` 足以完成工程和流程
2. `Codex` 不足以独立承担高质量原画拆解与正式装配美术生产

## 10. 可行路线建议

如果你准备正式推进，推荐按下面顺序做，而不是一上来把主工程改成 `Spine runtime`。

### 阶段 P0：先建独立素材工程

目标：

1. 独立目录
2. 独立预览页
3. 独立导出目录
4. 明确输入输出契约

完成标准：

1. 能读取 `player / enemy` 目录
2. 能导出带 manifest 的 bundle
3. 主工程此时还不改

### 阶段 P1：做一个玩家 + 一个敌人的样本

目标：

1. 只做一个玩家样本
2. 只做一个敌人样本
3. 只做 `idle / attack / hit / recover`

完成标准：

1. 素材工程里能独立预览
2. 导出物结构稳定
3. 文档、命名、bundle 契约稳定

### 阶段 P2：主工程只做战斗展示接入

目标：

1. 当前 battle render backend 增加 `Spine` 适配器
2. 保留现有 DOM 展示作为 fallback
3. Feature flag 控制开关

完成标准：

1. `Spine` 只在战斗页生效
2. 关闭 `Spine` 时主流程保持不变
3. 规则层、存档、技能、Buff、关卡均不需要知道 `Spine`

### 阶段 P3：再考虑批量角色与装备

只有在 `P0 ~ P2` 稳定后，才值得继续做：

1. 多敌人骨骼模板
2. 皮肤混装
3. 装备替换
4. 更细粒度部件

## 11. 最终建议

最终建议明确如下：

1. 是的，这部分应当被视为相对独立的工程或至少独立工作区。
2. 是的，当前工程只应该消费素材制作工程的正式产物。
3. 是的，素材制作工程可以读取 `enemy` 列表和 `player` 配置列表，但建议新增独立视觉元数据层，而不是直接复用玩法字段。
4. 如果目标是 `Spine`，应优先先建“独立素材工程 + 导出契约 + 样本预览”，再做主工程接入。
5. 只依赖 `Codex` 并不能可靠完成最终级原画拆解，但足以完成工程链路、导出体系、预览体系、导入适配和大量前置自动化工作。

当前最合理的推进方式不是：

1. 直接在主工程里强塞 `Spine`

而是：

1. 先把 `Spine` 素材工作流独立出来
2. 用一个玩家和一个敌人的样本把输入输出契约跑通
3. 再让主工程只消费这套产物
