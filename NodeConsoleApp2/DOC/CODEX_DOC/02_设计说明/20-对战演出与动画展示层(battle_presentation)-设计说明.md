# 对战演出与动画展示层设计说明

创建时间：2026-04-07

最后整理时间：2026-04-11

状态：`当前有效`

## 1. 设计目标

本设计解决的不是“战斗规则怎么结算”，而是“规则已经结算后，画面如何把结果更清楚地表达出来”。

当前合理的对战演出层应满足：

1. 与 `CoreEngine / TimelineManager / DataManagerV2` 严格低耦合
2. 只消费事件与只读运行时快照，不写回规则层
3. 即使动画全部关闭、失败或未加载，游戏主流程仍可正常推进
4. 可以在独立 probe 页中单独验证，不依赖完整战斗主流程才能看见效果
5. 能覆盖玩家最容易观察的最小演出集合：
   - 攻击前冲
   - 受击闪白 / 抖动
   - 伤害飘字
   - 血条缓动
   - 治疗回弹
   - Buff / Debuff 上身提示
   - 护甲受损 / 修复高亮

## 2. 非目标

本轮明确不做：

1. 不重写战斗规则
2. 不让时间线等待动画播放完成
3. 不把动画状态写入存档
4. 不引入必须依赖动画的判定逻辑
5. 不在本轮追求正式美术资产或复杂骨骼系统

## 3. 分层原则

### 3.1 规则层

规则层负责：

1. 生成战斗状态
2. 发出事件
3. 推进回合与时间线

规则层不负责：

1. 控制角色怎么动
2. 管理飘字 DOM
3. 维护动画生命周期

### 3.2 展示层

展示层负责：

1. 监听战斗事件
2. 把状态差异转成视觉反馈
3. 为独立测试页提供可复用驱动

展示层不负责：

1. 改写伤害
2. 改写 Buff 结果
3. 改写敌人决策
4. 决定时间线是否继续

## 4. 推荐结构

本轮采用三段式展示层结构：

### 4.1 `BattlePresentationController`

职责：

1. 订阅事件总线
2. 维护上一帧只读快照
3. 把 `TIMELINE_ENTRY_START / BATTLE_TAKE_DAMAGE / BATTLE_UPDATE` 等事件翻译成展示命令

### 4.2 `FighterPresenter`

职责：

1. 持有某一侧角色与 HUD 的 DOM 引用
2. 负责角色 root、sprite container、shadow、hp bar、status row、armor row 的局部表现
3. 对外暴露“攻击”“受击”“治疗”“状态脉冲”等语义化接口

### 4.3 `BattleAnimationDriver`

职责：

1. 执行短生命周期 DOM 动画
2. 管理临时 class、飘字节点和超时清理
3. 保证动画只影响展示，不向规则层回写

## 5. 事件映射

推荐使用以下事件作为展示层的正式输入：

| 事件 | 来源 | 展示层用途 |
| --- | --- | --- |
| `BATTLE_START` | `CoreEngine` | 建立初始快照，刷新角色基线 |
| `BATTLE_UPDATE` | `CoreEngine` | 比对 HP / 护甲 / Buff / Debuff 差异，触发缓动和提示 |
| `TURN_START` | `CoreEngine` | 回合开始轻提示、清理残留态 |
| `TIMELINE_ENTRY_START` | `TimelineManager` | 触发攻击前冲或施法起手 |
| `TIMELINE_ENTRY_END` | `TimelineManager` | 用于收尾恢复态 |
| `BATTLE_TAKE_DAMAGE` | `CoreEngine` | 触发受击、伤害飘字、护甲破损提示 |

补充原则：

1. 展示层优先依赖结构化事件
2. 不建议依赖解析 `BATTLE_LOG.text` 文案来驱动关键动画
3. 对治疗、Buff 增减等缺少专用事件的场景，可通过 `BATTLE_UPDATE` 前后快照差异推导

## 6. DOM 契约

展示层只依赖稳定的展示容器，不依赖业务按钮。

本轮最小 DOM 契约如下：

1. 场景区：
   - `.battle-scene`
   - `.fighter.player-character`
   - `.fighter.enemy-character`
   - `.character-sprite-container`
   - `.sprite-layer`
   - `.fx-layer`
2. HUD 区：
   - `.player-hud`
   - `.enemy-hud`
   - `.bar.hp span`
   - `.status-row`
   - `.armor-row[data-key]`

如果某个节点缺失：

1. 展示层应降级跳过该效果
2. 不抛出阻断主流程的异常

## 7. 动画降级策略

### 7.1 失败降级

任何一类动画失败时：

1. 仅丢失该视觉效果
2. 不影响按钮可点击性
3. 不影响时间线推进
4. 不影响结算和存档

### 7.2 关闭策略

展示层需要支持显式关闭。

本轮建议：

1. 控制器支持 `enabled` 开关
2. probe 页直接提供开关按钮
3. 主流程即使不暴露 UI 开关，也必须支持代码级关闭

### 7.3 性能降级

如果浏览器性能不足：

1. 优先保留：
   - 伤害飘字
   - 血条缓动
   - 受击闪白
2. 次级效果可减弱或关闭：
   - 长时间 idle 漂浮
   - 多层纸娃娃摆动
   - 复杂屏幕闪烁

### 7.4 重置与新战斗清理

展示层还必须承担“瞬态状态自清理”职责。

原因：

1. 飘字、攻击态、受击态都属于展示层临时状态
2. 它们不应跨回合、跨新战斗或跨 probe 重置残留
3. 如果残留到下一轮，玩家会误判这轮的真实结算结果

因此推荐约束为：

1. `BATTLE_START` 到达时清理上一轮瞬态类与飘字
2. `TURN_START` 到达时清理未自然结束的短生命周期表现
3. 显式关闭动画时，也应立即清理当前残留效果

## 8. 轻量纸娃娃试验边界

用户已接受本轮走“混合路线”：

1. 先建立纯展示层动画驱动
2. 再对一个角色做轻量纸娃娃试验

本轮纸娃娃试验边界如下：

1. 只验证一个角色
2. 只验证：
   - `idle`
   - `attack`
   - `hit`
   - `recover`
3. 允许使用现有 `body / armor / weapon` 分层做轻量位移和旋转
4. 不要求进入正式美术资产工作流
5. 试验应优先放在独立 probe 页，而不是先侵入主流程

## 9. probe 页定位

`test/battle_presentation_probe.html` 的职责不是“替代主流程页”，而是：

1. 单独证明动画展示层可以成立
2. 单独解释为什么某个按钮动作能证明某项能力
3. 单独承载本轮轻量纸娃娃试验

probe 页还应额外遵守：

1. 如果某个按钮声称“单击即可验证某能力”，那么默认基线必须真的能触发该能力
2. 例如 `玩家治疗` 若用于一键验证治疗回弹，则默认状态不能是满血
3. 这类验证前提必须直接写在页面说明里，不能让人工验收者自己猜

本页必须明确说明：

1. 它是展示层验证页
2. 它验证的不是伤害公式，而是视觉反馈链
3. 它与 `mock_ui_v11.html` 的关系：
   - probe 页验证“展示层可以独立成立”
   - 主流程页验证“展示层接入后不阻塞真实游玩”

## 10. `mock_ui_v11.html` 接入原则

接入主流程时必须遵守：

1. 仍以用户既有主流程为准：
   - 开始冒险
   - 关卡选择
   - 进入关卡
   - 提交规划
   - 执行
   - 进入下一回合
2. 动画不能改变上述点击顺序
3. 动画不能要求新增“确认动画结束”的人为步骤
4. 动画不能依赖时间线区域中当前未启用的“开始 / 单步”等按钮

## 11. 本轮验收口径

### 11.1 独立验证

在 `test/battle_presentation_probe.html` 中应能人工观察到：

1. 攻击前冲
2. 受击
3. 飘字
4. 治疗回弹
5. 状态上身提示
6. 轻量纸娃娃动作

### 11.2 主流程回归

在 `mock_ui_v11.html` 中应能确认：

1. 动画接入后主流程仍可从开始冒险走到执行
2. 动画失败不会卡住回合
3. 多轮执行后不会出现持续残留态覆盖界面

## 12. 动画参数配置器定位

`WBS-4.3.7` 不应被理解为“把当前几个动画参数做成可输入文本框”。

合理定位应是：

1. 它是展示层的工具节点，用于管理动作演出参数与展示配置资产。
2. 它输出的是“展示配置产物”，不是运行时临时状态，也不是规则层数值。
3. 它的配置产物要能同时被当前主工程和未来 `SpineAssets` 协作链消费。

### 12.1 工具职责

动画参数配置器至少应负责：

1. 编辑关键展示参数：
   - 攻击前冲距离 / 时长
   - 受击闪白 / 抖动强度
   - 治疗回弹幅度
   - 飘字节奏
   - 镜头轻推 / 震动
2. 提供预览能力，让操作者能在独立入口快速验证参数效果。
3. 支持保存与加载正式配置。
4. 输出稳定 schema，使主流程页与 probe 页都能消费同一份配置。

### 12.2 与主流程的关系

配置器与主流程之间必须保持单向关系：

1. 主流程只读取展示配置产物。
2. 配置器不直接操纵战斗规则、时间线或敌人决策。
3. 即使配置缺失或加载失败，主流程仍按默认展示降级运行。

### 12.3 与 SpineAssets 的关系

如果后续将角色正式切换到 `SpineAssets` 产物，当前工具仍应继续成立。

因此这里的边界应明确为：

1. `SpineAssets` 负责角色资源、动作资源、素材侧约束。
2. 当前主工程中的动画参数配置器负责展示参数、消费映射和配置管理。
3. 双方通过稳定的数据接口衔接，而不是让主工程直接承担素材制作职责。

## 13. 后续扩展方向

本轮之后可继续拆分：

1. `D3.2`
   - 与 `SpineAssets` 的协作规则与导入契约基线
2. `D3.3`
   - 场景节奏强化与镜头感补足
3. `D3.4`
   - 对更多角色和更多技能类别扩展分层动作
4. `D3.5`
   - 在不改规则层的前提下补齐可配置的演出模板
5. `D3.7`
   - 把动画参数配置器做成正式展示工具，并为 `SpineAssets` 协作保留配置资产接口
