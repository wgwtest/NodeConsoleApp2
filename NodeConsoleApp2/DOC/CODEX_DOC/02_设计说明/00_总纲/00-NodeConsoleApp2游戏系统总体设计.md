# NodeConsoleApp2 游戏系统总体设计

最后整理时间：2026-05-21

状态：`当前有效 / 总纲入口`

## 1. 目标与边界

本文是 `NodeConsoleApp2` 设计说明层的总纲入口，用于说明主游戏流程、战斗运行时、关卡地图、技能、Buff、数据存档、UI 交互之间的系统边界。

本文不替代各子系统详细设计。涉及字段、页面、算法、编辑器行为、验收路径时，应进入对应子系统设计文档。

## 2. 子系统划分

| 子系统 | 名称 | 主要责任 | 典型入口 |
| --- | --- | --- | --- |
| `S1` | 主游戏流程 | 主菜单、关卡选择、进入确认、结算流转、流程态信息优先级 | `mock_ui_v11.html` 主流程 |
| `S2` | 战斗运行时 | 核心引擎、敌人、道具、时间线、战斗展示层 | battle runtime / timeline |
| `S3` | 关卡地图与编辑器 | 关卡系统、地图选择、地图包、关卡编辑器、地图编辑器 | map selector / editor pages |
| `S4` | 技能系统与编辑器 | 技能系统、技能树、技能编辑器、技能规划、技能平衡 | skill tree / skill editor |
| `S5` | Buff 系统与编辑器 | Buff 运行时、Buff 数据契约、Buff 编辑器 | buff editor / runtime buff effects |
| `S6` | 数据存档与内容契约 | 数据结构、本地存档、内容装载和跨模块配置契约 | assets data / local save |
| `S7` | UI 与交互基线 | UI 语言、模态规则、1920x1080 验收视口、Spine 与展示协作 | UI pages / prototypes |

## 3. 关键依赖关系

1. `S1 主游戏流程` 是用户入口，依赖 `S3` 提供关卡地图数据，依赖 `S4/S5` 提供成长与构筑信息，依赖 `S6` 保存流程状态。
2. `S2 战斗运行时` 是战斗执行核心，消费 `S3` 的关卡配置、`S4` 的技能配置、`S5` 的 Buff 配置和 `S6` 的统一内容契约。
3. `S3 关卡地图与编辑器` 是内容组织层，既面向主流程的地图式选择，也面向作者工具的编辑、导入、导出和校验。
4. `S4 技能系统与编辑器` 与 `S5 Buff 系统与编辑器` 是成长与效果配置层；二者需要在技能效果、Buff action、参数 schema、运行时说明上保持一致。
5. `S6 数据存档与内容契约` 是跨系统底座，不能被某个 UI 页面或编辑器私有化。
6. `S7 UI 与交互基线` 约束用户可见页面和编辑器页面的交互方式，但不替代各子系统的数据和行为设计。

## 5. 设计演进规则

1. 先更新所属子系统设计，再实施跨模块代码改动。
2. 涉及两个以上子系统的主题，先在本总纲或治理文档中确认边界，再下沉到子系统。
3. 原型可作为设计依据，但正式结论必须回写到设计说明或明确被设计说明引用。
4. 验收发现的问题不直接改写为设计结论；先进入验收意见处理或过程记录，确认后再同步到正式设计。
