# 设计说明文档治理与粒度规则

最后整理时间：2026-05-23

状态：`当前有效 / 文档治理规则`

## 1. 目标

本文定义 `DOC/CODEX_DOC/02_设计说明/` 的目录职责、文档粒度和新增规则，避免设计说明重新退化为按时间堆叠的扁平文档池。

## 2. 分层规则

| 层级 | 目录 | 放什么 | 不放什么 |
| --- | --- | --- | --- |
| 总纲层 | `00_总纲/` | 跨子系统架构、文档治理、全局数据/交互边界 | 单个页面修复、临时验收意见 |
| 子系统层 | `S1-S8/` | 子系统主设计、稳定专题设计、长期有效补充设计 | 一次性讨论、调试记录 |
| 过程层 | `06_过程文档/` | 会话交接、验收意见处理、迁移记录、历史计划 | 稳定设计结论的唯一入口 |
| 测试层 | `05_测试文档/` | 自测报告、验收清单、验收记录、验收结论 | 系统设计正文 |
| 原型层 | `08_原型与附图/` | 设计图、截图基线、HTML 原型、原始材料 | 长期设计条款的唯一来源 |

## 3. 粒度规则

1. 每个子系统优先维护稳定主设计入口。
2. 只有满足以下任一条件时，才新增独立设计文档：
   - 形成长期独立评审对象；
   - 影响多个实现文件或多轮迭代；
   - 需要被测试、验收、原型或计划反复引用；
   - 与主设计合并会明显降低可读性。
3. 一次性修复、临时讨论、过程判断、验收反馈处理，不新增设计说明文档。
4. 设计探索草案可以先进入 `docs/superpowers/specs/` 或 `06_过程文档/`；被采纳后再同步为正式设计说明。

## 4. 命名规则

1. 现有设计文档保留原 `NN-中文主题(英文slug)-设计说明.md` 文件名，`NN` 仅表示历史稳定编号。
2. 新增子系统主设计建议使用 `Sx-中文主题-设计说明.md`。
3. 新增子系统补充设计建议使用 `Sx-中文主题-YYMMDD-HHMM-补充主题.md`。
4. 跨子系统治理文档使用 `00_总纲/NN-中文主题.md`。

## 5. 引用规则

1. 正式文档引用优先使用仓库相对路径。
2. 新文档不得引用已经迁移前的扁平路径，例如 `DOC/CODEX_DOC/02_设计说明/03-技能系统...`。
3. 历史迁移记录可以保留旧路径作为历史事实，但应说明当前路径以 README 索引为准。
4. 用户验收入口、执行计划、当前研发文档需要更新到当前路径；历史交接记录不强制回写。

## 6. 本轮迁移约束

1. 本轮只做结构治理和入口索引，不重写原 26 份设计说明正文。
2. 原文中的旧路径引用只更新设计说明内部和当前正式入口；历史记录保留事实。
3. 本轮不处理技能树 UI 未提交代码和 2026-05-20 原型材料归属。
