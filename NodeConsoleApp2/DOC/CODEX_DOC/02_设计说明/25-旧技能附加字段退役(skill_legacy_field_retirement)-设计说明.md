# 旧技能附加字段退役设计说明 (Legacy Skill Field Retirement)

状态：`当前有效`

最后更新：`2026-04-11`

## 1. 背景与目标

当前工程中的 旧技能附加字段 / 旧技能附加参数字段 / 旧枚举字段，最初是为了做**内容分析、检索、平衡统计**而引入的辅助概念。

它们的设计定位本来就不是：

1. 运行时事实来源
2. 正式 UI 行为开关
3. 契约校验的主依据
4. 作者工具中的长期必填字段

但当前工程里，旧技能附加字段 已被若干功能反向消费，导致它从“分析辅助概念”变成了“运行时与工具链的半正式依赖”。

这会产生三个问题：

1. 同一语义在 `target / actions / buffRefs / costs / requirements / tags` 中重复表达，长期必然漂移。
2. 一旦 旧附加字段 与真实字段不一致，UI、校验、分析页就会一起失真。
3. 后续技能扩展会被迫同时维护“正式字段 + 附加字段镜像”，增加内容生产成本。

本设计说明的目标不是“修几条标签”，而是确定一条**先去依赖、后删概念**的完整迁移路径：

1. 先全面盘点所有依赖 旧技能附加字段 的功能点；
2. 逐项判断该功能是否应保留；
3. 对保留功能给出去旧附加字段化的替代方案；
4. 在替代方案落地后，彻底移除 旧技能附加字段 作为活动工程概念。

## 2. 本轮检索范围

本轮检索覆盖以下活动区域：

1. `script/`
2. `test/`
3. `assets/data/`
4. `DOC/CODEX_DOC/02_设计说明/`
5. `DOC/CODEX_DOC/04_研发文档/`

本轮不把以下内容视为“当前活动契约”：

1. `06_过程文档/` 下的历史会话、历史验收、历史批注处理记录
2. 历史自测报告与历史验收记录中的旧术语

这些历史文件可以保留原始表述，作为时间证据；它们不应反向定义当前系统必须继续保留 旧技能附加字段。

## 3. 检索结论

### 3.1 总结

当前 旧技能附加字段 的活动依赖可分为四类：

1. **正式功能**
   - 技能树过滤
   - 技能面板分类
   - 技能面板搜索 / 图标增强
   - 契约校验与状态板
2. **作者工具 / 分析工具**
   - 技能编辑器标签面板
   - 技能平衡工具
3. **测试与验证**
   - `skill_contract_probe`
   - `codex_regression_runner`
   - `c3_skill_contract_remediation`
4. **活动数据与活动设计文档**
   - `skills_melee_v4_5.json`
   - 活动设计说明中对 旧附加字段/旧附加参数字段 的正式描述

### 3.2 总判断

结论很明确：

1. **没有任何一个必须保留的正式功能，天然需要依赖 旧技能附加字段。**
2. 必须保留的是“功能目的”，不是“tags 实现方式”。
3. 旧技能附加字段 未来最多只能以“分析时临时派生维度”存在，不能再作为活动数据 schema 的一部分。

## 5. 逐项判断

### 5.1 技能树过滤

当前用途：

1. 不让演示技能出现在正式技能树里。

为什么这个功能要保留：

1. 正式技能树应该只呈现玩家真实可学习的技能节点。
2. 演示样本、验收样本、实验技能不应污染玩家构筑视图。

为什么不该依赖 旧附加字段：

1. “是否显示”是明确的 UI 行为开关，不是分析标签。
2. 当前工程里已经有 `editorMeta.hiddenInSkillTree`，语义更直接，也更稳定。

结论：

1. 保留过滤功能。
2. 去掉 `DEMO` 标签依赖。
3. 统一改成显式字段控制。

### 5.2 技能面板分类

当前用途：

1. 在主战斗技能面板中给技能一个粗分类，例如 `HEAL / DAMAGE / DEFENSE / BUFF`。

为什么这个功能要保留：

1. 这关系到玩家阅读效率。
2. 分类本身是 UI 呈现需求，不是分析需求。

为什么不该依赖 旧附加字段：

1. `HEAL / DMG_HP / DMG_ARMOR / BUFF_APPLY` 这些都可以从正式字段直接读出来。
2. 把派生结果存进 旧附加字段 只会制造重复维护。

结论：

1. 保留分类功能。
2. 改成纯派生函数。
3. 不再把派生结果写回配置文件。

### 5.3 契约校验

当前用途：

1. 用于发现技能配置是否自相矛盾。
2. 当前部分问题码直接围绕 旧附加字段 与正式字段的偏差建立。

为什么这个功能要保留：

1. 契约校验本身是必要的基础设施。
2. 没有它，很多“能跑但语义已经歪了”的问题只能靠人工撞出来。

为什么不该依赖 旧附加字段：

1. 旧附加字段 不是正式字段事实来源。
2. 用辅助概念去校验正式字段，会倒置主次。

结论：

1. 保留契约校验。
2. 彻底删除 tag-based issue code。
3. 改成只校验正式字段之间的一致性。

### 5.4 技能编辑器

当前用途：

1. 暴露 旧附加字段、旧附加参数字段 的编辑入口。
2. 将 旧枚举字段 当作固定枚举来源。

为什么这个功能不应保留：

1. 用户已明确要求 旧技能附加字段 彻底退役。
2. 继续保留编辑入口只会继续制造新数据债务。

结论：

1. 直接删除标签编辑能力。
2. 编辑器未来不再认识旧附加字段、旧附加参数字段与旧枚举字段。

### 5.5 技能平衡 / 分析工具

当前用途：

1. 通过 旧附加字段 做统计、过滤、占比预警。
2. 缺 tag 时才临时推断。

为什么这个功能要保留：

1. 内容分析本身仍然有价值。
2. 用户最初引入 tag 的目的就是分析，而不是运行时行为。

为什么不能继续依赖持久化 旧附加字段：

1. 一旦持久化 旧附加字段 和正式字段漂移，分析结果就失真。
2. 分析维度完全可以基于正式字段实时派生。

结论：

1. 保留分析工具。
2. 取消持久化 tags 输入。
3. 改成“实时派生分析维度”。

## 7. 完成判据

完成“彻底移除 旧技能附加字段 概念”的判据如下：

1. `script/` 下的活动代码不再读取 旧技能附加字段 / 旧技能附加参数字段
2. `test/` 下的活动工具页与回归页不再要求 旧枚举字段
3. `assets/data/skills_melee_v4_5.json` 中不再存在：
   - 旧技能附加字段
   - 旧技能附加参数字段
   - 旧枚举字段
4. 技能树隐藏 DEMO 的逻辑已改为显式字段，不再用标签
5. 技能面板分类和图标逻辑已改为正式字段派生
6. 契约校验不再生成任何 tag-based issue code
7. 当前正式设计文档不再把 旧附加字段/旧附加参数字段 作为活动 schema

## 8. 非目标

本设计说明不包含以下目标：

1. 不要求重写全部历史归档文档的旧术语
2. 不要求保留一个“换壳后的新标签数组”
3. 不要求把原有 旧附加字段 原样迁移成另一个同质字段

如果后续仍需要“分析维度”，它只能以**派生分析结果**存在，而不是重新回到 `skills.json` 里成为持久化 schema。

## 9. 当前结论

本轮结论可以压缩成一句话：

1. **保留功能目的，删除 旧技能附加字段 依赖，最后删除 旧技能附加字段 本身。**

换言之：

1. 要保留的是“过滤、分类、校验、分析”这些能力；
2. 要删除的是“这些能力都去依赖 旧技能附加字段 的实现方式”。
