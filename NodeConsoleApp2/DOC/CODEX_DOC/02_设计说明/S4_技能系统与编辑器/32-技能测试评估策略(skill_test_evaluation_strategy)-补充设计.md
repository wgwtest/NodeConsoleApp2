## 1. 补充目的

本补充用于定义技能批量测试后的评估策略。它不替代技能测试器实现，也不替代 Codex 的最终设计判断，而是规定：

1. 技能测试分为哪些类型。
2. 每种测试类型使用什么策略和机制。
3. 各测试类型之间有哪些共性。
4. 总体指标和分层专项指标如何区分。
5. 如何基于测试结果提出技能改进方案，并通过复测验证。

技能测试评估的目标不是证明当前设计“值得做”或“不值得做”，而是基于当前设计持续发现问题、提出改进方案，并用工程化模拟复测这些方案。

## 2. 测试类型总览

当前技能测试评估分为三种主要测试类型。三者是递进关系，不应互相替代。

| 测试类型 | 核心问题 | 使用时机 | 输出重点 |
| --- | --- | --- | --- |
| 纯结果优化与无脑组合排查 | 当前设计是否存在显而易见的低复杂度通解 | 第一轮必须先做；每次大改后也要复测 | 最快组合、最高覆盖组合、无脑风险、重复循环 |
| 标签指导的探索式测试 | 当前技能是否形成可解释的策略方向 | 无脑通解被识别或缓解后 | 流派成立度、角色链条、技能差异、上下文适配 |
| Codex / 人工逐关决策测试 | 面对具体关卡时是否存在有意义的玩家选择 | 前两类问题基本收敛后 | 每关选择理由、可辩护方案、体验问题和精修建议 |

复测不是第四种测试类型，而是三种测试类型都必须具备的闭环动作。每次提出技能改进方案后，都应选择对应测试类型重新运行，验证问题是否缓解。

## 3. 共性策略

三种测试类型虽然目标不同，但必须共享同一套评估共性，否则不同轮次结果无法比较。

### 3.1 共性流程

每轮测试应遵守以下流程：

1. 明确本轮测试类型和目标。
2. 固定技能包、Buff 包、关卡包、敌人包和 KP 口径。
3. 生成候选构筑或人工指定构筑。
4. 运行回合模拟并保存原始记录。
5. 计算总体指标。
6. 按当前测试类型计算专项指标。
7. Codex 基于证据写结论。
8. 提出技能改进方案和复测目标。

工具只能提供证据。最终结论必须由 Codex 或人工基于证据给出。

### 3.2 共性证据

每轮测试至少保留以下证据：

| 证据 | 用途 |
| --- | --- |
| 数据源路径 | 证明本轮使用的技能、Buff、关卡和敌人版本 |
| KP / AP 口径 | 证明构筑约束一致 |
| 候选构筑清单 | 证明测试不是只看单一结果 |
| 选中构筑 | 解释本轮实际模拟对象 |
| 每回合动作日志 | 分析循环是否固定、是否依赖状态 |
| Buff / HP / 护甲变化 | 分析策略链条是否成立 |
| 跳过技能和未用技能 | 暴露技能条件或循环不匹配 |
| 原始 JSON 记录 | 支撑后续复盘和复测 |

### 3.3 共性边界

1. 标签和语义画像可以写入技能主 JSON，但运行时不消费。
2. 编辑态脚本可以检查字段结构、引用完整性和明显矛盾，但不能自动判断技能好坏。
3. 批量分析器可以统计风险信号，但不能直接生成最终设计结论。
4. Codex 结论必须引用原始证据，不能只给等级。
5. 任何“好玩”判断都必须落回可观察证据，例如选择差异、策略链条、代价收益和上下文适配。

## 4. 各测试类型的机制与问题

### 4.1 纯结果优化与无脑组合排查

#### 4.1.1 测试机制

这一类测试允许工具优先寻找最快、最高伤害、最容易通关的组合。

推荐机制：

1. 在当前 KP 预算下枚举可购买构筑。
2. 对每个构筑生成合法回合组合。
3. 按击败回合、伤害效率、AP 使用率、KP 使用率排序。
4. 记录最佳组合和前若干候选组合。
5. 对高频技能组做替换对照，例如同角色技能互换。
6. 统计 30 关范围内的核心技能覆盖、回合模式重复和角色单调度。

这一层可以使用脚本做大规模遍历和统计，因为它解决的是结果空间搜索问题，不是设计语义判断问题。

#### 4.1.2 解决的问题

这一类测试用于发现：

1. 单技能通解。
2. 固定二技能或三技能通解。
3. 低阶技能堆叠打穿多数关卡。
4. 同一回合循环反复出现。
5. 同角色技能替换后结果几乎不变。
6. 高效率组合不依赖敌人状态、Buff 窗口或关卡差异。

“快”和“高伤害”本身不是问题。快、高伤害、低条件、低变化、高覆盖同时出现，才是无脑组合风险。

### 4.2 标签指导的探索式测试

#### 4.2.1 测试机制

这一类测试以语义画像为入口，按策略方向生成构筑和回合循环。

推荐机制：

1. 为当前测试范围内的技能建立 `semanticProfile`。
2. 按 `roleTags` 组织策略模板，例如流血、连击、招架反击。
3. 每个策略模板生成若干角色完整构筑。
4. 对同角色技能进行替换对照，观察策略是否改变。
5. 按关卡阶段、敌人 HP、护甲和 Buff 状态观察策略适配。
6. 记录策略链条是否完整，例如建窗、维窗、读窗、消窗、终结。

这一层不应依赖脚本自动给技能打标签。标签应由 Codex 或人工基于技能设计意图维护，脚本只做一致性检查。

#### 4.2.2 解决的问题

这一类测试用于发现：

1. 流血、连击、招架反击等方向是否真实成立。
2. 每个方向是否有完整角色链条。
3. 同角色技能是否存在可解释差异。
4. 策略是否随 KP 成长出现新选择。
5. 策略是否随敌人和关卡变化调整。
6. 设计是否存在“看似很多技能，实际只有一种打法”的问题。

### 4.3 Codex / 人工逐关决策测试

#### 4.3.1 测试机制

这一类测试由 Codex 或人工读取具体关卡、敌人和技能语义，逐关选择构筑和行动理由。

推荐机制：

1. 每关读取关卡 KP、敌人目标、可购买技能和语义画像。
2. Codex 先给出候选方案，而不是只给一个答案。
3. 每个候选方案必须说明选择理由、预期打法和风险。
4. 运行模拟后记录实际结果。
5. 如果结果与预期不一致，记录原因。
6. 形成逐关策略复盘。

这一层不适合在第一轮就做。如果前两类测试还存在显著无脑通解，逐关决策会被通解污染。

#### 4.3.2 解决的问题

这一类测试用于发现：

1. 玩家是否有多个可辩护的选择。
2. 每关选择是否需要理解敌人和状态。
3. 慢速通关或失败是否能被解释。
4. 技能组合是否能支持真实玩家的策略表达。
5. 前两类测试没有暴露的体验问题。

## 5. 总体指标

总体指标适用于所有测试类型。它们描述基础事实和全局风险，但不直接替代专项指标。

### 5.1 结果与效率指标

| 指标 | 含义 | 用法 |
| --- | --- | --- |
| `clearRate` | 策略在批次中的通关比例 | 判断基本可行性 |
| `defeatTurn` | 击败敌人的回合数 | 判断快慢 |
| `totalDamage` | 总伤害或有效伤害 | 判断输出强度 |
| `overkillDamage` | 溢出伤害 | 判断输出是否过剩 |
| `apUseRate` | AP 使用率 | 判断每回合资源是否被有效使用 |
| `kpUseRate` | KP 使用率 | 判断构筑是否有效利用预算 |
| `unusedPurchasedSkills` | 买了但没有进入循环的技能 | 判断构筑和循环是否匹配 |

### 5.2 全局风险指标

| 指标 | 含义 | 用法 |
| --- | --- | --- |
| `universalCoreCoverage` | 同一核心技能组覆盖多少关 | 判断通解风险 |
| `turnPatternRepeatRate` | 回合动作模式重复程度 | 判断循环单调度 |
| `primaryRoleCount` | 循环中不同主角色数量 | 判断角色多样性 |
| `replacementSensitivity` | 同角色技能替换后结果变化 | 判断技能差异是否有效 |
| `stateDependencyRate` | 技能释放是否依赖 Buff、护甲、HP 或其它状态 | 判断是否需要状态决策 |
| `contextAdaptation` | 策略是否随关卡、敌人或 KP 改变 | 判断上下文适配 |

总体指标只说明“发生了什么”。是否构成设计问题，必须结合测试类型和专项指标判断。

## 6. 分层专项指标

### 6.1 第一层专项指标：无脑组合排查

| 专项指标 | 表现不好 | 表现较好 |
| --- | --- | --- |
| 核心技能组覆盖 | 同一 2-3 个技能覆盖 `>= 24 / 30` 关 | 没有单一核心组覆盖超过半数关卡 |
| 固定循环重复 | 同一回合模式占胜利记录 `>= 70%` | 不同关卡出现明显不同回合模式 |
| 角色单调度 | 主角色数量 `<= 2` 且仍能通关多数关卡 | 通关需要至少 3 类以上有效角色 |
| 替换不敏感 | 同角色技能互换后击败回合和总伤害差异很小 | 替换后结果、节奏或风险明显变化 |
| 状态无关性 | 多数回合不读取 Buff、护甲、HP 或窗口状态 | 行动选择明显受状态影响 |
| 低阶压制 | 低 KP 技能组长期压过中后期技能 | KP 增长会带来新的有效选择 |

第一层表现不好时，应优先修改技能数值、AP/KP 成本、技能条件、Buff 窗口或敌人压力，不应急于进入逐关人工决策。

### 6.2 第二层专项指标：标签指导探索

| 专项指标 | 表现不好 | 表现较好 |
| --- | --- | --- |
| 角色链条完整度 | 只有建窗或基础输出，缺少维窗、读窗、消窗或终结 | 策略能形成清晰链条 |
| 技能差异可解释性 | 无法解释为什么选 A 不选 B | 每个同角色技能都有明确选择条件 |
| 策略方向独立性 | 不同流派最终收敛到同一组技能 | 流血、连击、招架反击等方向有不同打法 |
| 成长变化 | 早期组合一直用到后期 | KP 增长后出现新角色、新收益或新风险 |
| 敌人适配 | 面对不同敌人仍固定打法 | 敌人 HP、护甲或状态会改变选择 |
| 代价收益 | 高收益技能缺少条件、代价或时机 | 强技能需要明确投入或状态前置 |

第二层表现不好时，应优先修改技能关系、语义画像、技能机制差异和流派链条。

### 6.3 第三层专项指标：Codex / 人工逐关决策

| 专项指标 | 表现不好 | 表现较好 |
| --- | --- | --- |
| 选择理由 | 每关理由几乎一样，只有“伤害最高” | 每关理由能关联敌人、KP、状态和目标 |
| 备选方案 | 只有唯一明显正确答案 | 至少存在两个可辩护方案 |
| 状态响应 | 行动计划不随战斗状态变化 | 会根据流血窗口、护甲、HP 等调整 |
| 失败解释 | 慢速或失败无法解释 | 能指出失败来自输出不足、条件不匹配或资源约束 |
| 体验问题 | 人工判断仍觉得“照流程按键” | 人工判断能感到取舍和风险 |

第三层表现不好时，说明问题已经不只是数值或标签，而是技能系统无法支撑有意义的玩家决策。

## 7. 结论输出格式

每轮批量测试后，结论必须区分总体指标、专项指标和 Codex 判断。

建议格式：

```text
一、测试类型
- 本轮属于哪类测试：
- 本轮目标：

二、总体指标
- 通关率：
- 击败回合：
- AP/KP 使用：
- 核心技能覆盖：
- 重复循环：
- 原始记录路径：

三、专项指标
- 本层关键指标：
- 表现不好项：
- 表现较好项：

四、Codex 结论
- 是否存在无脑通解：
- 当前策略方向是否成立：
- 当前最严重问题：
- 证据：

五、改进方案
- 建议修改的技能：
- 修改目的：
- 预期影响：
- 风险：

六、复测计划
- 使用哪类测试复测：
- 观察哪些指标：
- 什么结果表示改进有效：
```

结论不能只有“较差”“一般”“优秀”这种等级，也不能只显示技能 ID。每个结论必须有可追溯证据。

## 8. 第一轮落地建议

为了避免停留在讨论层，第一轮应优先跑通完整闭环：

1. 为当前剑系核心技能补最小语义画像。
2. 跑 30 关纯结果优化，完成无脑组合排查。
3. 如果发现显著无脑组合，先提出技能修改方案并复测。
4. 无脑组合风险缓解后，再跑标签指导探索。
5. 第二层能产生可解释策略方向后，再进入 Codex / 人工逐关决策。

第一轮不追求一次解决所有技能设计问题。它的验收标准是：流程能稳定产生证据、结论和下一轮可测试改进项。

## 9. 暂不直接并入主文档的原因

本补充仍处于策略草案阶段，尚未被用户确认。当前不直接并入主设计说明，原因是：

1. 测试类型划分和专项指标还需要用户确认。
2. 风险阈值需要通过一轮真实批量测试校准。
3. 语义画像尚未正式写入当前技能 JSON。
4. 批量分析器尚未完整消费本评估体系。
5. Codex 结论报告格式尚未通过实际样例验证。

用户确认后，应将本补充分别并入：

1. `06-技能平衡`：作为技能组合好坏、无脑通解和策略质量评价方法。
2. `30-技能测试器`：作为批量测试和分析器的结论输出契约。
3. `03-技能系统`：补充语义画像与测试评估字段的 authoring 边界。
