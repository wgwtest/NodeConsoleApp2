> 本文档描述 `NodeConsoleApp2` 中技能测试器的目标态设计。技能测试器是面向技能设计、构筑循环验证和问题复盘的工具页，不替代技能编辑器，也不替代战斗运行时。

## 1. 文档目的

技能测试器用于回答一个技能方案是否“能被买到、能被循环使用、能打出合理战斗过程、能暴露设计问题”。

它重点处理以下问题：

1. 某一关大约可用多少 KP。
2. 在 KP 约束下能购买哪些技能组合。
3. 技能组合在每回合 AP、每回合同一技能只能释放一次的约束下，能形成什么输出循环。
4. 这个循环对敌人造成什么状态变化，例如护甲下降、HP 下降、流血窗口增加或被消耗、治疗发生、技能因条件不足而跳过。
5. 测试过程暴露了哪些设计问题，例如 AP 长期用不完、技能买了但循环不用、只有固定二技能重复、流血窗口建立后缺少兑现手段。

技能测试器不是为了寻找全局最优解，而是为了寻找和展示一组“可解释的合理解”，并把测试记录沉淀为后续技能设计依据。

## 2. 系统定位

### 2.1 一句话定位

技能测试器是技能系统的验证探针和设计复盘入口：它把关卡、敌人、KP 预算、技能包、Buff 语义和输出循环放到同一张可读表里，帮助判断技能树是否合理。

### 2.2 与其他工具的边界

| 工具 / 系统 | 负责 | 不负责 |
| --- | --- | --- |
| 技能编辑器 | 修改技能字段、技能树位置、连接关系、保存和发布技能包 | 判断技能组合在关卡节奏下是否有趣 |
| Buff 编辑器 | 修改 Buff 定义、参数、触发时机和状态机 | 枚举技能购买路线 |
| 战斗运行时 | 真实战斗状态、敌人行动、时间线、完整结算 | 批量解释多种构筑循环 |
| 技能测试器 | KP 约束、可购买技能、输出循环、测试记录、问题归纳 | 修改正式技能包、替代完整敌人 AI |

### 2.3 本阶段口径

本阶段技能测试器优先做“玩家技能输出循环”测试，不做完整敌我 AI 对战。

原因是当前讨论的核心问题是：技能组在有限 KP 和有限 AP 下是否能形成合理循环。完整敌人 AI 会引入额外变量，容易遮蔽技能本身的问题。后续敌人编辑器和敌人技能稳定后，再把敌人行动压力纳入同一测试页面。

## 3. 输入与数据源

### 3.1 输入项

| 输入项 | 说明 |
| --- | --- |
| KP 关卡 | 从关卡包解析后的正式 story 关卡下拉单选；只用于读取关卡顺序、单关奖励和累计 KP |
| 输出目标 | 从敌人包解析后的敌人下拉单选；独立于 KP 关卡，用于输出循环承伤模拟 |
| KP 模式 | `平衡模型：初始 5 KP + 每关 3 KP`、`玩家初始 KP + 实际关卡奖励累计`、`手动 KP` |
| 初始 KP | 平衡模型使用，默认 5；用于替代早期调试用的 30 KP |
| 每关 KP | 平衡模型使用，默认 3 |
| 手动 KP | 当需要临时验证某个预算段时使用，仅手动模式生效 |
| 玩家 AP | 默认 5；可手动改成 6、7 用于中后期假设 |
| 测试回合上限 | 默认 12 回合，防止无效循环无限运行 |
| 技能范围 | 默认剑系；可切换到全部正式技能 |
| 技能包路径 | 默认读取当前玩家技能 authoring 包；后续可接技能编辑器当前打开文件 |
| Buff 包路径 | 默认读取 `assets/data/buffs_v2_7.json` |
| 关卡包路径 | 默认读取 `assets/data/levels.json` |
| 敌人包路径 | 默认读取 `assets/data/enemies.json`，可后续扩展最近敌人 authoring 包 |
| 玩家数据路径 | 默认读取 `assets/data/player.json`，用于读取当前玩家初始 `skillPoints` |

### 3.2 KP 计算

KP 计算必须允许三套口径并存：

1. **平衡模型预算。** 默认按 `初始 5 KP + 关卡序号 * 3 KP` 计算。它是技能设计阶段的目标预算，不是运行时存档事实。第 5 关默认预算为 `5 + 5 * 3 = 20 KP`。
2. **玩家初始 + 实际奖励累计。** 从 `player.json` 读取当前玩家初始 `skills.skillPoints`，再从 `levels.json` 的正式 story 关卡中按顺序累计 `rewards.kp`。这反映当前工程默认玩家配置叠加关卡奖励后的可消费 KP。
3. **手动预算。** 直接使用人工输入值，用于临时验证某个预算段。

如果平衡模型和“玩家初始 + 实际奖励累计”差异很大，测试器必须在发现区提示。这类差异是平衡问题，不应该被工具静默吞掉。

历史说明：`assets/data/player.json` 曾经使用 `skillPoints: 30` 作为调试初始值，用于打开技能树随意点技能。正式平衡测试不应继承该调试口径，当前默认初始 KP 应调整为 5。

### 3.3 关卡与敌人解耦

技能测试器中，关卡包和敌人包的职责必须拆开：

1. **关卡包负责 KP 上下文。** 页面从关卡包解析正式 story 关卡，生成“KP 关卡”下拉单选。选中的关卡只用于确定关卡序号、单关奖励和“玩家初始 + 实际奖励累计”模式下的 KP 预算。
2. **敌人包负责输出目标。** 页面从敌人包解析敌人清单，生成“输出目标”下拉单选。选中的敌人用于 HP、护甲和 Buff 窗口模拟。
3. **关卡不强制绑定敌人。** 如果关卡数据中带有 `primaryEnemy` 或 waves 敌人池，只能作为后续默认建议或兼容回退；正式测试时以敌人下拉选择为准。

### 3.4 敌人模型

本阶段优先使用页面选中的敌人 ID；如果没有显式选择敌人，则兼容取关卡的 `primaryEnemy.templateId`，如果不存在，则取第一波 `waves[].enemyPoolId` 对应敌人池的第一个成员。

敌人模型读取：

```text
levels.json
  -> level.primaryEnemy.templateId
    -> enemies.json / enemy authoring pack
      -> stats.hp / stats.maxHp / bodyParts
```

当前测试器只使用敌人的 HP、护甲部位和名称。敌人技能、AI、行动压力作为后续扩展。

## 4. 核心算法

### 4.1 技能过滤

技能测试器以 skill pack 的 `skills[]` 为事实源。

默认过滤剑系技能：

```text
editorMeta.growthTrack 以 sword 开头
  或 id/name 中明确属于剑系
```

后续可以加入“按技能树分组”“按 Buff 引用”“按设计标签”过滤，但不能要求技能编辑器额外维护一套测试专用字段。

### 4.2 可购买构筑枚举

构筑枚举必须遵守：

1. 技能总学习成本 `sum(unlock.cost.kp) <= KP 预算`。
2. 若购买某技能，则它的 `prerequisites` 必须全部已购买。
3. 缺失前置引用的技能不能被枚举为合法构筑。
4. 枚举结果按“KP 使用率、关键机制覆盖、可用技能数量、技能层级”排序。

枚举器不寻找全局最优树，只提供一批候选构筑。页面默认展示排名靠前的若干个构筑，并允许后续人工指定构筑。

### 4.3 每回合技能组合生成

一回合内的技能组合必须遵守：

1. `sum(costs.ap) <= 玩家 AP`。
2. 同一个技能在同一回合只能释放一次。
3. 不引入冷却时间。
4. 不允许 `AAA` 或 `AAB` 这种重复同一技能的组合。
5. 允许跨回合重复同一组合，例如每回合都出 `A+B`。

组合生成器只生成候选循环。某个技能是否因流血窗口不足而跳过，由模拟阶段判断，并记录到本回合日志里。

### 4.4 抽象战斗模拟

本阶段模拟不是完整 `CoreEngine` 对战，而是一个与 Skill/Buff 语义对齐的轻量模型。

模拟支持：

| 机制 | 当前支持 |
| --- | --- |
| `DMG_ARMOR` | 对指定部位造成护甲伤害，护甲不足时溢出为 HP 伤害 |
| `DMG_HP` | 明确 HP 伤害；若目标是自身则作为代价扣除自身 HP |
| `HEAL` | 回复自身 HP |
| `ARMOR_ADD` | 回复指定部位护甲 |
| `amountType=ABS` | 直接数值 |
| `amountType=BUFF_REMAINING` | 读取指定 Buff 剩余回合，支持 `multiplier/maxRead/missingAs` |
| `repeat.count` | 多段效果按次数重复 |
| `buffRefs.apply` | 支持对自身或敌人施加 Buff |
| `buffRefs.remove.consumeRemaining` | 支持消耗指定 Buff 剩余回合 |
| `requirements.targetBuff.minRemaining` | 支持释放前检查 Buff 剩余回合 |

流血在本阶段采用已定规则：

```text
流血只有持续回合数，不使用层数作为伤害。
重复施加流血 = 增加持续回合数。
回合结束时，若目标有流血：
  先造成 5 点 HP 伤害；
  再让流血剩余回合 -1；
  剩余回合归零后移除。
```

`撕裂伤口` 这类 “目标行动前追加流血窗口” 的 Buff，在轻量模型中以“敌方行动前钩子”模拟：玩家行动结束后、回合结束结算前触发一次。

### 4.5 评分不是结论

评分只用于排序候选，不作为设计结论。页面必须同时展示原始过程表，避免工具用一个分数掩盖问题。

推荐评分维度：

| 维度 | 用途 |
| --- | --- |
| 击败回合 | 判断是否过慢或过快 |
| 平均 AP 使用率 | 暴露“每回合剩 2 AP”一类问题 |
| KP 使用率 | 暴露有预算但买不到合理技能 |
| 技能多样性 | 暴露单调循环 |
| 状态节奏 | 判断是否存在建窗、读窗、消窗 |
| 跳过技能次数 | 暴露技能需求与循环不匹配 |
| 终局剩余 HP / 护甲 | 判断输出是否不足 |

## 5. 页面设计

### 5.1 页面结构

技能测试器采用工作台布局：

```text
顶部控制条
  - 技能包 / Buff 包 / 关卡包 / 敌人包 / 玩家包
  - KP 关卡下拉 / 关卡奖励摘要 / 输出目标下拉 / 敌人 HP 与护甲摘要
  - KP 模式 / 手动 KP / AP / 回合上限 / 技能范围 / 运行测试

左侧上下文栏
  - 当前 KP 关卡
  - KP 预算
  - 当前输出敌人 HP 与护甲
  - 数据源路径

中间候选构筑区
  - 推荐构筑列表
  - 已购技能
  - KP 使用量
  - 循环组合
  - 评分与警告

右侧回合回放区
  - 每回合使用的技能组合
  - AP 使用
  - 敌人 HP / 护甲 / Buff 变化
  - 跳过原因
  - 关键发现
```

### 5.2 视觉口径

页面应采用浅色、密集、表格化的测试工具风格。它是内部分析工具，不做英雄区，不做营销式大卡片，不使用大面积暗底。

重点是：

1. 让 KP、AP、敌人血量、护甲、Buff 窗口这些数字一眼可读。
2. 让每回合变化像日志表一样可追踪。
3. 让发现的问题以醒目但克制的方式显示。
4. 不用图例堆解释，直接在表格和徽标上表达状态。

## 6. 测试记录

技能测试器生成的记录属于 Skill 测试记录，不放入原型与附图目录。

记录目录：

```text
DOC/CODEX_DOC/05_测试文档/05_Skill技能系统测试记录/
```

记录命名必须包含日期和时间：

```text
YYYY-MM-DD-HHmmss-技能测试器记录/report.json
```

本阶段页面优先保存 JSON 记录。JSON 记录至少包含：

1. 测试时间。
2. 数据源路径。
3. 关卡序号、关卡 ID、敌人 ID。
4. KP 模式、KP 预算、AP 预算。
5. 候选构筑摘要。
6. 选中构筑和输出循环。
7. 每回合状态变化。
8. 发现的问题与设计含义。

HTML 展示由技能测试器页面负责重新渲染，不再为每轮测试强制生成一次性 HTML 文件。

## 7. 实现约束

1. 核心计算必须放在可被 Node 测试直接导入的模型模块中。
2. 页面层只负责读取文件、调用模型、渲染结果和保存记录。
3. 模型层不能依赖 DOM。
4. 测试器不能修改正式技能包。
5. 测试器不能把某个具体技能 ID 写死为唯一流程；可以把当前剑系流血技能作为默认样例，但算法必须适用于同 schema 的其他技能。
6. 任何自动评分都必须保留过程表和发现项，不能只给“推荐/不推荐”的结论。

## 8. 验收口径

本阶段完成后应满足：

1. 可以从 `http://127.0.0.1:3101/test/skill_tester.html` 打开技能测试器。
2. 输入第 5 关、默认平衡模型时，KP 预算显示为 20。
3. 测试器能读取关卡、敌人、技能包和 Buff 包。
4. 关卡包解析后能生成 KP 关卡下拉，敌人包解析后能生成输出目标下拉，二者可以独立选择。
5. 测试器能在 KP 约束下枚举可购买技能构筑。
6. 测试器能生成符合 AP 和同回合不重复规则的技能组合。
7. 测试器能展示逐回合状态变化，而不是只给总伤害。
8. `DMG_ARMOR` 对护甲不足的部位会溢出为 HP 伤害。
9. 可以保存包含时间戳的 JSON 测试记录。
10. 模型层有 Node 单元测试覆盖核心算法。
