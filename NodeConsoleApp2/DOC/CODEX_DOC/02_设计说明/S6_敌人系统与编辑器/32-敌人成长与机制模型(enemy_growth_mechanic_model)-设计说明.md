## 1. 文档目标

本文补齐敌人系统的抽象层，使敌人设计从“种族 + 职业 + 若干属性 + 若干技能”的静态配置，升级为可计算、可诊断、可编辑器提示的机制模型。

敌人不是随意堆叠 `hp`、`armor`、`speed`、`ap` 和 `skills[]` 的对象。每个正式敌人都应能回答：

1. 它的目标击败轮数是多少。
2. 它的有效耐久由哪些部分组成。
3. 它给玩家制造哪一种主要压力。
4. 玩家用哪类技能、Buff 或构筑反制它。
5. 它的技能复杂度是否匹配普通敌人、精英或 Boss 定位。
6. 它的行动选择是怎样的确定性循环，而不是随机乱放技能。

本文不替代 `30-三章关卡敌人与技能平衡` 和 `31-流派敌人职责与反制矩阵`。那两份文档描述当前三章内容矩阵；本文描述矩阵背后的通用建模方法。

## 2. 核心抽象

推荐把敌人模板理解为以下对象：

```text
EnemyTemplate =
  GrowthBudget
  + ResponsibilityProfile
  + BodyProfile
  + SkillProfile
  + DecisionPolicy
  + CounterContract
  + PresentationProfile
```

各部分职责如下：

| 模块 | 解释 | 主要字段或派生指标 |
| --- | --- | --- |
| `GrowthBudget` | 敌人在当前章节、关卡和职责下允许占用多少战斗预算 | 目标轮数、有效耐久、输出压力、控制压力、状态压力 |
| `ResponsibilityProfile` | 敌人在关卡矩阵中负责检查玩家什么能力 | 快攻、护甲、修复、状态、控制、精英组合、Boss |
| `BodyProfile` | 敌人的部位护甲、弱点和受击策略 | HP、部位护甲、弱点倍率、重点部位 |
| `SkillProfile` | 敌人技能组制造压力、续航或机制窗口的方式 | 技能 AP、Buff 引用、技能复杂度、压力标签 |
| `DecisionPolicy` | 敌人在运行时如何观察、选择、承诺和释放技能 | 条件、优先级、阶段、冷却、最近使用记录 |
| `CounterContract` | 玩家应如何反制该敌人，以及失败应如何诊断 | 反制流派、预期失败信号、最小可用技能族 |
| `PresentationProfile` | 编辑器、地图和战斗 UI 展示敌人的素材与摘要 | 原画、地图头像、战斗立绘、编辑器摘要 |

其中前六项是战斗设计真值，`PresentationProfile` 是展示和作者工具需要的配套信息。原画可以强化敌人识别，但不能替代机制设计。

## 3. 有效耐久模型

敌人血量不能单独设计。玩家实际打败敌人所面对的是“有效耐久”。

```text
目标有效耐久 = 玩家阶段有效输出 * 目标击败轮数

有效耐久 =
  HP 预算
  + 护甲预算
  + 回复预算
  + 减伤预算
  + 延迟预算
```

玩家阶段有效输出可粗略写为：

```text
玩家阶段有效输出 = 玩家理论单轮输出 * 输出折扣
```

当前总纲基准为：

```text
基础 AP = 5
1 AP = 10 点指定部位即时护甲伤害
玩家理论单轮输出 = 50
```

输出折扣来自：

1. 敌人迫使玩家防御、补甲、治疗。
2. 敌人用护甲和部位结构限制玩家最优攻击。
3. 敌人用流血、中毒、减速、晕眩、易伤等 Buff 制造处理成本。
4. 敌人用回复、补甲、减伤和反击拉长战斗。
5. 敌人速度和行动顺序压缩玩家反应窗口。

因此，一个 `300 HP` 敌人不一定比一个 `220 HP + 高护甲 + 补甲技能` 的敌人更耐打。编辑器和测试报告应展示派生有效耐久，而不是只展示 HP。

### 3.1 耐久预算分解

| 预算 | 来源 | 计价方式 |
| --- | --- | --- |
| HP 预算 | `stats.hp / stats.maxHp` | 直接计入基础耐久 |
| 护甲预算 | `bodyParts.*.current/max` | 按玩家可打部位和弱点折算，不把全部护甲简单相加 |
| 回复预算 | 治疗、吸血、补甲技能 | 按预期释放次数和实现率折算 |
| 减伤预算 | 护盾、免伤、减伤 Buff | 按覆盖回合与受击概率折算 |
| 延迟预算 | 晕眩、减速、AP 压制、目标限制 | 折算为玩家损失的有效输出回合 |

第一阶段不要求所有折算都进入运行时，只要求编辑器和设计评审按同一口径讨论。

### 3.2 HP 反推口径

推荐先确定目标轮数，再反推血量：

```text
目标 HP ~= 玩家理论单轮输出 * 目标轮数 * 输出折扣 - 其他有效耐久
```

示例：

| 关卡阶段 | 目标轮数 | 输出折扣 | 其他有效耐久 | 推荐解释 |
| --- | ---: | ---: | ---: | --- |
| 第一章普通敌 | 3-5 | 0.45-0.65 | 低 | 血量不高，靠速度或少量状态建立识别 |
| 第二章标准敌 | 6-9 | 0.50-0.70 | 中 | 护甲、减速、修复让玩家不能全力输出 |
| 第三章精英 | 8-12 | 0.45-0.65 | 中高 | 状态、修复、反击或吸血形成组合检查 |
| Boss | 10+ | 按阶段变化 | 高 | 不能只靠 HP，必须靠阶段、窗口和反制契约 |

## 4. 章节成长曲线

敌人成长不是线性加 HP。每章应主要增加“机制维度”和“组合压力”，而不是只增加数值。

| 阶段 | 敌人成长重点 | 玩家被检查的能力 | 禁止倾向 |
| --- | --- | --- | --- |
| 前期 | 低血量、低护甲、清晰弱点、单压力点 | 基础输出、基础补甲、低 AP 行动 | 用高 HP 拖长教学战 |
| 中期 | 护甲、减速、修复、部位选择 | 破甲、速度管理、输出窗口 | 所有敌人都变厚 |
| 后期 | DOT、吸血、反击、组合状态、阶段变化 | 构筑循环、Buff 处理、爆发窗口 | 1000 HP 但无机制 |

第一章应让玩家理解“敌人会提出问题”。第二章应让玩家理解“不同敌人需要不同工具”。第三章应让玩家理解“构筑循环和 Buff 管理决定胜负”。

## 5. 敌人职责与压力预算

敌人职责是关卡矩阵中的功能位。职责决定这个敌人应该把预算花在哪里。

| 职责 | 主要压力 | 预算重点 | 玩家反制 | 设计风险 |
| --- | --- | --- | --- | --- |
| 快攻 | 先手、小伤、多次行动、轻状态 | 速度、低 AP 技能、少量流血或减速 | 低 AP 防御、补甲、先手击杀 | 血量也高会变成无解压迫 |
| 护甲 | 部位护甲、弱点选择 | 护甲分布、弱点倍率 | 破甲、部位选择、DOT | 护甲和修复叠太多会拖沓 |
| 修复 | 治疗、补甲、延长战斗 | 回复预算、技能释放窗口 | 爆发、打断、DOT、破甲推进 | 与高 HP 叠加会变成纯磨血 |
| 状态 | 流血、中毒、易伤、标记 | Buff 持续时间、实现率、处理成本 | 治疗、净化、速杀、状态缓解 | 玩家无反制时只能数值碾压 |
| 控制 | 减速、晕眩、AP 压迫 | 节奏预算、触发频率 | 速度管理、低 AP 方案、预防性防御 | 分布过密会剥夺玩家回合 |
| 精英组合 | 两个压力点组合 | 两个预算中等叠加 | 对应两类反制的组合构筑 | 不应只是普通敌 HP 翻倍 |
| Boss | 章节主题考试 | 阶段、窗口、压力循环 | 章节核心反制和高阶构筑 | 没有提示的机制会显得随机 |

一个普通敌人原则上只承担一个主职责和一个弱副职责。精英可以承担两个压力点。Boss 可以覆盖章节主题，但必须通过阶段和提示分批呈现。

## 6. 敌人技能复杂度分级

敌人技能应复用 `S4 SkillDef` 和 `S5 BuffDef`，不另造敌人专用执行协议。差异只体现在技能包、敌人模板引用和 AI 选择策略。

推荐按 Buff 与技能复杂度分级：

| 层级 | 允许对象 | 能力范围 | 适用敌人 |
| --- | --- | --- | --- |
| L1 | 直接动作或简单 Buff | 伤害、补甲、治疗、施加/移除单个 Buff | 普通敌人 |
| L2 | 参数化 Buff 和条件技能 | 按固定参数施加状态，按 HP/护甲/部位选择技能 | 普通敌人后段、精英 |
| L3 | 读取 Buff 或战场状态 | 读取 remaining/stacks、目标已有状态、护甲破裂状态 | 精英、Boss |
| L4 | 消耗窗口或阶段机制 | 消耗 Buff 窗口、阶段切换、蓄力兑现、复合循环 | Boss 或少数章节精英 |

### 6.1 普通敌人技能约束

普通敌人建议：

1. 2 个技能为主。
2. 主要使用 L1/L2。
3. 技能 AP 成本必须能被 `stats.ap` 稳定支付。
4. 不使用多层随机概率作为强度平衡手段。
5. 每个技能必须映射到一个压力类型和一个反制方向。

### 6.2 精英技能约束

精英敌人建议：

1. 3 个技能为主。
2. 可以组合两个压力点，例如护甲 + 修复、状态 + 速度。
3. 可以使用少量 L3，但必须让玩家在战斗日志、UI 或前置关卡中看懂。
4. 不用 HP 翻倍替代机制组合。

### 6.3 Boss 技能约束

Boss 建议：

1. 4 个技能为主。
2. 技能组围绕章节主题组织，而不是把所有强技能堆在一起。
3. 可以使用 L3/L4，但必须绑定阶段、蓄力、提示或脆弱窗口。
4. 每个阶段都应有清晰反制，而不是靠概率给玩家机会。

### 6.4 概率设计限制

后续新增敌技应尽量采用确定性设计。当前敌人技能包中存在 `chance < 1` 的历史技能，它们可以作为兼容事实保留，但不应作为新设计的主要平衡方式。

推荐替代方式：

| 旧方式 | 推荐方式 |
| --- | --- |
| 30% 概率晕眩 | 每 3 回合一次确定性晕眩，或先蓄力再晕眩 |
| 50% 概率中毒 | 低伤害确定性中毒，或目标已破甲时中毒 |
| 20% 概率重击 | 明示蓄力，下回合确定性重击 |
| 随机治疗 | 低血阈值触发治疗，带冷却或 AP 限制 |

确定性不等于简单。它让玩家能读懂、规划和反制，也让自动测试能稳定复现问题。

## 7. 决策循环模型

敌人 AI 不应只是随机从 `skills[]` 中取一个可支付技能。推荐把敌人回合拆成以下循环：

```text
Observe -> Intention -> Commit -> Telegraph -> Resolve -> Recovery
```

| 阶段 | 含义 | 数据来源 |
| --- | --- | --- |
| Observe | 读取自身、玩家、Buff、护甲、AP、阶段和回合数 | CoreEngine、BuffManager、EnemyTemplate |
| Intention | 选择本回合意图，例如进攻、防御、修复、施压、收割 | `aiProfile`、职责、阶段 |
| Commit | 在 AP 预算内选定技能序列 | `skills[]`、技能 AP、最近使用记录 |
| Telegraph | 对高压技能给出可见提示或日志 | UI、战斗日志、阶段提示 |
| Resolve | 执行技能 actions 和 buffRefs | S2/S4/S5 |
| Recovery | 记录冷却、阶段、最近使用和诊断信息 | runtime state、测试日志 |

第一阶段运行时可以简化成：

```text
读取战斗状态
  -> 给可用技能打分
    -> 选择 AP 可支付的最高分技能
      -> 执行
        -> 记录选择原因
```

技能评分最小输入应包含：

1. 自身 HP 比例。
2. 自身部位护甲是否破裂。
3. 玩家 HP 比例。
4. 玩家关键部位护甲是否破裂。
5. 双方 active Buff。
6. 当前 AP。
7. 当前回合数。
8. Boss 阶段或精英状态。
9. 最近使用过的技能。

### 7.1 行动意图

推荐最小意图集合：

| 意图 | 触发条件 | 常用技能 |
| --- | --- | --- |
| `pressure` | 默认进攻或玩家防御不足 | 伤害、破甲、流血、中毒 |
| `finish` | 玩家低 HP 或关键护甲破裂 | 重击、收割、连击 |
| `defend` | 自身关键护甲破裂或预期受击高 | 补甲、减伤、格挡 |
| `recover` | 自身低 HP 或状态不利 | 治疗、吸血、清状态 |
| `setup` | Boss 阶段准备或精英连携 | 蓄力、标记、易伤 |
| `exploit` | 玩家已有目标状态 | 读取 Buff、消耗窗口、追击 |

## 8. Boss 模型

Boss 应是章节机制考试，而不是高 HP 普通敌人。

推荐对象：

```text
Boss =
  ChapterTheme
  + PhaseRules
  + PressureCycle
  + VulnerabilityWindow
  + CounterContract
```

| 模块 | 解释 |
| --- | --- |
| `ChapterTheme` | 本章主要压力，例如快攻、护甲控制、持续压迫 |
| `PhaseRules` | 何时转阶段，按 HP、回合数、护甲破裂或 Buff 状态触发 |
| `PressureCycle` | Boss 在若干回合内循环制造什么压力 |
| `VulnerabilityWindow` | 玩家什么时候能获得高收益输出窗口 |
| `CounterContract` | 玩家需要哪些技能族或构筑能力才能合理通关 |

一个基础 Boss 阶段结构可以是：

```text
阶段 1：展示主题压力
阶段 2：加入防御、修复或控制，拉开构筑差异
阶段 3：暴露脆弱窗口，奖励正确反制
阶段 4：终局压迫，要求玩家完成收割
```

Boss 不一定都需要四阶段。第一章 Boss 可以两阶段；第二章 Boss 可以三阶段；第三章 Boss 可以完整引入压力循环和窗口兑现。

## 9. 反制契约

每个正式敌人和核心敌技都应维护反制契约。

```text
CounterContract =
  pressure
  + expectedCounter
  + failureSignal
  + minimumPlayerTool
  + tuningAction
```

| 字段 | 解释 |
| --- | --- |
| `pressure` | 敌人制造的压力，例如高速度、护甲、DOT、控制、修复 |
| `expectedCounter` | 玩家合理反制方式，例如补甲、破甲、净化、速杀、爆发窗口 |
| `failureSignal` | 如果玩家失败，日志或测试应看到什么信号 |
| `minimumPlayerTool` | 到该关卡时玩家至少应可获得的技能族或 Buff 处理能力 |
| `tuningAction` | 失败后优先调整敌人、敌技、关卡敌人池、玩家技能树还是数值 |

示例：

| 敌人职责 | 压力 | 预期反制 | 失败信号 | 优先动作 |
| --- | --- | --- | --- | --- |
| 快攻 | 先手连续破甲 | 低 AP 补甲或先手击杀 | 玩家前 2 回合关键部位护甲归零 | 降速度或补早期防御技能 |
| 护甲 | 长时间无法破胸甲 | 破甲或攻击弱点 | 敌人 HP 长期不下降，回合数过长 | 调护甲分布或补破甲入口 |
| 状态 | 毒/流血持续耗血 | 治疗、净化、速杀 | 玩家死于 DOT，敌人未形成输出窗口 | 检查净化/治疗是否可获得 |
| 修复 | 敌人反复补甲 | 爆发窗口、DOT、打断 | 敌人护甲回复量高于玩家有效输出 | 限制释放频率或增加窗口 |
| 控制 | 玩家长期无法行动 | 速度管理、预防性防御 | 玩家 AP 或行动频率被压到不可玩 | 降控制密度，不先加 HP |
| Boss | 阶段压力叠加 | 章节核心反制 | 推荐构筑也无法进入脆弱窗口 | 调阶段提示、窗口和技能池 |

反制契约的关键是让失败可归因。失败不是一概说明“敌人太强”；也可能说明玩家技能树缺反制、敌人技能提示不足、关卡顺序过早、或测试构筑不合理。

## 10. 与技能、Buff 系统的边界

敌人系统不应复制技能和 Buff 的执行逻辑。

1. 敌人模板保存 `skills[]` 引用，不保存技能动作副本。
2. 敌人技能仍使用 `S4 SkillDef`，通过 `target / costs / actions / buffRefs` 结算。
3. 敌人状态仍使用 `S5 BuffDef`，通过 Buff 生命周期、trigger 和 action 生效。
4. 敌人 AI 可以读取 Buff 的通用状态，例如 remaining、stacks、tags，但不写具体 `buffId` 特判作为核心逻辑。
5. 敌人编辑器可以展示技能和 Buff 摘要，但不在敌人模板中内嵌技能或 Buff 定义。

推荐把敌人技能看成“AI 可调用的战术动作”，把 Buff 看成“跨回合机制原子”，把敌人模板看成“选择这些动作并把它们放进关卡矩阵的内容对象”。

## 11. 敌人编辑器派生指标

敌人编辑器后续应从敌人模板、技能包、Buff 包和关卡引用中派生以下诊断指标：

| 指标 | 用途 |
| --- | --- |
| 目标阶段 | 显示敌人属于第一章、第二章、第三章或通用样本 |
| 职责标签 | 快攻、护甲、修复、状态、控制、精英、Boss |
| 目标轮数 | 作者填写或由关卡矩阵派生 |
| HP 预算 | `stats.hp/maxHp` |
| 护甲预算 | 按部位、弱点和可受击性派生 |
| 回复预算 | 由治疗、吸血、补甲技能派生 |
| 状态压力 | 由 `buffRefs.apply` 和 Buff 生命周期派生 |
| 控制压力 | 由减速、晕眩、AP 压制技能派生 |
| 技能复杂度 | L1-L4 |
| 反制契约完整度 | 是否填写压力、反制、失败信号和最小工具 |
| 引用安全 | 关卡是否引用、技能是否存在、Buff 是否存在 |

### 11.1 编辑器警告

第一阶段建议加入以下警告，不一定阻塞保存：

1. 普通敌人同时拥有 3 个以上压力标签。
2. 普通敌人引用 L3/L4 技能。
3. 精英或 Boss 没有反制契约。
4. 技能 AP 大于敌人 `stats.ap`。
5. 敌人既高 HP 又高护甲又有高频回复。
6. 新增技能仍使用 `chance < 1` 作为主要强度来源。
7. 状态敌出现于玩家尚无治疗、净化或速杀工具的关卡。
8. Boss 没有阶段、窗口或压力循环说明。

这些警告的目标不是限制作者创作，而是强迫作者解释设计理由。

## 12. 验证口径

敌人成长模型需要三层验证。

### 12.1 静态契约验证

静态验证检查：

1. 敌人 ID、名称、职责、章节标签存在。
2. `skills[]` 引用全部可解析。
3. 敌人技能引用的 Buff 全部可解析。
4. 技能 AP 不超过敌人可支付 AP。
5. Boss 和精英具备反制契约。
6. 普通敌人复杂度不超过职责允许范围。

### 12.2 数值报告

数值报告应按章节和职责输出：

1. HP 分布。
2. 护甲分布。
3. 速度分布。
4. AP 分布。
5. 回复预算。
6. 状态和控制压力数量。
7. 目标轮数和实际模拟轮数。

报告目标是发现“成长曲线形状”是否合理，而不是只找某一个敌人的单项数值。

### 12.3 模拟与浏览器诊断

模拟和浏览器 smoke 应记录：

1. 玩家失败时死于直接伤害、DOT、控制、回复拖延，还是输出不足。
2. 敌人每个技能的使用次数。
3. 玩家有多少 AP 用于输出、防御、治疗和状态处理。
4. 目标轮数与实际轮数差距。
5. 推荐构筑和偏科构筑的结果差异。

如果推荐构筑和偏科构筑表现接近，说明敌人没有真正检查构筑。如果偏科构筑完全不能玩，说明反制空间可能过窄。
