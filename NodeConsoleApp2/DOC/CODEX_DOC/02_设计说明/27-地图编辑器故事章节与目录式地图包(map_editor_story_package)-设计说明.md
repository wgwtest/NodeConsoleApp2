# 地图编辑器故事章节与目录式地图包设计说明

创建时间：2026-05-21

状态：`草案`

关联节点：

1. `WBS-3.2.3 地图编辑器基础能力与地图包导出链`
2. `26-关卡地图选择与地图包(level_map_selection)-设计说明`

## 1. 问题定义

当前地图编辑器只是在一个既有 `level_map_pack` JSON 上编辑地图、节点和边。这个实现能验证地图渲染、节点拖拽、边编辑和资源引用，但还不是一个完整的地图编辑器产品。

缺失的不是单个按钮，而是顶层业务对象和对象边界：

1. 没有“一个地图包就是一个故事”的概念。
2. 没有“当前故事包下新建章节”的概念。
3. 没有“章节下管理多张地图”的概念。
4. 没有地图级的新建、复制、删除、排序和归属章节操作。
5. 没有完整地图包目录作为主游戏加载单位。

因此现阶段编辑器更准确地说是“地图 JSON 工作区”，不是“故事地图包编辑器”。

## 2. 目标工作流

目标工作流应调整为：

1. 新建或打开一个地图包；一个地图包就是一个故事。
2. 在地图包中管理故事基本信息。
3. 在当前故事包下新建章节。
4. 在章节下新建、复制、删除和排序地图。
5. 在单张地图中编辑背景、节点、边和资源引用。
6. 保存编辑工作稿。
7. 发布成主游戏可加载的目录式地图包。
8. 主游戏只指向一个地图包根目录，从该目录读取当前唯一地图包。

当前版本只支持“一个目录下只有一个地图包文件夹”。未来如果支持多个故事、多个剧情线、DLC 或玩家自定义包，也应优先按多个地图包切换，而不是在一个地图包内部切换多个故事。

## 3. 业务对象模型

### 3.1 地图包 / 故事

地图包是主游戏加载和编辑器保存的顶层单位，也是玩家理解剧情地图的故事单位。业务上必须按“一包一故事”处理。

建议字段：

1. `packageId`
2. `packageVersion`
3. `title`
4. `status`
5. `createdAt`
6. `updatedAt`
7. `storyId`
8. `summary`
9. `entryChapterId`
10. `chapters[]`
11. `assetLibrary`

底层 `maps.json` 可以继续保留 `stories[]` 兼容字段，但当前阶段必须满足 `stories.length === 1`。编辑器 UI 不提供包内新建故事、删除故事或切换故事；创建另一条故事线等价于创建另一个地图包。

### 3.2 章节

章节是当前故事包下的地图分组单位。

建议字段：

1. `chapterId`
2. `title`
3. `order`
4. `description`
5. `entryMapId`
6. `maps[]`

编辑器必须提供：

1. 新建章节
2. 删除章节
3. 调整章节顺序
4. 设置章节入口地图

### 3.3 地图

地图是章节下具体可编辑的空间图。

建议字段沿用当前 `level_map_pack_v1.maps[]` 的核心字段：

1. `mapId`
2. `chapterId`
3. `name`
4. `backgroundRef`
5. `space`
6. `display`
7. `entryNodeId`
8. `nodes[]`
9. `edges[]`

编辑器必须补齐：

1. 新建地图
2. 复制地图
3. 删除地图
4. 地图排序
5. 修改地图所属章节

## 4. 编辑器信息架构

编辑器不应一上来就是“地图列表”。正确层级应是：

1. 顶部：地图包版本操作
   - 新建地图包，也就是新建故事
   - 打开地图包
   - 保存工作稿
   - 发布到主游戏
   - 导出完整包
2. 左侧：故事与章节结构
   - 当前故事包
   - 章节列表
   - 章节下地图列表
   - 新建 / 复制 / 删除地图
3. 中央：当前地图画布
   - 背景
   - 节点
   - 边
   - 缩放与拖拽
4. 右侧：当前地图元素管理
   - 节点列表
   - 边列表
   - 选中节点详情
   - 选中边详情

节点操作和边操作必须放在右侧对应区域，不能混在地图包版本操作里。

## 5. 目录式地图包

主游戏不应依赖编辑器页面状态，也不应要求用户手工理解多个零散文件。推荐的发布产物是一个目录：

```text
map_packs/
  current/
    story_pack_v1/
      package.json
      maps.json
      assets/
        backgrounds/
        nodes/
        portraits/
```

其中：

1. `package.json`
   - 地图包入口文件。
   - 声明包 ID、版本、标题、入口故事、入口章节、资源基路径和数据文件路径。
2. `maps.json`
   - 保存故事、章节、地图、节点、边。
   - 可以由当前 `level_map_pack_v1` 演进而来。
3. `assets/`
   - 保存该地图包实际依赖的背景图、节点图、立绘图。
   - 资源路径应相对 `package.json` 或包根目录。

当前“只导出 JSON”的能力只能叫 `导出 manifest JSON`，不能叫完整地图包。

## 6. 主游戏加载边界

主游戏的配置应指向一个地图包根目录，而不是指向编辑器工作稿。

建议：

```json
{
  "mapPackRoot": "assets/map_packs/current/"
}
```

加载规则：

1. 当前阶段 `mapPackRoot` 下只允许存在一个地图包文件夹。
2. 主游戏扫描该文件夹，读取其中的 `package.json`。
3. 再按 `package.json` 读取 `maps.json` 和资源路径。
4. 如果存在多个地图包文件夹，当前阶段应报错，不应随机选择。
5. 如果缺少 `package.json`、`maps.json` 或资源文件，应在加载期报明确错误。

这样主游戏只需要知道“地图包目录”，不需要知道编辑器内部文件命名。

## 7. 工作稿与发布包的区别

编辑器内部仍可保留工作稿概念，但界面应使用中文表达。保存、发布和导出的主体都是当前地图包，也就是当前故事。

1. `编辑工作稿`
   - 给编辑器反复修改。
   - 可以包含未发布状态、校验提示、草稿字段。
2. `发布地图包`
   - 给主游戏读取。
   - 必须经过校验。
   - 必须包含入口文件、地图数据和资源依赖。

界面不应直接展示 `authoring` 这个词。它可以继续作为文件名的一部分存在，但不应该成为用户操作语言。

## 8. 分阶段落地

### 阶段一：界面术语与操作分层

1. 去掉界面上的 `Authoring` 文案。
2. 顶部只保留地图包 / 地图工程操作。
3. 节点和边操作移动到右侧检查器。
4. “导出 / 写回”改成“导出 / 发布”。
5. 明确当前导出只是 manifest JSON。

### 阶段二：编辑器数据模型补顶层对象

1. 在工作区模型中加入单故事包语义。
2. 底层保留 `stories[]` 兼容字段，但编辑器只消费第一条故事。
3. 地图归属到章节。
4. 兼容读取旧 `maps[]`：旧数据自动放入默认故事和默认章节。

### 阶段三：地图级管理

1. 新建地图。
2. 复制地图。
3. 删除地图。
4. 地图排序。
5. 修改地图所属章节。

### 阶段四：目录式地图包发布

1. 生成 `package.json`。
2. 生成 `maps.json`。
3. 收集资源依赖。
4. 复制资源到包内 `assets/`。
5. 生成资源校验报告。

### 阶段五：主游戏目录加载

1. 配置 `mapPackRoot`。
2. 扫描唯一地图包目录。
3. 读取 `package.json` 和 `maps.json`。
4. 运行时继续只消费发布包，不读取编辑工作稿。

## 9. 当前结论

当前编辑器的问题不是“缺几个按钮”，而是缺少“地图包即故事”、章节、地图和目录式发布包这些顶层业务模型。

短期应先修正界面分层和术语，避免继续误导使用者。中期必须把编辑器从“固定 JSON 编辑页”升级为“单故事地图包编辑器”。长期再讨论多个地图包、多剧情线和主游戏中的包选择策略。
