## 1. 本文目标

本文目标不是直接给出“最优剑系技能树”，而是把剑系流血窗口设计抽象成一个可测试的问题。

今晚测试的目标是：在明确的 AP、KP、流血窗口、敌人压力和成长阶段约束下，把“剑系流血”先抽象成一组可购买、可组合、可升级的战术棋子，再搜索出 2-3 套“看起来合理、可解释、可继续人工审定”的候选技能方案。

这里的“棋子”不是新系统，而是一种设计抽象：每个技能像一枚棋子，有购买成本 KP、行动成本 AP、适用条件、即时效果、延迟收益、与其它棋子的组合关系。玩家前期买不起太多棋子，所以只能形成简单组合；随着 KP 增长，玩家逐步买入更多棋子，形成更复杂的战术组合。

因此，今晚的输出不一定是一棵唯一技能树。更合理的交付是 2-3 套候选方案，例如：

1. 保守候选：尽量继承当前 `004020` 的结构，只修正流血窗口与技能区分度。
2. 标准候选：保留核心位置逻辑，但允许重排局部前置、替换一批低区分度技能。
3. 重构候选：从“棋子组合”模型重新设计技能组，再检查它如何落回技能编辑器、坐标和前置关系。

“合理”不等于最优。合理表示：

1. 前期能玩，不只靠一个基础攻击。
2. 中期有策略选择，不是无脑重复同一个流血技能。
3. 后期有高阶兑现，但不能因为流血窗口滚大而压平其它路线。
4. 简单敌人、中等敌人、困难敌人下都有可用策略。
5. 技能之间有明确角色分工，而不是多个技能都只是“少量伤害 + 流血”。
6. 设计能落到当前 Skill/Buff 契约，不依赖单个技能 ID 的引擎特判。

本文不把 `004020` 当成唯一来源。`004020` 是强参考基线，因为它包含当前最新手工排布、坐标和已有剑系设计意图；但如果机制模型证明当前结构限制过大，可以提出重构候选。重构候选必须说明它为什么偏离 `004020`，以及后续如何落回编辑器中的节点位置和连线关系。

## 2. 问题抽象

### 2.1 成长预算问题

把玩家成长抽象为一条 KP 预算曲线。

基础场景：

```text
故事战斗数量：30 场
平均每场获得 KP：2-3
全程总 KP：60-90
流血系目标投入：50-60 KP 以内
```

这里的 30 场故事战斗只用于推导 KP 成长曲线，不表示今晚要模拟 30 个敌人或 30 场不同战斗。实际测试应使用少量阶段代表敌人，避免把问题复杂化到不可判断。

这里需要同时测试多个预算场景，而不是只用一个数：

| 场景 | 总 KP | 流血系可投入 KP | 用途 |
| --- | ---: | ---: | --- |
| 保守成长 | 60 | 35-45 | 每场约 2 KP，玩家还要买其它路线 |
| 标准成长 | 90 | 45-60 | 每场约 3 KP，允许较完整流血专精 |
| 专精压力 | 90 | 60 | 玩家重度投入流血系，检查是否过强 |

阶段切分：

| 阶段 | 累计可投入 KP | 设计问题 |
| --- | ---: | --- |
| 初期 | 0-15 | 是否有启动技能、基础破甲、轻量流血，不至于无聊 |
| 中期 | 16-35 | 是否形成建窗、读窗、控制或续航的循环 |
| 后期 | 36-60 | 是否有消窗终结、复杂联动和困难敌人处理手段 |

### 2.2 战斗资源问题

战斗内资源以 AP 为核心约束。

当前基础模型：

```text
基础 AP = 5
1 AP = 10 点指定部位即时护甲伤害
流血 = 每回合结束固定 5 点 HP 伤害
流血资源 = buff_bleed.remaining
```

测试时应保留 AP 紧缺感：

1. 初期按 5 AP 测。
2. 中期仍以 5 AP 为主，可少量测试 6 AP。
3. 后期可测试 6-7 AP，但不能假设 AP 无限增长。

流血窗口不能被当成免费伤害。每增加 1 回合流血窗口，名义收益是未来 5 点 HP 伤害，但存在延迟、目标死亡、状态清除和过量伤害风险。除此之外，剑系主动命中的普通伤害默认是指定部位护甲伤害，只有明确写成生命伤害/真实伤害/绕甲时才允许绕过护甲语义。

### 2.3 技能组合问题

流血系技能不是一组孤立技能，而是一组“棋子”。每个棋子必须有战术角色，并且要回答三个问题：

1. 玩家为什么现在买它？
2. 它和已购买棋子能形成什么组合？
3. 它是否让后续棋子有更明确的价值？

这意味着技能数量不是越多越好。需要设计的是一套“有限棋子集合”：它在 0-15 KP、16-35 KP、36-60 KP 三个阶段都能提供有意义的组合变化，而不是简单堆叠同类效果。

第一版角色集合：

| 角色 | 说明 | 典型约束 |
| --- | --- | --- |
| 基础破甲 | 不依赖流血，保证任何时候都能行动 | AP 效率接近护甲伤害基准，不附带强状态 |
| 建窗 | 增加流血窗口 | 直接护甲伤害低于基准 |
| 维窗 | 在目标已流血时延长窗口 | 需要 `targetBuff.minRemaining` 或其它条件 |
| 读窗 | 按流血窗口获得伤害、治疗、控制等收益，但不消耗窗口 | 必须设置 `amountSource.maxRead`，倍率偏低 |
| 消窗 | 消耗流血窗口换爆发或续航 | 高 AP、高 KP、状态门槛或低血门槛 |
| 窗口控场 | 目标流血达到门槛时施加减速、虚弱、易伤等 Buff | 不应同时给高护甲伤害 |
| 防反/保命 | 给剑系提供防止快速掉血的短窗口 | 短持续、每回合限制、低直接收益 |

从棋子组合角度，流血系至少要覆盖四类组合：

| 组合类型 | 说明 | 风险 |
| --- | --- | --- |
| 基础破甲组合 | 不依赖流血，保证基础可玩 | 如果太强，会压掉流血路线 |
| 建窗组合 | 用低即时护甲伤害换未来流血窗口 | 如果太慢，普通敌人来不及体现 |
| 读窗组合 | 不消耗窗口，按窗口状态获得低倍率收益 | 如果无上限，会在长战斗中失控 |
| 消窗组合 | 消耗窗口换爆发、回血或控制 | 如果倍率过高，会变成唯一正确玩法 |

### 2.4 敌人压力问题

技能合理性必须跨敌人类型验证。

第一版敌人抽象不依赖具体敌人 JSON，可先使用阶段代表模型。不要为 30 场故事战斗分别造敌人；只需要覆盖初期、中期、后期的压力差异。

| 阶段 | 代表敌人数量 | 血量/护甲 | 压力特征 | 设计验证点 |
| --- | ---: | --- | --- | --- |
| 初期 | 1-2 种 | 低血低压 | 少量攻击，少控制 | 流血不能比基础破甲慢到无意义 |
| 中期 | 1-2 种 | 250-300 HP，有稳定压力 | 玩家不能每轮全力输出 | 流血窗口应能提供跨回合收益 |
| 后期 | 2 种 | 高血、高护甲或机制压力 | 可能清状态、压 AP、打断防御 | 高阶读窗/消窗是否有意义但不过强 |

后期代表敌人不一定都要真实实现。今晚测试可以先用参数化敌人模型，例如：

1. 不清流血。
2. 每 3 回合清一次流血。
3. 高护甲、低 HP 暴露窗口。
4. 高 HP、低护甲、长战斗。

敌人模型的职责是给候选技能组施压，不是还原完整关卡生态。它只回答：这套棋子在短战斗、标准战斗、长战斗、清状态压力下是否仍有策略差异。

### 2.5 数学/逻辑模型抽象

这一节是今晚测试的核心。前面的“棋子”不能只作为比喻，必须转成可计算、可检查、可解释的模型。

#### 2.5.1 技能棋子模型

把每个技能抽象成一枚棋子 `s`：

```text
s = {
  id,
  role,
  kpCost,
  apCost,
  prerequisites,
  conditions,
  effects,
  buffReads,
  buffWrites,
  buffConsumes,
  tags
}
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `role` | 基础破甲、建窗、维窗、读窗、消窗、控场、防反等战术角色 |
| `kpCost` | 买入这枚棋子的成长成本 |
| `apCost` | 每次使用这枚棋子的战斗成本 |
| `prerequisites` | 购买前置，决定技能树形结构 |
| `conditions` | 使用条件，例如目标正在流血、HP 低于阈值 |
| `effects` | 即时伤害、治疗、护甲、状态等效果 |
| `buffReads` | 读取流血窗口但不消耗，例如按 remaining 造成低倍率收益 |
| `buffWrites` | 增加或维持流血窗口，例如 `remaining += 2` |
| `buffConsumes` | 消耗流血窗口，例如 `consume remaining <= 4` |
| `tags` | 用于分组、统计和报告，不直接驱动引擎特判 |

一个技能是否合理，不能只看单点效果，而要看它在棋子集合中承担什么角色。两个技能如果 `role`、`apCost`、`conditions`、`buffWrites` 都接近，就高度可疑，哪怕名字不同。

#### 2.5.2 成长购买模型

玩家在某个 KP 阶段拥有的技能不是任意集合，而是满足前置闭包的已购买棋子集合 `B(K)`：

```text
B(K) = { s | sum(kpCost(s)) <= K, and prerequisites(s) all in B(K) }
```

这里的 `K` 取三个阶段：

```text
K_early = 0-15
K_mid = 16-35
K_late = 36-60
```

这个模型回答的问题是：随着 KP 增长，玩家到底买到了多少枚有意义的棋子，以及这些棋子是否形成新的组合。

需要检查的不是“节点数量是否足够”，而是：

1. `B(K_early)` 是否至少形成基础攻击 + 轻流血两个选择。
2. `B(K_mid)` 是否至少形成一个完整流血循环，例如建窗 -> 读窗 / 控场 / 补刀。
3. `B(K_late)` 是否形成高阶兑现，但没有把其它路线压平。
4. 每个阶段新增的棋子是否改变玩法，而不只是数值变大。

#### 2.5.3 战斗状态模型

每一回合开始时，用一个简化状态 `X_t` 表示战斗局面：

```text
X_t = {
  enemyHp,
  enemyArmor,
  playerHp,
  playerArmor,
  bleedRemaining,
  enemyPressure,
  apBudget
}
```

其中 `bleedRemaining` 是流血窗口 `W`，只表示剩余回合数，不表示伤害层数。

每回合玩家选择一个动作序列 `A_t`：

```text
A_t = [s1, s2, ... sn]
sum(apCost(si)) <= apBudget
si in B(K)
conditions(si, X_t) == true
unique(id(si)) == true
```

也就是说，玩家不是只选一个技能，而是在 AP 预算内组合多枚棋子。技能设计的趣味性来自这个动作序列空间，而不是单个技能孤立强弱。

这里需要增加一个区别于一般卡牌游戏的约束：同一个技能在同一个 turn 内只能使用一次，但跨 turn 可以反复使用。也就是说：

```text
允许：
turn 1: A + B
turn 2: A + B
turn 3: A + B

允许：
turn 1: A + C
turn 2: B + D
turn 3: A + B + C

禁止：
turn 1: A + A + A
turn 1: A + A + B
turn 1: B + B + B
```

这个规则的目的不是制造冷却时间，而是在不引入冷却系统的前提下，阻止“同一枚棋子在单回合内被重复刷到最优”。因此，战斗动作空间应建模为：

```text
A_t is an ordered unique-skill sequence
A_t may repeat across turns
```

如果顺序会影响条件和状态，例如先建窗再读窗，则 `A_t` 仍然是有顺序的序列；但序列内部不能重复同一个技能 ID。

#### 2.5.4 流血窗口状态转移

流血窗口按回合状态转移：

```text
W_start = X_t.bleedRemaining
W_after_skill = W_start + writes - consumes
if W_after_skill > 0:
  enemyHp -= 5 at turn end
  W_next = max(0, W_after_skill - 1)
else:
  W_next = 0
```

关键约束：

1. 增加流血不是“叠层数”，而是增加 `remaining`。
2. 读窗技能可以读取 `remaining`，但必须有读取上限。
3. 消窗技能会消耗 `remaining`，必须扣除未来流血结算的机会成本。
4. 一回合流血如果在当回合结束时结算，结算后会变成 0。

因此，建窗技能的收益不是即时全额收益，而是一个延迟收益：

```text
nominalBleedValue = addedRemaining * 5
realBleedValue <= nominalBleedValue
```

真实收益会被目标死亡、清状态、过量伤害、AP 压力和回合数压缩。

#### 2.5.5 策略序列模型

给定一个候选技能组 `C`、一个 KP 阶段 `K`、一个敌人模型 `E`，测试脚本不需要寻找全局最优策略，只需要比较几类可解释策略序列：

```text
policy(C, K, E) -> [A_1, A_2, ... A_n]
```

策略包括：

1. 基础破甲优先。
2. 建窗优先。
3. 建窗后读窗。
4. 建窗后控场。
5. 大窗口后消窗。
6. 高压时插入防反/保命。

如果一个候选技能组只在某一种策略下有效，或者所有敌人下同一个策略都明显最强，说明这组棋子缺少博弈空间。

需要注意：同一套组合跨回合重复，不一定是失败。例如某场战斗中连续多回合使用 `A+B` 是允许且可以接受的，因为这代表玩家找到了一个稳定循环。真正不允许的是单回合内重复同一个技能，例如 `A+A+A` 或 `A+A+B`。

因此，测试脚本判断动作序列时应区分两类问题：

| 情况 | 是否允许 | 说明 |
| --- | --- | --- |
| 单回合内重复同一技能 | 不允许 | 违反 turn 内唯一使用规则 |
| 多回合重复同一技能组合 | 允许 | 表示稳定循环，不等同于冷却缺失 |
| 所有阶段、所有敌人都只用同一组合 | 可疑 | 不是规则违规，但说明技能组策略分化不足 |
| 为阻止重复而加入冷却时间 | 暂不采用 | 冷却会引入额外状态和复杂度 |

#### 2.5.6 候选搜索问题

今晚真正要解的问题可以描述为：

```text
Input:
  KP budgets K = {early, mid, late}
  enemy models E = {earlyEnemy, midEnemy, lateEnemyA, lateEnemyB}
  skill role templates R
  hard constraints H

Decision:
  candidate skill set C
  prerequisite graph G
  skill parameters P

Evaluate:
  simulate(policy, C, G, P, K, E)

Output:
  2-3 candidate plans ranked by explainability and risk
```

这里的决策对象不是单个技能，而是：

1. 需要多少枚棋子。
2. 每枚棋子的角色是什么。
3. 棋子之间是否有购买先后。
4. 不同 KP 阶段能形成哪些动作组合。
5. 面对不同敌人压力时，是否需要切换组合。

这比“生成一批技能 JSON”更重要。JSON 只是最后落地形式；模型先决定这批技能为什么存在。

#### 2.5.7 目标函数不是最大 DPS

如果把目标函数写成“最大伤害”或“最短击杀回合”，搜索一定会偏向数值漏洞。更合适的目标是多目标筛选：

| 目标 | 说明 |
| --- | --- |
| 功能完整 | 每个阶段至少有可用组合 |
| 策略分化 | 同阶段至少有 2 种可解释打法 |
| 成长连续 | 新买入棋子会改变玩法，而不只是加数值 |
| 数值克制 | 长期 AP 效率不能压过全局基准 |
| 反滥用 | 没有一个循环能通吃所有敌人 |
| 实现健康 | 不依赖技能 ID 特判，不引入高耦合机制 |
| 可编辑性 | 能落回技能编辑器的节点、前置和坐标 |

这意味着评分表只是模型输出的一部分。真正的判断顺序是：

```text
先过硬约束 -> 再看策略矩阵 -> 再看软评分 -> 最后人工审定
```

## 3. 可测试变量

### 3.1 技能数量与 KP 分布

今晚不应直接假设“技能越多越好”。应扫描不同规模：

| 规模 | 技能数 | 总 KP | 用途 |
| --- | ---: | ---: | --- |
| 小型流血包 | 8-10 | 35-45 | 验证核心循环是否成立 |
| 标准流血包 | 11-14 | 45-60 | 作为主候选规模 |
| 大型流血包 | 15-18 | 60+ | 检查是否过度挤占剑系其它路线 |

KP 分布建议：

| 层级 | KP 倾向 |
| --- | ---: |
| 基础节点 | 1 |
| 低阶工具 | 1-2 |
| 中阶循环 | 3-5 |
| 高阶兑现 | 5-8 |
| 终点技能 | 8 |

### 3.2 AP 与效果参数

可扫描的技能参数：

| 参数 | 候选范围 |
| --- | --- |
| 基础破甲 AP | 1-3 |
| 建窗 AP | 1-2 |
| 建窗 W | 1-3 |
| 非消耗读窗倍率 | 2-4 |
| 非消耗读窗 `maxRead` | 3-5 |
| 消窗倍率 | 6-10 |
| 消窗上限 | 3-8 |
| 控场门槛 | W >= 2 / 3 / 5 |
| 终结血线 | HP < 35% / 30% / 25% |

### 3.3 技能关系变量

要测试的不只是数值，还有结构。

结构变量：

1. 建窗技能数量：2-4 个。
2. 读窗技能数量：2-4 个。
3. 消窗技能数量：1-3 个。
4. 控场/防反技能数量：2-4 个。
5. 是否存在跨分支前置，例如流血 + 招架共同解锁终结。
6. 是否允许玩家在初期横跳到非流血剑系技能。

## 4. 合理性指标

评分不能替代人工审定。今晚的目标不是通过公式找到“最高分最优解”，而是用硬约束淘汰明显错误方案，再用软评分排序几个值得看的候选。

如果评分设计过强，可能导致没有任何候选通过；如果评分过弱，又会回到凭感觉堆技能。因此本轮采用“硬约束 + 软评分 + 放松顺序”的结构。

### 4.1 硬约束

硬约束用于淘汰无效候选，不参与加分。

| 约束 | 说明 |
| --- | --- |
| 流血语义 | 流血只使用 `buff_bleed.remaining`，表示持续回合数，不存在流血层数伤害 |
| 流血结算 | 流血在回合结束结算，固定 5 HP；重复施加是增加 remaining |
| 读窗上限 | 非消耗读窗必须有读取上限，例如 `amountSource.maxRead` |
| 消窗明确 | 消窗技能必须明确消耗多少 remaining，不能既吃窗口又不付代价 |
| AP 紧缺 | 不得假设 AP 无限增长；主要测试 5 AP，后期可测试 6-7 AP |
| 单回合唯一 | 同一个技能 ID 在一个 turn 内只能使用一次；跨 turn 可重复使用 |
| 契约健康 | 不允许为单个技能 ID 写引擎特判；新增机制必须是通用 Skill/Buff 契约 |
| 阶段可用 | 0-15 KP、16-35 KP、36-60 KP 都必须有可解释行动组合 |
| 反单解 | 不能出现所有阶段、所有敌人都只推荐同一个动作序列 |

`004020` 不作为硬约束。它是继承型候选的输入基线，也是重构型候选的对照组。

### 4.2 数值指标

每个候选方案至少输出以下指标：

| 指标 | 意义 |
| --- | --- |
| 单 AP 即时伤害 | 普通主动命中不能长期超过 `1 AP = 10 点指定部位护甲伤害` 基准；HP 直伤必须单独标记为稀有特性 |
| 流血窗口增长速度 | 不能 1-2 回合就堆出过高 W |
| 非消耗读窗收益 | 需要低倍率和 `maxRead` |
| 消窗净收益 | 要扣除被消耗 W 的未来 5 点/回合名义流血 |
| 达成核心循环所需 KP | 初中期是否能玩到完整循环 |
| 满配总 KP | 是否压缩其它剑系/跨系选择 |

### 4.3 策略指标

每个阶段应至少存在 2 种可解释的打法：

| 阶段 | 可接受策略 |
| --- | --- |
| 初期 | 基础破甲速杀 / 轻流血延迟收益 |
| 中期 | 建窗读窗 / 建窗控场 / 基础破甲补刀 |
| 后期 | 大窗口消窗 / 流血续航 / 防反保命后反打 |

如果某个阶段只有一个明显最优动作序列，则方案不合格。

### 4.4 反滥用指标

必须检查以下失败模式：

1. 只反复施加流血就比所有技能强。
2. 非消耗读窗技能在大 W 下反复刷收益。
3. 消窗技能虽然消耗 W，但倍率过高导致净收益失控。
4. 控场技能附带过高伤害，导致控制和输出合一。
5. 低 KP 节点提供高 AP 效率，导致成长经济失衡。
6. 高阶技能只在长血条敌人有用，普通战斗没有存在感。

### 4.5 软评分

软评分只用于排序和解释，不作为绝对真理。一个候选低分时，应先看失败原因是否符合直觉，再决定是调整技能、调整评分，还是保留给用户审定。

| 维度 | 分数 | 说明 |
| --- | ---: | --- |
| 初期可玩性 | 0-5 | 0-15 KP 时是否有选择，不是只按基础攻击 |
| 中期循环性 | 0-5 | 16-35 KP 时是否形成建窗、读窗、消窗或防反循环 |
| 后期兑现感 | 0-5 | 36-60 KP 是否有高阶目标，但不压平其它路线 |
| 敌人适应性 | 0-5 | 初期/中期/后期代表敌人下是否都有意义 |
| 反滥用安全 | 0-5 | 是否避免单策略碾压 |
| 实现契约健康 | 0-5 | 是否复用通用 Skill/Buff 机制 |
| 树形可读性 | 0-5 | 前置关系、KP 层级、编辑器排布是否容易理解 |

评分建议门槛：

```text
总分 >= 26 / 35 可作为推荐候选
总分 22-25 / 35 可作为备选候选
任一核心维度低于 3，需要在报告中解释
反滥用安全和实现契约健康低于 4，不推荐直接落地
```

### 4.6 约束过强时的放松顺序

如果没有候选通过，不应立刻推翻流血定义。优先按以下顺序放松：

1. 放松软评分门槛，保留失败说明。
2. 扩大 KP 预算范围，例如把标准专精从 45-60 KP 扩到 40-65 KP。
3. 增删 1 个工具棋子，例如补一个低阶建窗或低阶基础破甲。
4. 微调 AP、伤害、remaining 增减值。
5. 调整前置结构，让关键组合更早或更晚成型。
6. 最后才考虑新增 Buff；新增 Buff 必须单独列给用户审定。

## 5. 搜索方法

### 5.1 不做最优解搜索

今晚不做“找最高 DPS 构筑”的最优解搜索，因为这会把设计导向数值钻空子。

应做“合理候选搜索”：

1. 固定一组技能角色模板。
2. 在每类模板里扫描少量参数。
3. 用几种敌人压力模型跑脚本策略。
4. 输出通过合理性指标的候选方案。
5. 把失败方案记录下来，说明为什么失败。

候选来源分三类：

| 类型 | 来源 | 用途 |
| --- | --- | --- |
| 继承型 | 基于 `004020` 修改 | 验证当前手工排布能否通过局部调整变好 |
| 半重构型 | 保留主要层级，重做技能角色和局部前置 | 作为最可能落地的主候选 |
| 重构型 | 从棋子组合模型重新设计 | 检查是否存在比当前树更清晰的机制结构 |

三类候选都要回到同一套硬约束和软评分下比较。

### 5.2 脚本策略

测试时不需要写复杂 AI。先用几种可解释策略：

| 策略 | 说明 |
| --- | --- |
| 基础破甲优先 | 能杀就杀，不主动堆 W |
| 建窗优先 | 先把 W 堆到目标阈值，再读窗 |
| 控场优先 | W 到门槛后优先减速/虚弱 |
| 读窗优先 | W 达标后使用非消耗读窗 |
| 消窗终结 | 低血或大 W 时使用消窗技能 |
| 防反保命 | 受到高压时插入短期防反/减伤 |

这些策略不是玩家最终 AI，而是测试工具，用来验证技能组是否提供多种可解释路线。

### 5.3 输出候选

每轮搜索的目标输出不是一个抽象分数，而是 2-3 套可审定候选方案。每套候选至少要说明：

1. 技能节点数量。
2. 总 KP 与阶段 KP 分布。
3. 每个技能的棋子角色。
4. AP 成本、伤害、流血 remaining 操作、Buff 关系。
5. 前置关系和关键组合路径。
6. 与 `004020` 的关系：继承、半重构或重构。
7. 在阶段代表敌人下的测试结论。
8. 软评分和失败风险。

最终由用户判断哪套方案进入技能编辑器细化，不由脚本自动覆盖正式技能包。

### 5.4 最小仿真输出

每个候选方案至少输出一张阶段 x 敌人压力矩阵，避免只凭文字判断。

推荐表头：

| 阶段 | 敌人模型 | 策略 | 击败回合 | 总伤害 | 流血结算伤害 | 读窗收益 | 消窗收益 | W 峰值 | 平均 AP 使用 | 结论 |
| --- | --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 初期 | 简单敌人 | 基础破甲优先 |  |  |  |  |  |  |  |  |
| 初期 | 简单敌人 | 建窗优先 |  |  |  |  |  |  |  |  |
| 中期 | 中等敌人 | 建窗读窗 |  |  |  |  |  |  |  |  |
| 中期 | 清流血敌人 | 控场优先 |  |  |  |  |  |  |  |  |
| 后期 | 困难敌人 | 消窗终结 |  |  |  |  |  |  |  |  |

矩阵不是为了证明某个策略最高效，而是为了发现设计问题：

1. 如果所有敌人下都是同一个策略最强，说明技能组缺少策略分化。
2. 如果简单敌人下流血永远来不及结算，说明低阶流血体验过慢。
3. 如果清流血敌人下流血路线完全失效，说明缺少基础破甲、防反或低窗口替代动作。
4. 如果困难敌人下 W 峰值过高且非消耗读窗反复获利，说明读窗上限或窗口增长速度有问题。
5. 如果消窗收益占总伤害过高，说明终结技可能压平其它技能。

## 6. 今晚执行流程

### 6.1 输入基线

基础输入文件：

```text
assets/skill_packs/authoring/skills_melee_v4_5_sword_rework_20260528_004020.json
assets/data/buffs_v2_7.json
```

`004020` 是当前最新手工技能树基线，应优先读取它的剑系节点、坐标、前置关系和已有字段。不得再从过时的 `assets/data/skills_melee_v4_5.json` 直接派生玩家剑系候选。

但候选方案可以分为：

1. 继承型：直接从 `004020` 派生 JSON。
2. 半重构型：用 `004020` 做坐标和层级参考，但允许替换局部技能和前置。
3. 重构型：先产出技能组合设计表，再单独评估如何落到编辑器布局。

如果输出的是重构型方案，可以先不生成完整 JSON，但必须说明后续落到技能编辑器时需要新增、删除、移动哪些节点。

### 6.2 测试步骤

1. 读取 `004020` 的剑系节点、坐标、前置关系和已有技能字段。
2. 建立流血技能角色模板，把技能抽象为可购买棋子。
3. 生成 2-3 个候选流血方案：
   - 保守继承方案：基于 `004020`，技能少，数值稳。
   - 标准半重构方案：技能数量中等，策略较完整，作为主候选。
   - 重构探索方案：不强制沿用 `004020` 的局部结构，用来验证更清晰的棋子组合。
4. 对每个方案跑 KP 阶段测试：
   - 0-15 KP
   - 16-35 KP
   - 36-60 KP
5. 对每个方案跑敌人压力测试：
   - 简单敌人
   - 中等敌人
   - 困难敌人
   - 清流血敌人
6. 输出评分表、失败模式和推荐候选。
7. 输出 2-3 套候选供用户判断；只有被选中的候选才写入 authoring JSON。
8. 不发布到正式运行时包，除非用户明确要求。
