# 状态机 vs 工作流（示例文档）

本文档以本仓库 `NodeConsoleApp2` 的“时间轴战斗/技能系统”（`CoreEngine` / `TimelineManager` / `TurnPlanner` / 技能与 `buffs_*.json` 等）为例，对比 **状态机（FSM）** 与 **工作流（Workflow/编排）** 在建模粒度、约束方式与工程落地上的差异，并给出可复用的建模模板。

## 1. 先给结论（工程视角）

- **状态机不是“必须显式存在”**，但只要业务对象存在“状态 + 变更规则”，**隐式状态机一定存在**。
- **工作流不是状态机的替代品**，而是更偏“跨角色/跨系统/长事务”的**流程编排层**；工作流节点里通常仍然要落到某个业务对象的状态机转移。
- 最稳妥的组合：
  - **工作流**：描述“人和系统如何协作把事做完”（步骤、角色、SLA、回退、通知、并行）。
  - **状态机**：保证“业务对象在任何时刻都处于合法状态，并且只能按允许的规则变化”。

---

## 2. 概念对照：使用流程 / 工作流程 / 状态机

| 概念 | 关注点 | 常见产物 | 典型问题 |
|---|---|---|---|
| 使用流程（Use Flow / User Journey） | 用户怎么点、页面怎么走 | 原型、页面流转图 | “用户能不能完成任务？” |
| 工作流程（Workflow / Business Process） | 角色怎么交接、任务怎么流转 | 泳道图、BPMN、流程定义 | “组织怎么把事做完？” |
| 状态机（FSM / Lifecycle） | 单一业务对象状态如何变化 | 状态图、转移表、领域规则 | “对象状态是否一致且可验证？” |

要点：
- 使用流程 ≈ 前台视角
- 工作流程 ≈ 组织/协作视角
- 状态机 ≈ 业务一致性/规则视角

---

## 3. 示例系统：时间轴战斗（Timeline Battle）

本示例尽量对应仓库中已经出现的模块与数据文件：

- 引擎：`script/engine/CoreEngine.js`
- 时间轴：`script/engine/TimelineManager.js`
- 回合/行动规划：`script/engine/TurnPlanner.js`
- 数据：`assets/data/skills_*.json`、`assets/data/buffs_*.json`、`assets/data/player.json`、`assets/data/enemies.json`

### 3.1 关键业务对象（建议以“对象生命周期”看待）
- `Battle`：一场战斗实例（参与单位、当前时间、随机种子/日志等）
- `Unit`：单位（玩家/敌人），有 HP、资源、位置、Buff 列表等
- `SkillCast`：一次技能释放实例（施法者、目标、选择参数、结算结果）
- `BuffInstance`：一个 Buff 的实例（来源、层数、持续时间、触发条件等）

### 3.2 关键“流程”（建议以“任务编排”看待）
- “一次技能释放如何被拆解为多步骤并沿时间轴推进”
- “一次回合/一次单位行动如何被规划并执行（计划 -> 执行 -> 结算 -> 推进时间）”

## 4. 只用状态机如何建模？（对象生命周期）

### 4.1 示例 1：`Battle` 的状态集合（战斗实例生命周期）

示例状态（可按你当前引擎实际字段命名调整）：
- `Init`：已创建但未加载/未布阵
- `Ready`：数据加载完成，时间轴初始化完成
- `Running`：战斗进行中（时间轴推进/事件结算中）
- `Paused`：暂停（用于调试/编辑器验证/单步）
- `Ended`：战斗结束（胜/败/撤退）
- `Aborted`：异常终止（关键数据缺失/逻辑错误）

### 4.2 状态转移（转移表）

| 当前状态 | 事件/命令（Command） | 条件（Guard） | 下一状态 | 副作用（Side Effects） |
|---|---|---|---|---|
| `Init` | `LoadData` | 关卡/单位/技能/Buff 数据齐全 | `Ready` | 时间轴初始化、写入初始日志 |
| `Ready` | `Start` | 单位列表非空、起始顺序确定 | `Running` | 产生首批可执行事件 |
| `Running` | `Pause` | 调试模式启用 | `Paused` | 冻结推进、保留现场 |
| `Paused` | `Resume` | - | `Running` | 继续推进 |
| `Running` | `End` | 胜负条件满足 | `Ended` | 结算统计、输出战报 |
| `Init`/`Ready`/`Running` | `Abort` | 关键数据缺失或检测到不一致 | `Aborted` | 抛错并保留错误上下文 |

### 4.3 状态机适用点（在本系统里具体是什么）

适用于：
- 需要防止非法跳转：例如 **未 `Ready` 不允许 `Start`**，战斗 `Ended` 后不允许再推进时间轴。
- 需要保证一致性：例如 `Unit` 的 HP 不应被结算成负无限；`BuffInstance` 到期必须从单位身上移除。
- 需要可测试：对每个命令（如 `Start` / `Pause` / `End`）验证“给定状态 + 输入 => 是否允许”。

## 5. 只用工作流如何建模？（流程编排）

### 5.1 示例 2：“一次技能释放”的流程节点（编排视角）

把一次 `SkillCast` 拆成步骤（节点可以是同步或异步事件），并在 `TimelineManager` 上排程：

1. `SelectTargets`：根据技能规则与 UI 选取目标
2. `PayCost`：扣资源/扣弹药/进入冷却
3. `BuildTimelineEvents`：生成计划事件（例如“前摇结束”“伤害结算”“后摇结束”）
4. `ResolveHit`：命中/闪避/格挡判定
5. `ApplyDamage`：计算与写入伤害
6. `ApplyBuffs`：挂载/刷新 Buff（来自 `assets/data/buffs_*.json`）
7. `TriggerHooks`：触发被动/反应（例如受击触发）
8. `FinishCast`：释放结束，推进时间轴到下一可执行点

### 5.2 工作流（编排）在本系统里的意义

适用于：
- 你希望把“技能结算”做成可插拔的步骤链（便于新增结算规则而不把逻辑塞进一个巨大函数）。
- 你希望支持并行/分支：例如多段伤害（并行结算）、连锁反应（分支触发）。
- 你希望做可视化调试：时间轴上每一步是什么事件、由什么规则产生、执行耗时/顺序如何。

### 5.3 仅靠编排的风险（在本系统会表现为）

如果没有对象级状态机约束，常见问题：
- 技能流程执行到 `ApplyDamage`，但 `Unit` 已死亡/已离场仍被写入 HP。
- Buff 的到期/移除没有强约束，导致“显示上结束了但数据仍在”。
- 时间轴事件被重复执行（或回放）时缺少幂等约束，导致重复扣资源/重复结算。

## 6. 推荐组合：编排推进 + 状态机保证一致性（强约束）

> 注意：这里的“兜底”不是做回退路径，而是指 **以对象状态机作为最终一致性约束**。

### 6.1 推荐分层（对应你当前工程模块）
- **编排/调度层**（更像“工作流”在执行）：
  - `TimelineManager`：事件队列、时间推进、事件执行顺序
  - `TurnPlanner`：生成行动计划/下一步候选
- **领域一致性层**（更像“状态机”在把关）：
  - `CoreEngine`（或其内部 domain 模块）：
    - 约束 `Battle` / `Unit` / `SkillCast` / `BuffInstance` 的合法状态与合法转移
    - 所有会改变核心数据的写操作都必须走“命令 + 校验”

### 6.2 典型调用模式（时间轴事件驱动命令）

- 时间轴事件执行时，调用领域命令（示例命名）：
  - `engine.castSkill(casterId, skillId, targetSpec)`
  - `engine.applyDamage(sourceId, targetId, amount, tags)`
  - `engine.addBuff(targetId, buffId, stacks, duration)`
- 领域命令内部必须做：
  - 校验当前对象状态：例如目标是否仍 `Alive`、战斗是否 `Running`
  - 校验转移合法性：例如 `SkillCast` 是否允许从 `PayCost` 进入 `ResolveHit`
  - 写入状态并记录可回放的事件/日志（用于调试与复盘）

## 7. 一个快速判断：这件事更像状态机还是工作流？

用下面问题做筛选：

### 更偏状态机（对象生命周期）
- 是否存在明确“状态集合”？
- 是否存在严格“允许/禁止”的状态跳转？
- 是否更关注一致性、幂等、并发正确性？

### 更偏工作流（流程编排）
- 是否有多角色交接、多步骤、长等待？
- 是否需要并行分支、会签/或签、动态路径？
- 是否需要可视化配置与流程监控报表？

## 8. 可复用模板（建模时直接填）

### 8.1 状态机模板
- 业务对象：`<Object>`
- 状态集合：`<S1, S2, S3...>`
- 命令集合：`<C1, C2, C3...>`
- 转移表：
  - `<Sx> + <Cy> [Guard] -> <Sz> / <SideEffects>`
- 不变式（Invariant）：
  - 例：`Ended` 状态下不得再生成时间轴事件；`Dead` 单位不可被治疗（除非复活技能显式允许）

### 8.2 工作流模板
- 流程名称：`<WorkflowName>`
- 参与者/触发源：`<UI, AI, TimelineEvent, PassiveTrigger...>`
- 节点列表：`<Node1..NodeN>`
- 节点输入/输出：
  - 输入：`battleId / casterId / skillId / targetSpec`
  - 输出：`events[] / logs[] / stateDelta`
- 分支条件：`<if ... else ...>`
- 约束：`<事件幂等键、重放策略、异常即抛错>`
- 审计：`<记录哪些字段、哪些操作>`

## 9. 落地建议（前后端系统常用做法）

- 后端：
  - 至少把“状态枚举 + 合法转移 + 守卫条件”集中管理（可测试）。
  - 所有写操作使用“命令式 API”而不是随意 `update status`。
- 前端：
  - UI 显示使用流程（操作路径），但按钮可用性由后端状态机最终裁决。
  - 前端可基于状态机元数据做“按钮显隐/禁用”，降低误操作。
- 数据层：
  - 状态字段建议不可空；对关键状态变更写审计表（谁在什么时候做了什么）。
