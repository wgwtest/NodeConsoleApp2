   - `Todo`
     - 尚未正式进入执行
   - `In Progress`
     - 正在实现
     - 或正在按验收意见返修
   - `Hum Check`
     - 已实现
     - 已自测
     - 正在等待用户人工核验或重测
   - `Done`
     - 用户已在对应验收清单中给出明确 `【通过】` 或同义结论
     - 且本地清单、Issue、Project 已完成同步

## 4. 验收与证据规则

### 4.1 人工验收清单命名

当前工程继续采用更强约束的命名方式：

1. `WBS编号-[状态]-YYYY-MM-DD-HHMMSS-主题-人工验收清单.md`

### 4.2 自测、验收、结论分层

1. 自测证据进入 `01_自测报告/`
2. 人工执行脚本进入 `02_验收清单/`
3. 线下或历史记录进入 `03_验收记录/`
4. 汇总性的最终验收结论进入 `04_验收结论/`
5. 用户批注分析进入 `06_过程文档/03_验收意见处理/`

补充规则：

1. `00-当前待验收功能总入口.md` 只允许列出 `待验收` 或 `未通过待重测` 节点。
2. `已通过` 节点不得继续停留在“当前待验收”入口中。
3. `已通过` 节点如仍需保留参考价值，可继续保留在 `验收清单/` 目录及 README 索引中，但应与待验收入口分离。
4. 某个 `WBS` 节点被用户明确判定人工验收通过后，同一轮必须同步：
   - 更新该清单文件状态为 `[已通过]`
   - 从 `00-当前待验收功能总入口.md` 移除
   - 更新 GitHub Issue 正文中的 `当前状态`
   - 更新 GitHub Project 中对应条目的粗粒度状态
5. 对人工验收清单，默认只把用户写在 `【】` 内的内容视为正式用户输入。
6. 用户不会主动替开发者维护 `状态`、`验收结论`、复选框或其他占位字段；这些字段必须由 Codex 在读取 `【】` 后自行同步。
7. 如果开发者预填字段与用户 `【】` 批注冲突，应一律以 `【】` 内容为准。
8. 活跃清单一旦吸收过用户批注并准备进入下一轮重测，旧批注不继续保留在活跃清单正文中。
9. 默认做法是先在 `06_过程文档/03_验收意见处理/` 留痕，再刷新出干净版活跃清单。
10. 开发者在新写的活跃清单、策略文档和说明文档中，不主动使用用户批注保留标记。

### 4.4 独立测试页可理解性要求

当前工程对 `test/*.html` 这类独立测试页、I/O 验证页、probe 页和回归辅助页，增加以下固定要求：

1. 页面必须能让不读源码的验收者理解：
   - 这是个什么页面
   - 它验证什么目标
   - 为什么点击某个按钮就能证明某项功能
2. 页面说明默认至少应包含：
   - 页面用途
   - 关键按钮说明
   - 为什么这能证明
   - 关键词是什么意思
   - 与其他测试页、主流程页、验收场景之间的关系
3. 如果测试页涉及跨页验证，必须明确区分：
   - 当前页内存改动
   - 浏览器 `localStorage` 等站点级存储
   - 刷新或重新打开后才会生效的页面
   - 已经打开但不会自动同步的页面
4. 对说明内容较长的测试页，默认优先纵向阅读布局，不为了压缩空间牺牲理解成本。
5. 当前工程后续新增或重写下列页面时，都应默认遵守本条：
   - `test/skill_editor_io_test.html`
   - `test/buff_editor_io_test.html`
   - `test/level_editor_io_test.html`
   - 对应 `runtime_probe` 页
   - `test/codex_regression_runner.html`
