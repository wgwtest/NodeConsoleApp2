本节点属于：

## 2. 本节点目标

本节点应交付：

1. 正式关卡编辑器页面壳
2. 关卡列表、详情、波次与敌人池编辑入口
3. 章节信息、线性 fallback 展示元数据、奖励信息等作者交互
4. `unlockRules`、`victoryCondition`、`failureCondition` 的正式编辑入口
5. JSON 导入入口与正式入口/历史入口关系说明
6. 基于工作区真值而不是 DOM 临时状态的保存链

## 3. 责任边界

本节点负责：

1. 关卡列表与当前关卡切换
2. 章节与排序展示
3. 波次、敌人池、奖励等关卡内对象编辑
4. 页面内即时校验信息展示
5. 正式页中对导入 / Probe / 历史 I/O 页关系的明确说明

本节点不负责：

1. 运行时重装载是否成功
2. 下载文件、导入文件与跨页验证证据闭环
3. 地图节点、边和背景编辑

## 4. 当前建议实现切片

建议本叶子以“先可维护、再可扩展”为原则推进：

1. 保持 `LevelEditorPage` 只做页面绑定与交互编排
2. 继续把字段规范、默认值、修剪规则压回 `LevelPackWorkspace`
3. 页面上优先落与当前主流程强相关的作者对象：
   - 关卡基础信息
   - 章节与线性 fallback 展示元数据
   - 波次与敌人池
   - 奖励与提示信息
4. 把 `unlockRules`、`victoryCondition`、`failureCondition` 作为正式字段区落地，但仍保持页面编排与数据核心解耦
5. 让 JSON 导入入口直接并入正式编辑器，而不是继续要求作者回到历史 I/O 页

## 5. 验收标准

通过标准至少包括：

1. `level_editor_v1.html` 能作为正式入口独立打开
2. 页面改动能稳定写回工作区真值
3. 页面能直接提供 `unlockRules` / `victoryCondition` / `failureCondition` / JSON 导入入口
4. `node --test test/level_editor_page.test.mjs` 能覆盖关键交互与页面语义
5. 页面职责仍保持在“编辑工作区”边界内

## 6. Roadmap 时间

1. 开始：`2026-04-14`
2. 目标：`2026-04-15`

## 7. 当前结论

`WBS-3.2.2.2` 的意义不是“再加几个输入框”，而是把关卡编辑器正式确认为一个稳定作者工具，而不是历史样本页。
