## 2. 为什么 WBS-5 需要单独拆出 WBS-5.2

即使版本功能本身已经可以玩，如果交付时还依赖开发者口头说明，就不算真正可交付。

发布前至少要收口四类信息：

1. 新环境怎么启动
2. 当前版本包含什么
3. 已知问题有哪些
4. 交付目录里每类内容是干什么的

因此 `WBS-5.2` 的职责，是把“可运行版本”变成“别人能拿去复核和继续协作的版本”。

## 3. 当前切片策略

`WBS-5.2` 当前按 `Prep` 启动，先整理说明和目录口径，不先做安装器或复杂打包系统。

本轮原则：

1. 先固定启动说明和环境要求
2. 先固定版本说明、已知问题和交付目录清单
3. 先把本地文档层与 GitHub 协作层的入口关系写清楚
4. 不在本轮提前承诺超出当前工程形态的分发方式

`WBS-5.2` 不是在验证“文档写得多不多”，而是在验证“这个版本是否已经具备交付和复核的最小说明能力”。
