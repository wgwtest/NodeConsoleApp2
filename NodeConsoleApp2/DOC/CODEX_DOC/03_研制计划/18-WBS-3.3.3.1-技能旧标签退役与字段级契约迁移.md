## 2. 为什么要从 WBS-3.3.3 中单独拆出这个子节点

`2026-04-11` 下午的技能人工测试再次说明，当前许多技能异常并不是“单条技能配错了标签”这么简单，而是：

1. 历史 `skill tags` 被若干正式功能、测试页和契约校验反向当成事实来源；
2. 一旦 `tags` 与 `target / actions / buffRefs / editorMeta` 漂移，异常会同时出现在：
   - 面板分类
   - 技能树过滤
   - 批量契约校验
   - 共享回归页说明
3. 如果继续把这类问题只挂在 `batch_target_tags / batch_effect_tags` 的余量统计里，`WBS-3.3.3` 会长期停留在“知道有问题，但没有独立承接对象”的状态。

因此，`WBS-3.3.3.1` 的职责不是“补几条 tag”，而是把 tag 漂移的根因单独收口：

1. 明确哪些正式功能仍依赖旧标签；
2. 逐项切换到正式字段或派生视图；
3. 再把旧标签从活动工程概念中退役。

## 3. 当前目标

本节点要解决的是：

1. 把 `skill tags` 相关异常从“问题码现象层”推进到“字段契约根因层”；
2. 让 `WBS-3.3.3` 的剩余异常不再以“继续修 tag”这种模糊口径挂起；
3. 给后续 skill editor、skill panel、skill tree、contract probe 和 regression runner 提供统一的去 tag 化迁移顺序。

## 6.1 当前实现结论

当前已落地：

1. `UI_SkillTreeModal` 只以 `editorMeta.hiddenInSkillTree` 决定节点可见性，不再读取 `DEMO` 之类的旧标签。
2. `UI_SkillPanel` 的类型标签、搜索与图标改为从 `actions / buffRefs / target` 派生，不再把旧标签当成事实来源。

4. `skillEditor / skill_balance_tool / skill_contract_probe / codex_regression_runner` 已切换到去标签口径。
5. 活动技能包 `skills_melee_v4_5.json` 已移除旧标签字段、旧附加参数字段和旧枚举定义。
