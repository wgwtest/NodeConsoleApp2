## 2. 为什么要按 3.2.3 模式重构

`WBS-3.2.2` 的功能基线其实已经存在：

1. `LevelPackWorkspace` 已承担纯数据工作区核心
2. `level_editor_v1.html` 已承担正式关卡编辑入口
3. `level_runtime_probe.html` 已承担运行时消费验证页

但此前的计划文档仍停留在“执行清单式单节点”状态，会导致三个问题：

1. 看树时无法分辨“已通过的历史基线”和“接下来继续深化的正式子任务”
2. 无法像 `WBS-3.2.3` 一样，仅通过父子编号就判断输入输出链与推进顺序
3. 后续再扩关卡编辑能力时，容易重新退回“补页面 / 补测试入口”的零散工作模式

因此本节点必须像 `WBS-3.2.3` 一样，转成：

1. 父节点负责稳定定位、边界、总输入输出和推进顺序
2. 子节点分别承担数据核心、编辑工作区、导入导出与运行时验证闭环

## 3. 本节点目标

本节点最终应交付：

1. 一个与主流程低耦合的正式关卡编辑器工作区
2. 一套由数据核心驱动的 `levels.json` 编辑真值
3. 章节、线性 fallback 元数据、波次、敌人池、奖励等最小作者能力
4. 可被 `DataManagerV2` 和 `WBS-3.2.3` 稳定消费的关卡导出链
5. 独立的导入导出、override 与运行时 probe 验证路径

## 4. 与其他节点的边界

### 4.1 与 `#69 / WBS-3.2.1`

1. `#69` 负责定义“关卡数据应该长什么样”
2. `#79` 负责把这些定义落成可编辑、可校验、可导出的作者工具链
3. `#79` 不额外发明另一套独立关卡语义

### 4.2 与 `#101 / WBS-3.2.3`

1. `#79` 维护关卡运行时配置与 `levelId`
2. `#101` 只消费这些 `levelId` 与章节/展示元数据
3. `#101` 不反向决定战斗规则
4. `#79` 也不直接承担地图视觉组织与图结构维护
