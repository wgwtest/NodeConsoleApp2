最后整理时间：2026-04-10

## 1. 目的

本表用于承接当前工程从“阶段字母 + 层级编号”写法切换到“纯数字 WBS 编码”后的阅读与追溯规则。

从本轮开始：

1. GitHub Issue tree 与 GitHub Project 标题统一以 `WBS-数字编号` 为正式主编号
2. `DOC/CODEX_DOC/03_研制计划/` 下的当前有效计划文档统一以 `WBS-数字编号` 作为正文主编号
3. 历史计划、历史验收清单、自测报告、会话交接和旧文件路径中保留的 `A1 / C2.1 / D3.2` 等写法，不再单独纠正
4. 当阅读到旧编号时，统一以本映射表回看

## 2. 适用范围

本表主要服务三类场景：

1. 当前有效文档需要和 GitHub 真源保持一致
2. 历史文档、历史证据文件仍保留旧编号，不能机械改名
3. 同一个节点在“当前计划正文”和“历史证据文件名”中可能同时出现新旧两种写法

补充说明：

1. 当前有效文档以新编号为准
2. 历史文档保留旧编号，不视为错误
3. 部分当前有效文档的文件路径仍带有旧阶段简称后缀，这是为了避免批量断链；遇到这类情况，以文档标题和正文中的新 WBS 编号为准

## 7. 阅读规则

当同一个节点同时出现在不同载体时，统一按下面方式理解：

1. GitHub issue 标题、Project 标题、当前有效计划正文：
   - 一律以 `WBS-数字编号` 为准
2. 历史计划、历史验收清单、自测报告、交接记录、旧文件路径：
   - 允许继续使用 `A1 / C2.1 / D3.2` 等旧写法
3. 若当前有效正文与历史文件名出现新旧并存：
   - 以正文节点号为主
   - 以历史文件名作为证据定位手段
4. 若用户批注、验收意见或历史聊天记录仍使用旧编号：
   - 开发者必须先通过本映射表换算后，再同步更新本地文档、Issue 与 Project

## 8. 当前执行要求

从本表生效后：

1. 新增计划文档正文不再把 `A/B/C/D/E` 或 `C1.4 / D3.2` 当成唯一正式编号
2. 当前有效文档在首次触及时，应优先改写为 `WBS-数字编号`
3. 历史归档、已冻结证据和已存在文件路径不要求批量改名
4. 如需回溯旧编号，应从本表进入，而不是重新在当前文档中恢复旧编号为主口径
