## 2. 为什么 WBS-5 需要先做 WBS-5.1

当前工程已经有较多测试页、回归页和验收清单，但这些入口随着研发推进持续增长。

如果没有发布前的冻结动作，就会出现三类问题：

1. 不知道哪些入口属于正式交付范围
2. 不知道哪些入口只是辅助定位问题
3. 不知道某个版本到底应按哪一套结论来复核

因此 `WBS-5.1` 的职责，是先把“发布候选版必须保留的验证入口”固定下来，为最终交付建立稳定的测试边界。

## 3. 当前切片策略

`WBS-5.1` 当前按 `Prep` 启动，不依赖 `WBS-4 Core` 关闭。

本轮原则：

1. 先冻结入口矩阵，再谈版本结论
2. 先区分正式入口、辅助验证入口和历史归档入口
3. 先让人工验收总入口具备版本结论角色，不再只是杂乱链接集合
4. 每个结论都必须能回链到真实目录、真实页面和真实验收清单

3. 为什么入口冻结后还能继续承接后续迭代，而不是重新散掉
4. 为什么 `WBS-5.1` 的修改不会反向污染 WBS 条目和验收清单的一一对应关系
