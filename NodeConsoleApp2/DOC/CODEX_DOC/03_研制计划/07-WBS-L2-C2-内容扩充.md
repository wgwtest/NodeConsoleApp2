## 2. 为什么 WBS-3 需要单独拆出 WBS-3.2

`WBS-3.2` 的对象不应再被称为泛泛的“内容扩充”。

真正需要收口的是：

1. 关卡节点本身的定义
2. 关卡进入前可见的情节、敌人、奖励与构筑提示
3. 关卡与关卡之间的章节组织和连接关系
4. 关卡编辑器维护这些数据时的对象边界与装载边界
5. 正式关卡编辑器不能被样本页或测试页偷换

如果只说“扩内容”，会同时丢失两件关键事情：

1. 当前主对象其实是 `关卡管理`
2. 当前阶段已经天然承接了 `关卡编辑器` 这条作者工具链

因此 `WBS-3.2` 的职责应明确定义为：

- `把关卡从零散配置，收口成可管理、可组织、可编辑、可验证的正式内容系统`

## 4. 当前切片策略

当前仍按 `Prep + 小步快跑` 推进，但 `Prep` 的对象已经明确为“关卡管理与关卡编辑链”，而不是抽象的“先多补点内容”。

本轮原则：

1. 先把关卡节点层做清楚
2. 再把奖励与章节组织做成玩家可见能力
3. 再把正式关卡编辑器做成独立作者工具
4. 当前版本允许继续采用 `线性章节链` 作为交付基线
5. 但数据与计划语义必须为未来 `网状 / 分叉 / 混合连接` 预留空间
6. 编辑器链继续保持低耦合，不把关卡编辑器做成主流程运行时依赖
7. `level_editor_io_test.html` 与 `level_runtime_probe.html` 只作为历史验证链或后续导出验证链，不再冒充 `WBS-3.2.4` 本体

## 7. 验收口径

本节点不是在验证“关卡数是不是多了一些”，而是在验证“关卡是否已经成为一个可管理、可组织、可编辑、可验证的正式系统”。

必须能回答：

1. 玩家能否在进入前看懂关卡差异和基础状态
2. 玩家能否看懂首通奖励、复通收益和完成状态之间的关系
3. 玩家能否理解当前章节位置与下一步推进方向
4. 开发者能否在正式关卡编辑器中新增、删除和连接关卡
5. 开发者能否导出可供主游戏直接消费的 JSON 文件
6. 编辑器与主流程为什么仍然保持低耦合
