# WBS-3.2.3.1 计划：地图组织方案与效果图验证

创建时间：2026-04-12

最后整理时间：2026-04-12

状态：`当前有效`

## 1. 节点定位

本节点属于：

1. `WBS-3.2 关卡与地图编辑`
2. `#101 / WBS-3.2.3 地图编辑器基础能力与地图包导出链`
3. `#100 / WBS-3.2.3.1 地图组织方案与效果图验证`

它不是正式编辑器本体，而是 `3.2.3` 下的前置验证叶子。

## 2. 本节点为什么现在要做

在进入正式地图编辑器实现前，必须先回答三件事：

1. 地图式关卡组织是否值得做
2. 地图包与主程序的最小交接面应该怎样设计
3. 地图层是否真的能够和主流程保持低耦合

因此本节点的职责是：

- `先把地图组织方案、效果图和 handoff payload 验清楚，再决定是否进入正式地图编辑器实现。`

## 3. 本轮目标

本轮只交付：

1. `地图包样例`
2. `独立效果图页`
3. `主程序最小 handoff payload`

非职责范围：

1. 不把自己包装成正式地图编辑器
2. 不直接产出最终联合交付包

## 4. 明确不做什么

本节点当前不做：

1. 不改 `mock_ui_v11.html` 主流程选关逻辑
2. 不改故事关卡当前线性推进链
3. 不改本地存档结构
4. 不改战斗运行时
5. 不把方案页误包装成正式地图编辑器

## 6. 验收标准

本节点通过标准不是“地图已经接入游戏”，而是以下问题全部能明确回答：

1. 地图式关卡组织是否已形成独立效果图页
2. 页面是否清楚说明“这是方案页，不会改当前主流程”
3. 地图包是否已从运行时关卡配置里独立出来
4. 页面是否明确展示 handoff payload
5. handoff payload 是否已经收敛为主程序最小交接面
6. `3.2.3 地图编辑器` 是否已经有清楚的后续承接方向

## 8. 后续承接关系

如果本节点通过，应按以下顺序承接：

1. 进入 `#101 / WBS-3.2.3 地图编辑器基础能力与地图包导出链`
2. 把 `level_map_pack` 纳入正式地图编辑器维护域
3. 再评估主流程是否需要切换为地图式选关

## 9. 当前结论

`#100 / WBS-3.2.3.1` 的价值在于：

- `把地图组织方案先变成可独立审视、可独立验证的叶子，避免在正式地图编辑器还没定义清楚之前就把多个对象混写在一起。`
