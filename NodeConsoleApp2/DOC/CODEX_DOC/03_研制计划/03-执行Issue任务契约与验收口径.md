# 执行 Issue 任务契约与验收口径

最后整理时间：2026-04-10

状态：`当前有效`

## 1. 目的

这份文档用于明确：

1. GitHub Project 负责什么
2. GitHub Issue 负责什么
3. `DOC/CODEX_DOC` 负责什么
4. 执行 issue 至少要写到什么程度，才算“可直接开工、可被审定”

## 2. 三层职责分工

### 2.1 GitHub Project

Project 只负责人类快速查看的排程和状态，不承载长文档细节。

Project 应维护：

1. `Status`
2. `Layer`
3. `Phase`
4. `Contributes To`
5. `Work Type`
6. `Start Date`
7. `Target Date`
8. `Parent issue`

Project 不负责：

1. 长篇设计说明
2. 详细点击验收脚本
3. 自测日志
4. 会话级交接细节

补充规则：

1. 当某个 WBS 条目已经收到明确的人工验收结论时，Project 状态必须同步更新
2. 用户明确通过后，对应条目应从待验收语义切换到 `Done`
3. 没有用户明确通过前，不得把 Project 状态提前记为 `Done`
4. 当前 Project 中：
   - `Layer` 承担 `WBS 层级投影` 语义
   - `Parent issue` 承担 `Parent Node` 语义
   - issue 标题中的 `WBS-编号` 承担 `WBS Node` 语义
   - issue 标题必须使用纯数字 WBS 编码，不再使用 `Lx + A/B/C` 的混合写法

### 2.2 GitHub Issue

Issue 是单个任务的最小工作契约。

共享策略的最小契约包括：

1. `工作目标`
2. `工作内容`
3. `潜在难点`
4. `Depends On`
5. `成果物`
6. `验收方法`
7. `当前状态`
8. `上级节点`

当前工程默认启用增强版固定字段，因此执行 issue 至少必须包含：

1. `目标`
2. `Owner`
3. `Write Scope`
4. `Depends On`
5. `验收入口`
6. `验收标准`
7. `输出物`
8. `证据链接`
9. `当前状态`
10. `上级节点`

Issue 应该做到：

1. 让人不看聊天记录，也能知道任务要做什么
2. 让人不翻代码，也能先看到验收入口在哪里
3. 让人能判断这是不是 `Prep`，是不是已经可进入开发
4. 父级工作包和叶子任务都要遵守同一模板，不允许只升级叶子节点、放任父节点继续用旧口径

### 2.3 DOC/CODEX_DOC

本地文档负责承载 issue 不适合直接塞进去的长内容。

适合放在文档中的内容：

1. 设计推导
2. 长篇契约说明
3. 详细人工验收步骤
4. 自测报告
5. 会话交接
6. 计划管理归档

原则：

1. Issue 中写“最小可执行契约”
2. 文档中写“详细说明与证据”
3. Issue 必须回链到关键文档
4. 当前本地文档根入口以 [CODEX_DOC README](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/DOC/CODEX_DOC/README.md) 和 [00-本地工程策略映射](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/DOC/CODEX_DOC/00-本地工程策略映射.md) 为准
5. 当前有效计划正文统一以 `WBS-数字编号` 为准；历史编号引用统一回链到 [16-WBS-编号迁移映射表](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/DOC/CODEX_DOC/03_研制计划/16-WBS-编号迁移映射表.md)

补充规则：

1. `Depends On / Write Scope / 输出物 / 证据链接` 中的路径统一使用仓库相对路径
2. 需要可点击打开时，再附 GitHub 仓库 URL
3. 不在 GitHub Issue 正文中放 `/home/...` 绝对路径

## 4. 当前状态语义

Issue 正文中的 `当前状态` 建议只使用下列值：

1. `待开发`
2. `开发中`
3. `已自测`
4. `待人工验收`
5. `已验收`
6. `阻塞中`

解释：

1. Project 里的 `Status` 继续保持粗粒度
2. Issue 正文里的 `当前状态` 负责表达更细的执行与评审状态

## 7. 自测与人工验收分层

从现在开始，必须把 `自测` 与 `人工验收` 分开设计。

### 7.1 自测

自测允许使用：

1. 自动回归页
2. 控制台日志
3. 代码断言
4. 临时测试页

### 7.2 人工验收

人工验收默认只应依赖：

1. 标准页面入口
2. 可重复的点击步骤
3. 验收者可直接观察到的现象

### 7.3 禁止项

以下内容不应再写成“人工验收必做项”：

1. 打开源码确认某个函数存在
2. 阅读实现证明逻辑有效
3. 以回归页 `PASS` 作为主要通过依据

## 8. 基础设施节点的特殊口径

像“统一装载入口”“契约收口”“共享边界”这类节点，仅靠正向流程不足以证明目标成立。

因此后续这类节点的人工验收默认要增加：

1. 标准入口验证
2. 扰动验证

扰动验证的含义是：

1. 先让标准入口正常运行
2. 对关键配置做一个最小、可恢复的改动
3. 观察标准入口是否随之变化
4. 恢复配置后再次验证标准入口恢复正常

详细原则见：

1. [06-人工验收设计原则](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/DOC/CODEX_DOC/04_研发文档/06-人工验收设计原则.md)

## 9. 人工验收清单与 WBS 的对应规则

从现在开始，人工验收清单必须与 WBS 执行节点严格对应。

约束如下：

1. 当前有效清单必须能指向一个明确的 issue / WBS 节点
2. 如果 issue 不存在，不应先写当前有效清单
3. 如果任务必要但 issue 尚未创建，应先补 issue，再写清单
4. 跨多个节点的组合性清单，不应继续作为当前有效入口

对当前工程，默认执行到：

1. L3 节点清单
2. L4 节点清单

## 10. 验收清单目录治理

`DOC/CODEX_DOC/05_测试文档/02_验收清单/` 后续分为：

1. 当前有效区
2. `归档/`

当前有效区只保留：

1. 仍在执行中的 WBS 对应清单
2. 用户下一轮会直接打开执行的清单

`归档/` 用于保留：

1. 旧策略阶段遗留清单
2. 无明确 WBS 对应的过程性清单
3. 已被新清单替代的清单

当前有效清单入口应在：

1. [验收清单 README](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/DOC/CODEX_DOC/05_测试文档/02_验收清单/README.md)

## 11. 验收清单命名格式

当前工程的人工验收清单文件名统一采用：

1. `WBS编号-[状态]-日期-时间-内容-人工验收清单.md`

当前有效区推荐状态值：

1. `已通过`
2. `待验收`

命名目的：

1. 让用户快速识别对应 WBS 节点
2. 让用户快速识别哪些已经验收通过
3. 让同一节点的多轮清单可并存、可追溯

## 12. 人工验收通过后的状态同步

从现在开始，如果某个 WBS 条目已被用户明确人工验收通过，除修改本地验收清单状态外，还应同步完成以下动作：

1. 更新 GitHub Project 对应条目的 `Status`
2. 将其同步到 `Done` 粗粒度状态，同时保持 issue 正文中的 `已验收` 语义
3. 必要时更新本地索引文档中的当前状态说明

这样做的目的，是让：

1. 本地验收清单
2. 执行 issue
3. GitHub Project 看板

保持同一口径。

## 13. 路径同步规则

从现在开始，如果本地文档发生以下变化：

1. 目录迁移
2. 文件改名
3. 文档分层调整

则必须在同一轮同步修正：

1. GitHub Issue 中的路径引用
2. GitHub Project README 中的路径引用
3. 对应本地索引 README 中的入口
