## 2. 为什么它属于阶段 B

原因如下：

1. `A3` 解决的是最小战斗 runtime 恢复闭环
2. `B3` 解决的是单局体验层的存档系统语义
3. `B3` 关心的不只是恢复，还包括：
   - 登录入口语义
   - 自动存档与手动槽位模型
   - 读档页展示
   - 默认保存与手动保存的一致性
   - 统一保存协调入口

因此它更适合放在“单局体验闭环”阶段。

## 3. 工作目标

本节点目标如下：

1. 不再让登录页“开始冒险”隐式默认读档
2. 把 `save_game` 正式收口为“自动存档槽位”的系统语义
3. 把手动槽位与自动存档在读档页明确区分
4. 统一主流程中的保存入口，减少状态分叉
5. 用自动回归与真实页面验证，把关键边界固定下来

## 4. 本轮切入前的问题摘要

本轮切入前，代码的主要问题不是“存档完全不能用”，而是“存档可用但语义混乱”：

1. 登录页点击“开始冒险”会优先读取默认工作副本
2. `save_game` 同时承担默认续玩、默认保存和槽位镜像职责
3. `save_game_last_slot` 只有写入，没有真正消费
4. 技能树提交、战斗结算等默认保存只更新 `save_game`
5. 手动槽位与当前工作副本会自然分叉
