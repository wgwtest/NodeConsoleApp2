# 地图编辑器保存发布工作流实施计划

目标：把地图编辑器改成真正的所见即所得工作台，并明确区分“保存”“发布”“导出”。

## 语义

- 所见即所得：地图设置表单变化后立即更新当前编辑工作区和画布，不再需要“保存当前地图设置”二次应用。
- 保存：把当前地图包写入编辑器作者目录 `assets/map_packs/authoring/<packId>/`，供下次继续编辑。
- 发布：把当前地图包写入主游戏运行目录 `assets/map_packs/current/<packId>/`，并让主游戏默认读取该目录包入口。
- 导出：下载 `package.json`、`maps.json`、`asset-manifest.json`，用于备份或迁移，不等同于保存或发布。

## 实现步骤

1. 测试先行：补 `level_map_editor_page.test.mjs`，验证地图设置实时写入工作区；保存和发布分别调用不同目标目录；导出只下载文件。
2. 编辑器控制器：移除地图设置确认弹窗依赖，给地图设置控件绑定 `change/input`，新增 `saveAuthoringPackage()`、`publishRuntimePackage()` 和本地 API 写入函数。
3. 编辑器页面：顶部按钮改为 `加载工作稿 / 保存工作稿 / 发布到主流程 / 导出包文件`；弹窗改为只解释导出备份。
4. 本地服务端：在 `app.js` 增加受限 POST 接口，只允许写入 `assets/map_packs/authoring/` 与 `assets/map_packs/current/` 下的目录包三文件。
5. 主游戏配置：把 `assets/data/config.json` 的 `levelMapPack` 改为读取 `../map_packs/current/story_pack_v1/package.json`。
6. 数据目录：补齐 `assets/map_packs/authoring/story_pack_v1/` 初始包，与 current 包结构一致。
7. 验证：跑地图编辑器测试、地图包 I/O、运行时加载相关测试和 `npm test`。
