## 3. 可交付目标定义

本项目的第一版交付目标，不是商业完整版，而是：

- `可稳定运行、可完整游玩、可保存与复测的单人回合制战斗 vertical slice`

这一定义包含三层要求：

### 3.1 玩家体验必须完整

玩家应能完成这条闭环：

1. 启动游戏
2. 进入主菜单
3. 选择关卡或读取进度
4. 配置技能或构筑
5. 进行多回合战斗
6. 观察 Buff、生效、敌人行为和结算结果
7. 获得胜负反馈与奖励反馈
8. 保存并继续后续游玩

### 3.2 系统必须稳定

必须满足：

1. 主流程无高频阻断性错误
2. 战斗、结算、成长、读档不会破坏运行态
3. 关键机制可手验，也可自动回归

### 3.3 内容必须达到最低可玩量

第一版至少需要：

1. `3` 个可连续游玩的关卡
2. `3` 类可观察的敌人行为风格
3. `6 ~ 10` 个真实可用技能
4. `5` 个核心 Buff 行为
5. 完整的人工验收清单与发布说明

## 5. 主推进策略

本项目不能按“想到什么做什么”推进，而必须按下面顺序推进：

1. 先做稳定战斗核心
2. 再做完整单局体验
3. 再做成长与内容闭环
4. 最后做交互收口与发布整理

原因只有一个：

1. 目前最稳定的是战斗原型
2. 如果战斗核心不稳，后面的结算、成长、内容、发布都会返工
3. 所以必须让 WBS 顺序直接对应工程依赖顺序

从 2026-03-28 起，阶段 A 不再只做战斗主流程闭环，还应并行启动一条“内容契约与作者工具链”启动线。

## 5.1 双门模型

从 2026-03-28 起，本项目正式采用“双门模型”管理推进与验收：

1. `研发推进门`
   - 用于判断研发是否可以继续向后推进
   - 关闭条件是：实现完成、自测通过、验收入口齐备、交接齐备
2. `阶段关闭门`
   - 用于判断某阶段是否可以正式记为 `已验收 / Closed`
   - 关闭条件是：用户已明确完成人工验收并确认结果

这样做的原因是：

1. 研发速度通常快于人工审核速度
2. 如果把“继续研发”和“阶段关闭”绑成同一扇门，会让后续推进被审核节奏直接卡死
3. 但如果完全绕过人工验收，又会让阶段边界失真，增加返工风险

因此本项目采用：

1. 用户验收不阻断低耦合后续研发准备
2. 用户验收仍然是阶段正式关闭的唯一依据

## 5.2 Prep / Core 规则

为了让后续研发尽量不依赖审核节奏，下一阶段工作统一分成两类：

1. `Prep`
   - 指低耦合、低返工风险、可提前启动的准备工作
   - 典型内容包括：
     - 契约草案
     - 测试脚手架
     - 样本数据
     - 只读分析
     - 独立 UI 壳子
     - 文档、验收基线、回归入口设计
2. `Core`
   - 指直接依赖上一阶段稳定输出的正式实现
   - 典型内容包括：
     - 绑定真实运行时契约
     - 依赖稳定存档结构的功能
     - 依赖稳定 Buff / 敌人行为语义的业务逻辑
     - 正式结算、奖励、成长入账等主链路实现

执行规则：

1. 当上一阶段处于 `待人工验收 / 待用户确认` 时：
   - 允许推进下一阶段 `Prep`
   - 不允许将下一阶段 `Core` 标记为正式开始
2. 当上一阶段已 `已验收` 时：
   - 才允许将下一阶段 `Core` 作为正式主线推进
3. 若 `Prep` 在推进中发现其前提其实依赖上一阶段稳定输出：
   - 应立即回退到 `Core` 分类
   - 停止继续推进，等待上一阶段验收结果

## 6.3 协同推进说明

当前 WBS 既服务单人推进，也服务多人协同。

支持的并行方式：

1. 同一阶段内的不同 L2 / L3 工作包并行推进
2. 下一阶段 `Prep` 与当前阶段 `待人工验收 / Hum Check` 并行推进
3. 文档、测试、数据、运行时、UI 壳子等低耦合切片并行推进

不支持的并行方式：

1. 在上一阶段未稳定前，直接把下一阶段 `Core` 当作正式主线推进
2. 在未明确依赖边界时，多人同时修改同一主链路模块
3. 用 GitHub Project 展示层替代 issue tree 真源

## 6.4 多人协同治理要求

若当前项目进入多人协同或多 agent 并行推进，任何准备并行执行的切片都应至少具备：

1. `Owner`
   - 负责当前切片的人或执行主体
2. `Write Scope`
   - 当前切片允许修改的文件、模块或目录边界
3. `Depends On`
   - 当前切片所依赖的上游 issue、阶段输出或契约

治理要求如下：

1. 未补齐这三个字段时：
   - 可以分析
   - 可以做 `Prep`
   - 不应进入多人并行 `Core`
2. 若两个切片的 `Write Scope` 重叠：
   - 应先拆边界
   - 或改成串行集成
3. 若 `Depends On` 指向仍在变动的上游契约：
   - 应标记为 `Prep`
   - 不应提前实现正式绑定逻辑
