## 2. 为什么 WBS-4 需要先做 WBS-4.1

当前主流程已经可以游玩，但页面仍带有明显研发原型痕迹：

1. 同一动作在不同页面里可能使用不同文案
2. 模态窗的关闭、返回、提交语义还不够统一
3. 主界面信息区的层次更偏向开发视角，不够像交付态界面

如果不先把这些基础 UI 语义收口，后续做新手引导、错误反馈和最终发布说明时，会反复围绕同一个问题返工。

因此 `WBS-4.1` 的职责是先固定主界面的共通语言和结构基线，让后续 `WBS-4.2 / WBS-5.1 / WBS-5.2` 都建立在稳定的可读页面语义之上。

## 3. 当前切片策略

`WBS-4.1` 当前按 `Prep` 启动，不改战斗规则、存档结构和成长结算。

本轮原则：

1. 先统一语言与交互语义
2. 先统一模态规则，不扩新功能入口
3. 先统一状态摘要与信息层级，不改实际计算逻辑
4. 每个叶子都必须能直接对应 `mock_ui_v11.html` 的真实按钮或信息区

## 6. 验收口径

`WBS-4.1` 不是在验证“页面有没有更多功能”，而是在验证“玩家是否能用一致的语义理解现有主流程”。

必须能回答：

1. 为什么同一动作在不同页面上看到的是同一套语言
2. 为什么关闭模态窗不会再让玩家猜测是否已提交
3. 为什么主界面优先展示的是玩家当前决策真正需要的信息
4. 为什么 `WBS-4.1` 的修改不会反向影响战斗、成长和存档逻辑
