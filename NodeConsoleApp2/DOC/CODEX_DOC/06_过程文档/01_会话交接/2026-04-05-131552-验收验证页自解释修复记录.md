# 2026-04-05 13:15 验收验证页自解释修复记录

## 1. 本轮背景

用户指出当前验证页“点几个页面就算通过”过于混乱，核心问题不是功能链断裂，而是页面没有把以下信息直接告诉验收者：

1. 这是个什么工具页
2. 为什么有这些按钮
3. 为什么按这个流程点击，就能证明对应 WBS 能力

## 2. 根因分析

### 2.1 说明存在于工程上下文，不存在于页面自身

1. 之前很多解释只在聊天、验收清单或开发者脑中
2. 页面虽然能用，但打开后缺少“自解释层”
3. 这会让人工验收者无法从页面本身理解验证逻辑

### 2.2 回归页存在真实展示缺陷

1. `codex_regression_runner.html` 新增了 `#usageGuide`
2. 但若测试函数覆盖 `document.body.innerHTML`，最终页面会丢失说明区
3. 所以“源码里有说明”不等于“用户打开页面时能看到说明”

## 3. 本轮改动

1. 给 Buff/关卡 I/O 页补充：
   - `本页用途`
   - `按钮说明`
   - `为什么这能证明`
2. 给 Buff/关卡 runtime probe 页补充：
   - 运行时消费页定位
   - 字段判读说明
   - 按钮与证据链关系
3. 给 `codex_regression_runner.html` 补充：
   - 自动回归页定位
   - `PASS / FAIL` 含义
   - 为什么不能替代人工验收
4. 修复回归页 DOM 污染：
   - 用 `data-regression-fixture` 隔离测试 DOM
   - 每条测试前后清理夹具
5. 放宽说明文案回归断言，避免被 HTML 标签断开误伤
6. 修正 `00-当前待验收功能总入口.md` 的整理时间精度

## 4. 当前状态

1. `codex_regression_runner.html` 浏览器回归结果：`58 / 58 PASS`
2. 5 个验证页的说明区已在最终渲染 DOM 中可见
3. 当前可直接交由用户按页面自带说明重测

## 5. 后续原则

后续再新增任何“验收辅助页 / 局部验证页 / 回归页”时，默认必须同时满足：

1. 页面正文直接说明用途
2. 页面正文直接说明按钮作用
3. 页面正文直接说明“为什么这能证明”
4. 自动回归页只作为开发侧证据，不冒充人工验收结论
