# WBS-3.4.28 浏览器真实低 KP 自然连续推进 Smoke 记录

## 背景

WBS-3.4.27 已经用脚本级真实低 KP 进度式模拟证明：

- `initialSkillPoints = 0` 时可 30/30 通关。
- 第一关没有预装后期技能。
- `skillTreeGapCandidates = 0`。

但该证据仍是模拟器层验证。当前长期目标要求“实际游戏测试”，因此本轮补齐浏览器真实 UI 层的低 KP 连续推进样本，重点验证：

1. 从新档和低 KP 初始技能树进入关卡。
2. 不调用强制结算，依靠真实回合自然胜利。
3. 结算后进入技能树学习。
4. 学到的新技能能在后续真实自然战斗中被部署。

## 改动

更新 `tools/acceptance_click_smoke.mjs`：

1. 新增 `runNaturalProgressiveCampaignSmoke(cdpEndpoint, mainUrl)`。
2. 固定样本关卡：
   - `level_1_1`
   - `level_1_2`
   - `level_1_3`
3. 固定低 KP 初始状态：
   - `initialSkillPointsOverride = 0`
   - 初始已学技能只保留 `skill_heal`、`skill_heavy_swing`、`skill_savage_charge`
4. 通过真实关卡选择 UI 进入关卡，不直接调用内部入口。
5. 每关使用真实规划、提交规划、执行回合链路，不调用 `GameEngine.endBattle(true)`。
6. 胜利结算后按优先级学习：
   - `skill_block`
   - `skill_1771769351059`
7. 记录并断言：
   - 学习前技能树
   - 学习后技能树
   - 结算后技能树
   - 每关自然战斗结果
   - completed/unlocked levels
   - 后续自然战斗是否实际部署 `skill_block`

主报告新增：

```js
naturalProgressiveCampaign: await runNaturalProgressiveCampaignSmoke(cdpEndpoint, urls.main)
```

## 当前判断

本轮仍不补玩家技能树。

理由：

1. 模拟器层低 KP 进度式已经证明 30/30 可通关且 `skillTreeGapCandidates = 0`。
2. 浏览器层低 KP 连续样本已证明早期“结算后学习 -> 后续自然战斗部署新技能”链路成立。
3. 自然检查点矩阵仍保持 6/6 胜利，终局 Boss 是 12/100 HP 低血自然胜利，没有退化成无压力碾压。

剩余风险：

1. 浏览器真实低 KP 连续样本当前覆盖早期 3 关，不是完整 30 关自然连续跑通。
2. 完整 30 关浏览器自然连续推进成本较高，建议后续只在重大平衡改动后再扩展到整章或全战役。
3. 当前结论是“不发现需要补玩家技能树的证据”，不是证明所有玩家构筑都稳定可通关。

## 后续建议

1. 后续平衡改动继续同时看三类证据：
   - 低 KP 进度式模拟。
   - 浏览器低 KP 连续自然样本。
   - 跨章节自然检查点矩阵。
2. 若继续加强验收，应优先扩展第一章完整自然连续浏览器流程，而不是继续增加敌人数量。
3. 若真实玩家测试反馈早期技能选择感不足，再补玩家技能树；当前自动证据暂不支持盲目加技能。
