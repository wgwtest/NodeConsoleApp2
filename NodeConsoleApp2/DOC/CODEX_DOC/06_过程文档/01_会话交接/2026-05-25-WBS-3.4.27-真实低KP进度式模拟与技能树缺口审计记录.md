# WBS-3.4.27 真实低 KP 进度式模拟与技能树缺口审计记录

## 背景

WBS-3.4.26 后，三章 30 关的敌人数量、敌人技能、固定构筑报告和浏览器自然检查点已经有较强证据。但重新审计 `campaign-progressive-report.json` 时发现一个验证口径问题：

- 默认 `assets/data/player.json` 提供 `30 KP`。
- 旧进度式模拟在每关战斗前自动学习。
- 因此第一关前会直接学完大量后期技能，报告更像“满推荐技能构筑”验证，而不是“从初始技能树逐步学习”的验证。

本轮目标是修正自动试玩证据链，证明低 KP 初始技能树是否会产生真实技能树缺口，再决定是否需要补玩家技能、调敌人、调敌技或调奖励。

## TDD 记录

新增 `test/campaign_balance_simulator.test.mjs` 用例：

1. `progressive campaign can start from a real low-KP skill tree without preloading late skills`
   - 调用 `runProgressiveCampaignSimulation({ initialSkillPoints: 0 })`。
   - 要求 `meta.initialSkillPoints = 0`。
   - 要求记录 `skillTreeBeforeLearning`。
   - 要求第一关战斗前只拥有默认初始技能：
     - `skill_heal`
     - `skill_heavy_swing`
     - `skill_savage_charge`
   - 要求第一关不能预装：
     - `skill_regroup`
     - `skill_execute_copy_1770044052832`
   - 要求低 KP 真实进度式能通过章节 Boss 和终局门卫，并保留损耗。

红灯结果：

```text
undefined !== 0
```

原因：旧模拟器没有 `initialSkillPoints` 参数，也没有学习前快照。

随后新增 CLI 用例：

1. `progressive campaign CLI writes a real low-KP report when requested`
   - 执行 `--mode progressive --initial-skill-points 0`。
   - 要求生成 JSON 报告和 Markdown 摘要。

红灯结果：

```text
30 !== 0
```

原因：CLI 未解析 `--initial-skill-points`，仍读取 `player.json` 的默认 `30 KP`。

## 当前判断

本轮不补玩家技能树。

理由：

1. 真实低 KP 进度式从 `0 KP` 开局也能 30/30 通关。
2. `skillTreeGapCandidates = 0`。
3. 第一关没有预装后期技能，逐步学习证据成立。
4. 固定构筑仍有 3 个高压失败点，说明 Boss / 精英压力没有完全消失。
5. 偏科构筑平均 HP 明显低于推荐构筑，说明章节压力仍能惩罚错误构筑。

本轮也不继续扩敌人。

理由：

1. 当前已有 23 个敌人模板。
2. 30 个正式关卡实际使用 22 个敌人。
3. 每章角色覆盖已满足 WBS-3.4.2。
4. 当前主要缺口已从“内容数量”转为“真实进度式验收证据”。

## 后续建议

1. 保留低 KP 真实进度式作为后续平衡主证据之一。
2. 不把 `player.json` 的 `30 KP` 默认演示档直接等同于正式战役起点。
3. 下一轮若继续优化，应优先做“真实浏览器连续流程”：
   - 从新档进入第一章。
   - 结算后学习技能。
   - 进入下一关。
   - 至少连续覆盖第一章，再扩到跨章节检查点。
4. 若只看数值，`level_3_10` 在低 KP 自动策略下 `65/100` 通过，略偏宽松，但固定构筑和浏览器后期压力仍保留低血证据；是否继续压终局 Boss 应以真实浏览器连续流程为准，避免根据单个模拟策略过调。
