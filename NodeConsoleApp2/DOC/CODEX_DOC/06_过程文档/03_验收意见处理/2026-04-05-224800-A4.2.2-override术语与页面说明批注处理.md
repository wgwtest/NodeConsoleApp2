## 2. 用户意见摘要

1. `Apply Runtime Override` 的真实动作解释不清。
2. `本地 override 存储` 这个说法不清楚，不知道它是文件、内存还是浏览器存储。
3. `Buff pack`、`消费`、`另一页独立重载后仍能消费` 这类表述太偏内部术语，不利于人工验收理解。
4. `buff_editor_io_test.html` 顶部 `guide-grid` 横向排列影响阅读，希望改成纵向排列。

## 3. 原因判断

1. 页面说明虽然已经补了“与其他页面关系”，但仍然延续了开发侧术语，默认验收者能理解 `override / pack / consume` 的内部含义。
2. 页面没有直接说清 `runtime override` 在本项目里的真实落点其实是浏览器 `localStorage`，导致“本地”这个词容易被理解成工程文件或多页共享内存。
3. `guide-grid` 采用自适应横向卡片布局，在说明块较长时会削弱顺序阅读体验。

## 4. 本轮处理动作

1. 将 `buff_editor_io_test.html` 中的关键术语改为更直白的说明：
   - `Buff 数据` 直接指向 `buffs_v2_7.json`
   - 明确 `Apply Runtime Override` 不会重写文件
   - 明确“本地 override 存储”实际是浏览器 `localStorage` 里的临时 JSON 数据
   - 明确哪些页面刷新后会读到它，哪些页面不会自动变化
2. 将 `buff_editor_io_test.html` 的 `guide-grid` 固定为纵向布局。
3. 将 `A4.2.2` 清单中的前置条件同步改写为同一口径，避免页面和清单一边清楚、一边继续使用旧术语。
4. 在 `codex_regression_runner.html` 中补充对应回归约束：
   - `guide-grid` 必须保持纵向布局
   - 页面必须包含 `localStorage / 不是本地文件 / 不是几个页面共用的一段内存` 等解释性文本

## 6. 当前回应

1. 这里的 `runtime override` 不是通用专业术语，而是当前项目里“浏览器临时替换默认 Buff 数据的一份本地记录”。
2. 它不是本地文件，也不是多个页面直接共享的一段内存；当前实现是浏览器 `localStorage`。
3. `Apply Runtime Override` 的真实动作是：把当前页内存中的 Buff 数据写进这段浏览器存储，不会改写 `buffs_v2_7.json`。
4. 其他同站点页面只有在刷新或重新打开后，才会重新读取到这份临时 Buff 数据。
