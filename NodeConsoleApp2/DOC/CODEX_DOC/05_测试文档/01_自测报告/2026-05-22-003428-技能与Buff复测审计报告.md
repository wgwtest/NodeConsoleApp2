## 2. 测试方式说明

本轮测试不是浏览器 UI E2E，也不是完整人工游玩。实际分三层：

1. 正式技能执行矩阵：遍历当前 31 个非隐藏正式技能，自动构造合法目标、部位、AP、护甲和必要前置 Buff，逐项调用 `CoreEngine.executePlayerSkill()`；
2. 后端运行时单元/集成测试：直接构造玩家、敌人、技能、Buff，调用 `CoreEngine` 和 `BuffSystem` 的真实运行时方法；
3. 战斗机器人验收测试：构造固定玩家/敌人/敌方技能，使用真实 `startBattle -> startTurn -> commitPlanning -> commitTurn -> TimelineManager -> executePlayerSkill/executeEnemySkill -> TURN_END -> 下一回合` 链路；
4. 默认项目回归：继续跑现有 UI 语义、技能字段、WBS 基线测试。

战斗机器人验收刻意隔离了浏览器 UI、真实存档、真实 GameLoop 副作用；使用假的 EventBus/FSM/DataManager，但核心回合流和 Buff/Skill 结算使用真实源码。

## 6. 战斗机器人验收覆盖

新增文件：`test/skill_buff_battle_robot.test.mjs`

固定脚本：

1. 玩家开局 70/100 HP，敌人开局 100/100 HP；
2. 敌人每回合固定使用 `光头`，保证敌人会真实行动，并触发行动前 Buff 逻辑；
3. 第 1 回合玩家使用 `撕裂伤口`，敌人获得 `buff_tear_wound`，敌人行动前叠 1 层 `buff_bleed`；
4. 第 2 回合玩家使用 `迸发`，按当前 1 层流血造成 5 HP 伤害，敌人行动前继续叠到 2 层；
5. 第 3 回合玩家使用 `吸血`，按当前 2 层流血回复 4 HP；
6. 断言最终 HP、顶层 HP 与 `stats.hp` 同步、Buff 层数、Timeline 结果、历史动作记录、战斗日志、无 `TIMELINE_ERROR`。

机器人测试首次失败暴露了两个真实问题：

1. BuffSystem 直接扣血/回血时只写 `stats.hp` 或只写顶层 `hp`，敌人同时存在两套 HP 字段时会不同步；
2. `BATTLE_ACTION_PRE` 原先派发给行动者和目标，导致目标身上的 `撕裂伤口` 在“别人攻击它”时也叠流血，不符合“持有者自己行动前”的语义。

已做通用修复：

1. BuffSystem 新增通用 HP 读写辅助，Buff 直接伤害/治疗会同步 `hp/maxHp` 与 `stats.hp/stats.maxHp`；
2. `onActionPre` 只派发给当前行动者自身 Buff；被攻击/受击响应仍应走 `BATTLE_ATTACK_PRE / BATTLE_TAKE_DAMAGE_PRE`。

## 8. Skill 引入 Buff 的复杂度分层

本轮引入了一个可审计的复杂度分层，避免把所有“Skill + Buff”都混为一种难度。

| 等级 | 类型 | 当前例子 | 处理策略 |
|---|---|---|---|
| S0 | 纯数值技能 | `重锤`、`治疗` | 直接动作结算 |
| S1 | 只上/下 Buff | `锯齿斩`、`光脚` | Skill 只引用 buffId |
| S2 | Buff 自己改属性 | `减速`、`增伤`、`减伤` | `statModifiers` 数据驱动 |
| S3 | Buff 响应事件 | `流血`、`撕裂伤口`、`护甲免伤` | BuffSystem 事件驱动 |
| S4 | Skill 读取 Buff 层数计算 | `迸发`、`吸血` | `BUFF_STACKS` 窄能力 |
| S5 | Skill 读取并消耗/转化 Buff | 暂无 | 暂不实现 |
| S6 | 任意公式/多 Buff 条件判断 | 暂无 | 暂不实现 |

结论：`迸发 / 吸血` 被控制在 S4，只允许读取一个指定 Buff 的层数并乘倍率，没有开放任意公式。

## 10. 风险判断

当前风险分层：

1. 高风险阻断：无；
2. 数据结构风险：低，当前引用和前置关系已经可审计；
3. 玩法语义风险：低到中，主要剩余点是是否要让 `迸发` 消耗流血层数；
4. 引擎耦合风险：低，本轮新增的是通用 `BUFF_STACKS`，没有新增单个 Buff 或技能 ID 的专属分支。

## 11. 建议下一步

建议下一步只讨论一个设计点：`迸发` 是否应该在结算后消耗或清空目标的 `buff_bleed`。

当前实现选择“不消耗”，理由是它属于 S4；如果改为消耗，会升级到 S5，需要新增 `consumeSource` 或 `buffRefs.remove` 的条件化能力，并增加额外测试。
