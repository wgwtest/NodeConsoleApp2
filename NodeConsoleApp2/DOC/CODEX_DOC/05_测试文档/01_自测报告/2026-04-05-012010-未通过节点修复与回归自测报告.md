## 2. 根因结论

### 2.1 A3 系列真实功能缺陷

1. `CoreEngine._persistRuntimeSnapshot()` 在战斗中会调用 `this.data.saveGame()`
2. 这导致 `startTurn()` 与 `commitPlanning()` 一边刷新内存快照，一边偷偷把 `save_game` 自动落盘
3. 所以用户即使没有点 `保存`，刷新后仍能从 `继续游戏` 恢复战斗
4. `UI_SystemModal` 的存档页只有 UI 外壳，没有真实 `getSaveList()` 支撑，所以槽位描述始终停留在 `空 | 关卡: -`
5. 存档页也没有显式反馈文案，用户无法判断点击 `保存` 是否生效

### 2.2 A4.2 / A4.3 系列主要是验收入口设计错误

1. 多个清单把 HTML 测试页写成了本地文件链接，实际人工点击时容易走 `file:///...`
2. 这些测试页依赖 ESM import 与 fetch，本地文件打开会触发浏览器同源/CORS 限制
3. 同一页面通过 `http://127.0.0.1:3101/...` 打开时可以正常工作
4. 另外，清单没有解释 `I/O 页` 与 `runtime probe 页` 的分工，导致“为什么要测这几个页面”本身不清楚
5. `A4.3.1` 原先要求人工直接翻 `levels.json` 字段，不符合人工验收应以可观察行为为主的原则

## 3. 修复内容

### 3.1 代码修复

1. 去掉战斗快照的隐式落盘逻辑，`_persistRuntimeSnapshot()` 现在只刷新内存 runtime
2. 显式保存仍走 `CoreEngine.saveGame(slotId)`，只有点击 `保存` 才写入 `save_game`
3. `DataManagerV2` 新增真实存档槽读取能力：
   - `getSaveList()`
   - `saveGame(slotId)`
   - `loadGame(slotId)`
4. 显式保存时，同步刷新 `save_game_slot_n` 与 `save_game`
5. `UI_SystemModal` 的存档页增加：
   - 保存成功反馈文案
   - 槽位关卡名与回合信息展示
   - 基于 `isEmpty` 的读取按钮状态

### 3.2 文档修复

1. 重写 `A3.1`，用短对象替代巨量 runtime dump
2. 重写 `A3.3`，显式加入“未保存不能继续，已保存才能继续”的双分支验收
3. 重写 `A4.2.2 / A4.2.3 / A4.3.1 / A4.3.3`，统一改成 `http://127.0.0.1:3101/...` 入口
4. 在 Buff/关卡局部验证清单中补上页面分工说明
5. 把 `A4.3.1` 从“翻 JSON 字段”改成“观察 I/O 页与 runtime probe 页的映射结果”
