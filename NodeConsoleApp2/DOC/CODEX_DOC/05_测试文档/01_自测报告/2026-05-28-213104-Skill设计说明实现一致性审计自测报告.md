## 3. 核心结论

### 3.1 已对齐为当前实现事实

1. `costs.ap` 是当前硬执行成本，`CoreEngine._getSkillApCostStrict()` 严格读取。
2. `target.subject`、`SCOPE_PART/SCOPE_MULTI_PARTS` 和 `bodyPart` 由 `CoreEngine._validateSkillTargetSelection()` 校验。
3. `actions[].effect.effectType` 当前支持 `DMG_HP/DMG_ARMOR/HEAL/ARMOR_ADD/AP_GAIN/BUFF_REMOVE`。
4. `BUFF_STACKS/BUFF_REMAINING` 是当前通用 Buff 读数能力。
5. `buffRefs.apply/remove` 由 `CoreEngine._applySkillBuffRefs()` 消费。

### 3.2 已改为设计债务而非当前事实

1. `requirements` 当前只进入契约摘要、UI 展示和统计，不是硬释放门槛。
2. `costs.partSlot.slotCost` 当前不按数量扣槽。
3. `costs.perTurnLimit` 当前不被 TurnPlanner 直接读取。
4. `placement.maxSlots` 当前不被规划器消费。
5. 多选技能目标态是单 action 多 `selectedParts`，但当前实现会按多个 `placedSlots` 展开为多条 action。
6. `buffRefs.apply[].chance` 当前仍被代码和数据使用，和确定性战斗口径存在设计冲突。

## 5. 未改代码的原因

本轮任务是文档与实现一致性审计。当前发现的关键问题属于“实现能力尚未达到目标态”的设计债务：

- `requirements/partSlot/perTurnLimit` 硬校验缺失。
- 多选技能 action 粒度不符合目标态。
- `chance < 1` 与确定性战斗口径冲突。

这些不应在文档审计中顺手改引擎或技能数据，否则会混淆“文档对齐事实”和“系统行为变更”。后续若进入实现阶段，应单独建任务、补负例测试，再改 Planner/CoreEngine/技能包。
