# 2026-04-07 B3 本地存档与读档系统收口自测报告

## 1. 范围

本轮自测对应以下 WBS 节点：

1. `#67 / B3 本地存档与读档系统收口`
2. `#66 / B3.1 自动存档槽位与入口语义收口`
3. `#68 / B3.2 统一保存协调层与存档回归补强`

本轮目标不是引入完整 manifest，而是先把现有 `save_game` 过渡实现收口为“自动存档槽位”语义，并把分散的保存动作统一到引擎协调层。

## 2. 本轮实现结论

### 2.1 登录页入口语义收口

1. 登录页不再使用 `开始冒险` 作为隐式读档入口。
2. 当前登录页明确拆成：
   - `新游戏`
   - `读取存档`
3. 在无本地存档时：
   - `读取存档` 按钮禁用
   - 页面会直接说明“当前没有可读取的本地存档，请先创建新游戏”

### 2.2 自动存档槽位正式显式化

1. `DataManagerV2.getSaveList()` 现在会把 `save_game` 暴露为正式槽位：
   - `id = auto`
   - `slotType = auto`
   - `title = 自动存档`
2. `存档 / 读档` 页现在明确区分：
   - `自动存档`
   - `手动槽位 1 ~ 3`
3. `自动存档` 只提供 `读取`，不提供 `保存`，避免和手动槽位职责混淆。

### 2.3 主菜单恢复语义收口

1. 战斗中的系统菜单不再显示语义模糊的 `继续游戏`。
2. 当前统一改为 `返回战斗`，明确表示这是“回到当前内存中的战斗态”，不是“读取某个持久化槽位”。

### 2.4 保存协调层统一

当前保存动作已经统一经过 `CoreEngine.requestSave()`，不再让业务逻辑直接散落调用 `data.saveGame()`。

本轮已收口的入口包括：

1. `CoreEngine.login()` 新游戏创建后的自动存档
2. `CoreEngine.resetGame()` 重开后的自动存档
3. `CoreEngine.learnSkill()` 学习提交后的自动存档
4. `CoreEngine.endBattle()` 结算后的自动存档
5. `CoreEngine.saveGame(slotId)` 手动槽位保存

### 2.5 非战斗保存清理边界补强

新增统一的非战斗保存清理逻辑后，非战斗落盘会清理以下 battle runtime 字段：

1. `levelData`
2. `turn`
3. `phase`
4. `initialState`
5. `history`
6. `queues`
7. `planning`
8. `battleRules`
9. `playerBattleState`

这解决了旧实现里只删一部分字段、还遗留 `playerTempState` 命名漂移的问题。
