# 技能与 Buff 渐进式解耦首轮自测报告

创建日期：2026-05-21

状态：`已自测 / 待人工验收`

## 1. 本轮目标

本轮测试的目标是启动技能与 Buff 的渐进式解耦测试，而不是一次性要求所有 Buff、Skill 都通过。

执行边界：

1. 不为单个 Buff 或单个技能向 CoreEngine 增加专属分支；
2. 先覆盖现有通用机制已经能表达的 Buff；
3. 对缺失引用、空运行时效果、命名语义不一致的内容进行记录；
4. 用独立 Node 测试固化首轮可复测入口。

## 2. 新增测试入口

新增文件：

```text
test/skill_buff_decoupled_runtime.test.mjs
```

覆盖层级：

1. L0：技能包与 Buff 包引用分类；
2. L1：BuffRegistry、BuffManager、BuffSystem 原子能力；
3. L3：Skill 只施加 buffRefs，Buff 后续由事件独立生效；
4. L4：CoreEngine 消费通用上下文字段，不识别具体 Buff ID。

## 3. 已覆盖的 Buff 能力

本轮已经验证的通用机制：

1. `buff_bleed / 流血`：`TURN_START -> DAMAGE_HP`；
2. `buff_poison / 中毒`：`TURN_END -> DAMAGE_HP`；
3. `buff_stun / 晕眩`：`TURN_START -> SKIP_TURN`；
4. `buff_slow / 减速`：`speed flat`；
5. `new_buff_1771487521271 / 加速`：`speed flat`；
6. `new_buff_1771481773827 / 护甲免伤`：`onTakeDamagePre -> preventArmorDamage`；
7. `new_buff_1771481936095 / HP免伤`：`onTakeDamagePre -> preventHpDamage`；
8. `new_buff_1771485041778 / 受击获甲`：`onTakeDamagePre -> HEAL_ARMOR`；
9. `new_buff_1771482673293 / 攻击获甲`：`onAttackPre -> HEAL_ARMOR`；
10. `new_buff_1771485482007 / 开销降低`：`TURN_START -> _planningApCostFlatDelta -= value`；
11. `new_buff_1771487055554 / 开销增多`：`TURN_START -> _planningApCostFlatDelta += value`；
12. `buff_ap_regen / 护甲回复`：已纳入同类 `HEAL_ARMOR` 机制覆盖范围。

## 4. 已覆盖的 Skill + Buff 集成样本

本轮用 `锯齿斩` 验证技能与 Buff 的责任边界：

1. `锯齿斩` 自身只执行直接伤害和 `buffRefs.apply`；
2. 敌人获得 `buff_bleed`；
3. 后续 `TURN_START` 由 BuffSystem 独立触发流血扣血；
4. 技能本身不解释流血生命周期、触发器或扣血规则。

## 5. 暂缓和待设计项

本轮记录但不强行修复的项目：

| 项目 | 当前事实 | 缺口类型 | 建议 |
|---|---|---|---|
| `西海岸 -> buff_shield` | 正式技能引用 `buff_shield`，但 `buffs_v2_7.json` 未定义 | 缺 Buff 定义 | 先由设计确认“护盾”是吸收池、护甲转移还是临时护甲 |
| `buff_vulnerable / 增伤` | 无 effects，无 statModifiers | 空运行时效果 | 若要做增伤，建议走通用 `damageTakenMult` 或明确新 action |
| `buff_pain_sup / 减伤` | 无 effects，无 statModifiers | 空运行时效果 | 若要做减伤，建议走通用 `damageTakenMult` 或 `armorMitigationMult` |
| `new_buff_1771487561747 / 减速` | 名称是减速，但无 statModifiers | 空运行时效果 | 可直接复用 `speed flat` 模型 |
| `buff_lifesteal / 吸血` | 当前 action 是 `HEAL_ARMOR`，不是 HP 吸血 | 命名与行为不一致 | 需要确认“吸血”保留为搞笑命名但实际获甲，还是改成 HP 回复 |
| `debuff_weak / 虚弱` | 有 `atk percent_base -0.2` | CoreEngine 伤害计算是否消费 buff 后 atk 需单独验证 | 后续单独测试 `_computeEffectAmount` 是否应改为 `getEffectiveStat('atk')` |

## 6. 解耦结论

本轮没有新增 CoreEngine 专属 Buff ID 或 Skill ID 分支。

当前比较稳定的方向是：

1. BuffSystem 负责把通用事件转换为通用上下文字段或通用 actor 状态；
2. CoreEngine 只消费 `preventHpDamage`、`preventArmorDamage`、`_planningApCostFlatDelta`、`getEffectiveStat('speed')` 这类通用能力；
3. Skill 只负责目标、动作、资源消耗和 `buffRefs`，不内嵌 Buff 生命周期；
4. 对需要“部位历史”“来源追踪”“层数查询/消耗”的机制，先进入待设计清单，避免污染主引擎。

## 8. 后续建议

下一轮建议继续保持渐进式测试：

1. 单独补 `debuff_weak` 的 `atk` 消费验证，判断是否需要通用 stat 接入口；
2. 单独验证 `buff_thorns / 反伤` 和 `buff_counter / 反击`，因为它们进入防守后事件和反击请求；
3. 对 `西海岸` 先出机制定义，不直接实现 `buff_shield`；
4. 对“同部位多回合叠甲更强”这类机制先做设计评审，不进主引擎硬编码。
