## 1. 结论

本轮找到的相对合理解是“标准半重构候选 v2”。

候选包：

`assets/skill_packs/authoring/skills_melee_v4_5_sword_bleed_window_standard_v2_20260529_013420.json`

该方案不是从零重画技能树，而是在 `004020` 的手工布局、坐标、层级和大部分前置基础上，把剑系流血支线收束为：

```text
少量建窗 -> 严格读窗 -> 明确消窗 -> 招架交叉兑现
```

它比上一版 `from_004020_v1` 更合理的地方：

1. 低阶建窗不再过度滚大窗口。
2. 读窗技能必须读取 `buff_bleed.remaining`，且全部有 `maxRead`。
3. 高阶爆发技能必须消耗 `remaining`，不能无限重复吃同一个大窗口。
4. 不依赖新 Buff，不增加冷却系统，不给单个技能 ID 写引擎特判。
5. 保留现有编辑器布局，后续审定和回滚成本低。

## 2. 当前实现边界

本轮读取了当前代码和数据后，确认以下事实：

| 项目 | 当前事实 | 对设计的影响 |
| --- | --- | --- |
| 流血 | `buff_bleed.remaining` 表示持续回合数 | 不能再写“流血层数”口径 |
| 流血伤害 | 回合结束固定 5 点 HP | 建窗收益是延迟收益，不是即时全额收益 |
| 重复施加流血 | `stackStrategy: extend` 增加 remaining | 可以做窗口资源，但要防止滚太大 |
| 单回合技能重复 | 当前规划行为基本是每技能每 turn 一次 | 不需要引入冷却 |
| AP | 当前真正硬约束是 AP 预算 | 不能把 `partSlot` 当成主要平衡依据 |
| `partSlot` | 主要是槽位/UI/规划语义 | 报告不把它作为数值硬约束 |
| 读窗 | 支持 `BUFF_REMAINING + maxRead` | 可做低倍率读窗 |
| 消窗 | 支持 `buffRefs.remove[].consumeRemaining` | 可做高阶兑现 |
| 条件 | 支持 `requirements.targetBuff` 和 `targetHpPercentBelow` | 可做门槛，不需描述假约束 |
| 普通主动命中 | 默认应写作指定部位护甲伤害 | 生命伤害只用于流血 DOT、自身代价、治疗/HP 相关或明确绕甲特例 |

## 3. 三套候选

### 3.1 A：保守继承方案

来源：

`skills_melee_v4_5_sword_bleed_window_from_004020_v1_20260529_003716.json`

特点：

1. 最大限度保留 `004020` 的技能含义和结构。
2. 已经把“流血层数”改成“流血窗口”。
3. 已经使用 `BUFF_REMAINING / maxRead / consumeRemaining`。

主要问题：

1. 低阶建窗仍偏多，浅割、锯齿斩、深切每回合组合后，窗口增长速度偏快。
2. 猩红收割读 8、消 8，兑现感强，但容易把长战斗导向“大窗口爆发”单一解。
3. 它更像“修正旧方案”，还不是从棋子模型重新筛出的方案。

### 3.2 B：标准半重构方案

来源：

`skills_melee_v4_5_sword_bleed_window_standard_v2_20260529_013420.json`

核心技能表：

| 技能 | KP / AP | 角色 | 效果 |
| --- | ---: | --- | --- |
| 剑击 | 1 / 1 | 基础破甲 | 对指定部位造成 10 点护甲伤害 |
| 浅割 | 1 / 1 | 轻建窗 | 对指定部位造成 7 点护甲伤害，增加 1 回合流血 |
| 锯齿斩 | 2 / 1 | 标准建窗 | 对指定部位造成 4 点护甲伤害，增加 2 回合流血 |
| 深切 | 3 / 2 | 重建窗 | 对指定部位造成 12 点护甲伤害，增加 2 回合流血 |
| 撕裂伤口 | 3 / 2 | 延迟建窗 | 造成 5 点护甲伤害，施加撕裂伤口 |
| 失血踉跄 | 3 / 1 | 窗口控场 | 目标至少 3 回合流血时，对指定部位造成 4 点护甲伤害并减速 |
| 血涌斩 | 5 / 2 | 非消耗读窗 | 目标至少 2 回合流血时，对指定部位造成 `6 + 3 * min(W, 4)` 点护甲伤害 |
| 饮血斩 | 5 / 2 | 读窗续航 | 目标至少 2 回合流血时，对指定部位造成 6 点护甲伤害并回复 `2 * min(W, 4)` 点生命 |
| 断脉一剑 | 8 / 3 | 终结消窗 | 目标 HP < 35% 且 W >= 3 时，对指定部位造成 `16 + 8 * min(W, 4)` 点护甲伤害，消耗最多 4 回合流血 |
| 猩红收割 | 8 / 4 | 大型消窗 | W >= 6 时，对指定部位造成 `18 + 5 * min(W, 6)` 点护甲伤害，回复 `2 * min(W, 6)` 点生命，消耗最多 6 回合流血 |
| 反手割裂 | 2 / 1 | 招架建窗 | 对指定部位造成 6 点护甲伤害，增加 1 回合流血，并获得反击姿态 |
| 血誓打击 | 8 / 3 | 流血+招架交叉兑现 | W >= 3 时，自身受到 6 点生命伤害，对指定部位造成 `22 + 4 * min(W, 3)` 点护甲伤害 |

阶段闭包：

| 技能 | 闭包 KP | 意义 |
| --- | ---: | --- |
| 浅割 | 2 | 初期获得轻流血选择 |
| 锯齿斩 | 4 | 初期形成标准建窗 |
| 深切 | 7 | 初期末可投入重建窗 |
| 撕裂伤口 | 7 | 初期末获得延迟建窗 |
| 失血踉跄 | 10 | 初期末到中期入口获得控场 |
| 血涌斩 | 12 | 中期入口获得读窗伤害 |
| 饮血斩 | 12 | 中期入口获得读窗续航 |
| 断脉一剑 | 20 | 中期成型终结技 |
| 猩红收割 | 20 | 中期成型大型消窗 |
| 血誓打击 | 28 | 中期后段形成流血+招架交叉兑现 |

为什么它更合理：

1. 低阶不再是三个“1AP 小伤害 + 流血”的同质技能。
2. `浅割 / 锯齿斩 / 深切 / 撕裂伤口` 分别对应轻建窗、标准建窗、重建窗、延迟建窗。
3. `血涌斩 / 饮血斩` 都是非消耗读窗，倍率低、上限低，不能无限刷大窗口。
4. `断脉一剑 / 猩红收割` 都明确消窗，且有 HP 或 W 门槛。
5. `血誓打击` 变成流血线和招架线的交叉兑现，不再只是普通 3AP 30 伤害。
6. 没有引入冷却时间，没有引入新 Buff，符合当前引擎解耦策略。

### 3.3 C：重构探索方案

模型草案：

```text
剑击
└─ 刻痕
   └─ 开口
      ├─ 压住伤口
      └─ 撕裂伤口
         ├─ 失血踉跄
         ├─ 血线压迫
         │  └─ 放血收束
         │     └─ 断脉一剑
         └─ 饮血斩
```

特点：

1. 更像从“棋子模型”从零搭建。
2. 建窗、维窗、读窗、消窗的角色最清晰。
3. 可以进一步降低旧技能名带来的语义负担。

主要问题：

1. 偏离当前 `004020` 布局较大，需要重排技能编辑器。
2. 会引入一批新名字和新节点，人工审定成本高。
3. 简化模拟中初期偏慢，容易在普通敌人身上显得拖。
4. 当前目标是先找到可落地合理解，不宜直接跳到全重构。

## 4. 简化策略矩阵

本轮没有把脚本写成正式工具，只做了辅助性简化模拟。模拟只用于发现设计风险，不作为最终平衡结论。

代表敌人：

| 模型 | 参数 |
| --- | --- |
| 初期敌人 | 80 HP，5 AP，最长 8 回合 |
| 中期敌人 | 280 HP，5 AP，最长 14 回合 |
| 后期敌人 | 650 HP，6 AP，最长 18 回合 |
| 清流血敌人 | 420 HP，5 AP，每 3 回合清一次流血 |

关键观察：

| 观察 | 结论 |
| --- | --- |
| A 方案窗口峰值偏高 | 说明旧式低阶建窗过密 |
| B 方案读窗上限更低 | 长战斗中不容易被非消耗读窗无限放大 |
| B 方案消窗后需要重新建窗 | 高阶兑现有代价 |
| C 方案结构最清楚，但初期慢 | 从零重构需要额外调初期体验 |
| 清流血敌人下 B 仍有基础破甲、招架、低窗口替代 | 流血路线不会完全失效 |

## 5. 推荐方案的风险

`v2` 仍有需要人工确认的地方：

1. `浅割 7 点护甲伤害 + W1` 是否过于接近基础剑击，需要在实战中看手感。
2. `锯齿斩 4 点护甲伤害 + W2` 的延迟收益是否足够明显。
3. `猩红收割` 当前是 4 AP，兑现感强但一回合投入很重，需要看玩家是否愿意用。
4. `血誓打击` 使用自伤 + 读窗，但没有消耗窗口，可能需要实战后决定是否也消耗 3 回合流血。
5. 时间轴按速度排序，不能保证同一回合先建窗再读窗，所以中高阶读窗技能更适合读取上一回合已经存在的窗口。
