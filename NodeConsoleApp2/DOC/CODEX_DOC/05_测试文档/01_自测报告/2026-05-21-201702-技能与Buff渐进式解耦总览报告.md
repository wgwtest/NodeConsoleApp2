## 1. 总体结论

本轮已经把技能与 Buff 的测试从“零散人工判断”推进到“可重复的渐进式自动测试”。

当前结论：

1. Buff 原子能力已经覆盖一批核心通用机制；
2. Skill 施加 Buff、Buff 后续独立生效的边界已经验证；
3. CoreEngine 已经补通若干通用消费入口，但没有新增具体 Buff ID 或 Skill ID 的硬编码；
4. 正式技能直接动作层已覆盖主要 effectType；
5. 当前正式技能已形成运行时能力分类，后续改设计时可以自动发现分类变化。

当前不应继续追求“所有技能和 Buff 一次性全绿”。更合理的下一步是对少数待决设计点做人工选择，再进入下一轮实现和测试。

## 6. 当前仍待人工判断的问题

### 6.1 已处理：scaling 语义

涉及技能：

1. `剑砍`
2. `重锤`
3. `诸神黄昏`
4. `提桶跑路`

处理结果：

1. 当前生效技能包 `assets/data/skills_melee_v4_5.json` 已删除全部 `effect.scaling`；
2. `meta.enums.amountTypes` 已移除 `SCALING`；
3. 解耦测试已加入约束：正式技能包不得再出现 `scaling` 字段。

### 6.2 已处理：`西海岸 -> buff_shield`

处理结果：

1. 暂不实现“上身护甲转移到腿部”；
2. `西海岸` 已从缺失的 `buff_shield` 改为引用 `new_buff_1771481936095 / HP免伤`；
3. 该处理保持为临时语义，后续如果确认护甲迁移模型，再单独扩展通用机制。

### 6.3 已处理：`愤怒`

处理结果：

1. `愤怒` 已从当前生效技能包删除；
2. `鲜血打击` 前置已改为 `提桶跑路`；
3. 敌人别名 `skill_rage` 已改为指向仍存在的 `鲜血打击`，避免运行时引用空洞。

### 6.4 Buff 数据空实现

当前事实：

1. `buff_vulnerable / 增伤`：描述提到 `damageTakenMult +20%`，但无 effects/statModifiers；
2. `buff_pain_sup / 减伤`：描述提到 `damageTakenMult -30%`，但无 effects/statModifiers；
3. `new_buff_1771487561747 / 减速`：名称是减速，但无运行时效果。

建议：

这些都适合先改数据定义，不需要 CoreEngine 专属逻辑。

### 6.5 已处理：`buff_lifesteal / 吸血`

处理结果：

1. `buff_lifesteal` 已从 `HEAL_ARMOR` 改为 `HEAL_HP`；
2. 回复量使用 `damageDealt` 公式，即按本次攻击造成伤害回复 HP；
3. 解耦测试已覆盖“攻击后回复 HP，且不再给护甲”。

## 7. 解耦原则执行情况

本轮遵守的边界：

1. 没有在 CoreEngine 中写 `if buffId === ...`；
2. 没有在 CoreEngine 中写 `if skillId === ...`；
3. 没有为单个 Buff 增加专属跨回合状态；
4. 没有把技能描述文案当运行时事实；
5. 对缺失或语义不清的内容做了分类记录，而不是强行修复。

CoreEngine 仅补了通用消费能力：

1. 通用伤害倍率字段消费；
2. 通用 explicit selectedParts 解析。

1. `duration` 是持续回合数；写成 `${param}` 时表示由技能施加时传参；
2. `maxStacks` 是最大层数；`stack=refresh` 表示重复施加时刷新持续时间，`replace` 表示替换；
3. `effects` 是事件触发动作，格式近似为 `触发时机:效果:目标:参数`；
4. `statModifiers` 是持续属性修正，格式近似为 `属性:修正方式:数值`；
5. 参数列中的默认值或说明用于判断技能传参是否合理。
