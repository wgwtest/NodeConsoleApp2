# 技能与 Buff 渐进式解耦第三轮自测报告

创建日期：2026-05-21

状态：`已自测 / 待人工验收`

## 1. 本轮目标

本轮进入 L2 Skill 直接动作层，不扩展复杂 Buff 设计，先确认正式技能中已经存在的直接动作能否稳定执行。

覆盖动作类型：

1. `HEAL`；
2. `ARMOR_ADD`；
3. `DMG_ARMOR`：`ABS / PCT_CURRENT / PCT_MAX`；
4. `DMG_HP`：`ABS / PCT_CURRENT / repeat`；
5. `AP_GAIN`；
6. `BUFF_REMOVE`；
7. explicit self target。

## 3. 红灯与根因

新增 L2 测试首跑结果：

```text
tests 15
pass 14
fail 1
```

失败项：`重拳` 没有消耗自身 `arm` 护甲。

根因：

```text
技能数据中 action.target.spec.scope = SCOPE_ENTITY，
但 selection.selectedParts = ["arm"]。
CoreEngine._resolveActionTarget() 旧实现只在 SCOPE_PART / SCOPE_MULTI_PARTS 下读取 selectedParts，
导致 explicit self target 虽然写了 arm，但运行时 bodyPart 为空。
```

这不是 `重拳` 专属问题，而是 explicit action 的通用目标解析缺口。

## 4. 通用修复

修改文件：

```text
script/engine/CoreEngine.js
```

修复内容：

1. `_resolveActionTarget()` 仍按 `scope` 优先处理部位技能；
2. 当 `scope` 不是部位级，但 explicit action 写了 `selection.selectedParts` 时，也会尊重第一个 selected part；
3. 没有写 `skillId` 或 `buffId` 专属分支。

修复后 `重拳` 的自身 `arm` 护甲消耗可以正确执行。

## 5. 数据语义待确认

本轮自动记录了“带 `scaling` 字段但 `amountType` 不是 `SCALING`”的正式技能：

| 技能 | 当前 amountType | 当前运行时事实 |
|---|---|---|
| `剑砍` | `ABS` | 按固定 `amount` 执行，`scaling` 不参与 |
| `提桶跑路` | `PCT_CURRENT` | 按当前护甲百分比执行，`scaling` 不参与 |
| `重锤` | `ABS` | 按固定 `amount` 执行，`scaling` 不参与 |
| `诸神黄昏` | `ABS` | 按固定 `amount` 执行，`scaling` 不参与 |

建议：这些不应在本轮硬改。需要先确认设计语义：

1. 如果希望按 `atk` 缩放，应把 `amountType` 改为 `SCALING`；
2. 如果希望固定值为准，应删除或忽略 `scaling` 字段，避免编辑器误导；
3. 如果希望“固定值 + 缩放值”，需要新增明确的通用数值模型，不能让引擎猜。

## 7. 后续建议

下一轮建议不要继续扩大到全量技能“硬通过”，而是做两个清单：

1. 当前正式技能按运行时能力分类：已稳定、数据语义待确认、需要新通用机制、建议删除或重做；
2. 对 `buff_vulnerable / buff_pain_sup / buff_shield / 吸血` 这几个待设计项分别给出可执行方案，再由人工选择。

这样可以继续保持解耦，不会为了某个口语化技能名或临时效果把复杂状态塞进 CoreEngine。
