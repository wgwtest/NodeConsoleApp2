# 技能与 Buff 渐进式解耦第二轮自测报告

创建日期：2026-05-21

状态：`已自测 / 待人工验收`

## 1. 本轮目标

本轮在首轮原子能力测试基础上，继续推进更靠近战斗管线的 Buff 能力：

1. 验证反伤、反击这类防守后触发；
2. 验证回合护甲回复、攻击后获甲复用 `HEAL_ARMOR`；
3. 验证 `debuff_weak` 的 `atk` 修正是否能进入 `SCALING` 技能伤害；
4. 验证 BuffSystem 写入的 `damageDealtMult` 是否被 CoreEngine 通用消费。

执行边界保持不变：不写 `buffId` 或 `skillId` 专属分支。

## 2. 红灯与根因

本轮先在 `test/skill_buff_decoupled_runtime.test.mjs` 增加 4 个测试。

首跑结果：

```text
tests 11
pass 9
fail 2
```

失败 1：`debuff_weak` 的 `atk percent_base -0.2` 没有影响 `SCALING` 技能伤害。

根因：

```text
CoreEngine._computeEffectAmount() 读取 actor.stats[statKey] 原始值，没有走 actor.buffs.getEffectiveStat(statKey, base)。
```

失败 2：`damageDealtMult` 被 BuffSystem 写进 context 后没有影响实际伤害。

根因：

```text
BuffSystem._applyStatModifiersToContext() 会写 context.damageDealtMult，
但 CoreEngine._applyBattleDamage() 只消费 damageTakenMult / shieldPool / preventHpDamage，
未消费 damageDealtMult。
```

## 3. 通用修复

修改文件：

```text
script/engine/CoreEngine.js
```

修复内容：

1. 新增 `_getEffectiveActorStat(actor, statKey, fallback)`；
2. `SCALING` 伤害计算改为通过 `_getEffectiveActorStat()` 获取属性；
3. `_applyBattleDamage()` 在 `BATTLE_TAKE_DAMAGE_PRE` 后消费 `context.damageDealtMult`；
4. 修复没有引入具体 Buff ID、具体技能 ID 或专属状态位。

这使以下能力进入通用路径：

1. `debuff_weak / 虚弱`：`atk -20%` 可影响按 `atk` 缩放的技能；
2. 后续任何 `damageDealtMult` 增伤/降伤 Buff 都可以通过同一上下文字段影响伤害。

## 4. 本轮新增覆盖

新增通过项：

1. `buff_thorns / 反伤`：`BATTLE_DEFEND_POST -> DAMAGE_HP(target=attacker)`；
2. `buff_counter / 反击`：`BATTLE_DEFEND_POST -> BUFF_ATTACK_REQUEST`；
3. `buff_ap_regen / 护甲回复`：`TURN_START -> HEAL_ARMOR`；
4. `buff_lifesteal / 吸血`：当前运行时事实是 `BATTLE_ATTACK_POST -> HEAL_ARMOR`；
5. `debuff_weak / 虚弱`：`atk` 通用属性修正进入 `SCALING` 技能伤害；
6. `damageDealtMult`：通用攻击方伤害倍率字段进入 CoreEngine 伤害结算。

## 5. 仍然暂缓的设计项

本轮未改数据包，以下项仍需设计确认：

| 项目 | 当前状态 | 暂缓原因 |
|---|---|---|
| `西海岸 -> buff_shield` | 技能引用缺失 Buff | 需要先定义“护盾”的运行时语义 |
| `buff_vulnerable / 增伤` | 描述写 damageTakenMult +20%，但数据无 effects/statModifiers | 需要决定补 `damageTakenMult` statModifier，还是改成 effect |
| `buff_pain_sup / 减伤` | 描述写 damageTakenMult -30%，但数据无 effects/statModifiers | 同上 |
| `new_buff_1771487561747 / 减速` | 名称是减速，但无运行时效果 | 可复用 speed flat，但需确认数值 |
| `buff_lifesteal / 吸血` | 实际是获甲，不是 HP 吸血 | 需确认是否保留“吸血”这个口语名但作为获甲效果 |

## 7. 后续建议

下一轮建议进入两个方向之一：

1. L2 Skill 直接动作层：批量验证正式技能的 `DMG_HP / DMG_ARMOR / ARMOR_ADD / HEAL / AP_GAIN / BUFF_REMOVE`；
2. 待设计项收口：先为 `buff_vulnerable`、`buff_pain_sup`、`buff_shield` 定义可执行语义，再写测试。

建议优先做 L2 Skill 直接动作层，因为这能先确认现有技能包里“已经能执行”的技能覆盖面，再把真正需要设计判断的机制单独暴露出来。
