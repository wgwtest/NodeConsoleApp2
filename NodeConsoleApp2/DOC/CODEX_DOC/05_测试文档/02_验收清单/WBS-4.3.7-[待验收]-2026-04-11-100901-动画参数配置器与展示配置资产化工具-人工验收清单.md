1. 是否已经存在独立的动画参数配置器。
2. 配置器是否明确说明“这里只改展示参数，不改战斗规则”。

4. 配置器写入的展示配置，是否能被 probe 页和主流程页共同读取。

6. 受击 / 防御强调是否已经去掉整屏位移晃动。

## 3. 为什么这些步骤能证明 WBS-4.3.7

1. `WBS-4.3.7` 验的不是“动画好不好看”，而是“动画参数是否已经成为正式配置资产和工具，而且不会拖累主流程”。
2. 因此需要同时看：
   - 配置器页是否能说明用途、编辑参数、写入 workspace；
   - 配置器页是否让你在同一工作区里改参数并直接看效果；
   - probe 页是否能读取同一份演出配置资产；
   - 主流程页是否也能读取同一份演出配置资产；
   - 展示层是否不再用整屏位移晃动制造冲击感。

## 4. 人工验收步骤

### 4.1 配置器用途与同屏工作区检查

步骤：

1. 打开 `battle_presentation_configurator.html`

观察点：

1. 页面明确写出 `动画参数配置器`
2. 页面明确写出：
   - `只改展示参数`
   - `不会改伤害、Buff 结果或敌人行为`
   - `写入浏览器 workspace`
   - `导出 JSON`
3. 页面明确说明它与 `battle_presentation_probe.html`、`mock_ui_v11.html` 的关系。
4. 同一屏内能同时看到：
   - 左侧 `资产与工作区 / 参数编辑`
   - 右侧 `实时预览`

### 4.2 参数编辑与预览检查

步骤：

1. 在配置器页选择一个 profile
2. 修改至少一个参数，例如：
   - `前冲距离`
   - `受击时长`
   - `场景强调时长`
3. 点击 `应用到预览`
4. 点击预览按钮，例如：
   - `攻击前冲`
   - `受击`
   - `治疗`
   - `Buff 提示`

观察点：

1. 改动参数后，右侧实时预览仍可正常动作。
2. 改动的是展示反馈，不是数值文本或规则逻辑。
3. 页面顶部读数会显示当前 profile 与资产来源。
4. 预览按钮里不再提供 `镜头冲击` 这类整屏晃动入口。
5. 当触发受击 / 防御时，允许有亮度强调，但不应出现整屏左右位移晃动。

### 4.3 workspace 写入与 probe 页读取检查

步骤：

1. 在配置器页点击 `写入浏览器 workspace`
2. 打开 `battle_presentation_probe.html`

观察点：

1. probe 页出现 `当前演出配置资产`
2. probe 页提供 `重新读取演出配置`
3. probe 页能明确说明自己和配置器页的关系
4. probe 页读取到的配置来源与配置器页刚写入的 workspace 口径一致

### 4.4 主流程页消费检查

步骤：

1. 打开 `mock_ui_v11.html`
2. 进入主流程：
   - 点击 `开始冒险`
   - 点击 `关卡选择`
   - 选择唯一关卡
   - 提交规划
   - 执行回合

观察点：

1. 战斗场景区域显示 `演出配置资产`
2. 主流程仍可按既有点击链继续，不会因动画配置而卡住
3. 这里看到的是展示配置读数，不是新的规则面板
4. 执行过程允许出现演出强调，但不应出现整屏位移晃动

## 5. 通过标准

以下全部满足才算通过：

1. 已存在独立的动画参数配置器。
2. 配置器明确说明自己只负责展示参数。
3. 配置器已形成“参数编辑与实时预览同屏”的工作区。
4. 配置器写入的展示配置能被 probe 页和主流程页共同读取。
5. 主流程未因本节点改动而增加新的规则依赖或阻断步骤。
6. 受击 / 防御强调已去掉整屏位移晃动。
