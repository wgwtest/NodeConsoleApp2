# C1.2 学习提交后主菜单成长摘要即时刷新人工验收清单

对应 WBS：`#64 / C1.2 学习提交后主菜单成长摘要即时刷新`

## 1. 本清单在验什么

本清单只验证 `C1.2` 这一个节点：

1. 从主菜单打开技能树并学习技能后，不需要重新打开系统菜单。
2. 返回主菜单时，`知识点 KP` 会立即刷新。
3. 返回主菜单时，`已学技能` 数量会立即刷新。
4. 主菜单原有 `关卡选择` 入口不被破坏。

本清单不验证：

1. 结算页是否出现 `前往技能树 / 构筑`
2. 技能树学习规则本身是否正确
3. 新学技能在下一局中的构筑差异

这些分别属于 `#65 / C1.3` 和 `#29 / C1` 的后续节点范围。

## 2. 验收入口

1. 主入口：`http://127.0.0.1:3101/mock_ui_v11.html`
2. 自动回归辅助入口：`http://127.0.0.1:3101/test/codex_regression_runner.html`

## 3. 前置条件

1. 统一通过 `http://127.0.0.1:3101/...` 打开页面。
2. 建议刷新页面后再开始，避免上一轮旧运行态干扰观察。
3. 如果当前档已经学过很多技能，不要求一定看到 `30 -> 29 / 3 -> 4`，也可以记录“提交前后的相对变化”。

## 4. 为什么这些步骤能证明 C1.2

1. `C1.2` 的目标不是“技能树能学技能”，而是“学习提交后的结果会不会立刻回流主菜单”。
2. 所以本清单必须同时看三件事：
   - 主菜单提交前的成长摘要
   - 技能树提交后的同一层主菜单摘要
   - 主菜单原有入口有没有被刷新过程打坏
3. 如果必须重新打开系统菜单才能看到变化，`C1.2` 就不能算通过。

## 5. 人工验收步骤

### 5.1 记录提交前摘要

目标：

1. 先记录主菜单学习前的成长状态

步骤：

1. 打开 `mock_ui_v11.html`
2. 点击 `开始冒险`

观察点：

1. 页面标题为 `游戏菜单`
2. 主菜单正文出现 `成长摘要`
3. 记下：
   - `知识点 KP` 当前值
   - `已学技能` 当前值

为什么这能证明本节点：

1. 后续是否即时刷新，必须和学习前的同一层菜单摘要对比

### 5.2 学习并提交

目标：

1. 验证学习动作确实发生，并从技能树返回主菜单

步骤：

1. 在主菜单点击 `技能树 / 构筑`
2. 选择一个当前状态为可学习的技能节点
3. 点击详情区中的 `学习`
4. 点击顶部 `提交并关闭`

观察点：

1. 技能树弹层正常打开
2. `学习` 按钮可用
3. `提交并关闭` 后技能树弹层关闭

为什么这能证明本节点：

1. 只有真实发生学习提交，才会触发 `PLAYER_SKILLS` 更新回流主菜单

### 5.3 不重开菜单，直接看摘要

目标：

1. 验证主菜单在同一次返回中已经立即刷新

步骤：

1. 不要重新打开系统菜单
2. 直接观察技能树关闭后的主菜单摘要

观察点：

1. `知识点 KP` 相比学习前发生变化
2. `已学技能` 数量相比学习前增加
3. 主菜单仍能看到 `关卡选择`

为什么这能证明本节点：

1. 这正是 `C1.2` 的核心目标
2. 如果你没有重新进菜单，却已经看到新摘要，说明刷新链路成立

## 6. 通过标准

以下全部满足才算 `C1.2` 通过：

1. `开始冒险` 后进入 `游戏菜单`
2. 主菜单在学习前能看到 `成长摘要`
3. 从主菜单进入技能树后可完成一次真实学习提交
4. `提交并关闭` 后，不重新打开系统菜单，主菜单摘要已变化
5. `关卡选择` 入口仍可见
