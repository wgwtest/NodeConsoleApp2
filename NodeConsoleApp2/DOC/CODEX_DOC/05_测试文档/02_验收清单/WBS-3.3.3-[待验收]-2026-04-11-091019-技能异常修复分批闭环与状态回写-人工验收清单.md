## 3. 为什么这些步骤能证明 WBS-3.3.3

1. `WBS-3.3.3` 验的不是“所有技能问题都修完”，而是“当前节点是否已经建立正式的状态总表和分批闭环入口”。
2. 因此需要同时看：
   - 技能状态总表能否一眼回答当前技能状态；
   - `WBS-3.3.3` 面板是否明确写出已关闭批次和后续批次；
   - 共享回归页是否能只看当前范围；
   - 文档是否已经把当前批次状态回写清楚。

## 4. 人工验收步骤

### 4.1 技能状态总表检查

步骤：

1. 打开 `skill_contract_probe.html`
2. 找到 `技能状态总表`

观察点：

1. 页面会先给出一张总表，而不是先让你看 issue code。
2. 总表至少包含以下列：
   - `名称`
   - `状态（正常，已修复，未修复）`
   - `修复时间`
   - `问题原因`
   - `修改方法`
3. 总表里至少能看到：
   - `skill_block / 修理护甲` 标记为 `已修复`
   - `skill_block` 的修复时间为 `2026-04-11 09:10:19 +0800`
   - 至少一条 `正常`
   - 至少一条 `未修复`
4. 对 `已修复` 技能，如果仍有后续问题，原因列会写清“仍有后续批次问题”，而不是直接隐藏。

### 4.2 异常分批闭环检查

步骤：

1. 在同一页继续查看 `WBS-3.3.3 异常分批闭环` 面板

观察点：

1. 页面明确写出：
   - `当前节点负责的批次`
   - `已关闭批次`
   - `后续批次`
   - `非本节点说明`
2. `首批结构选择异常` 显示为 `已关闭批次`
3. `首批结构选择异常` 明确显示关闭时间：
   - `2026-04-11 09:10:19 +0800`
4. `首批结构选择异常` 会显示 `已修复技能样本`
5. `主体 / 范围标签一致性` 与 `动作效果标签一致性` 仍显示为 `后续批次`

### 4.3 共享回归范围检查

步骤：

1. 打开 `codex_regression_runner.html?scope=WBS-3.3.3`

观察点：

1. 页面会明确解释自己是 `共享回归页`
2. 页面会明确解释：
   - `当前节点范围`
   - `scope=`
   - `非本节点失败项`
   - `只运行当前范围`
3. 当前范围下，至少以下断言为 `PASS`
   - `WBS-3.3.3 会把技能异常汇总到正式修复批次`
   - `WBS-3.3.3 会生成面向人工验收的技能状态总表`
   - `skill_contract_probe 会把技能状态总表作为主阅读入口`

### 4.4 文档状态回写检查

步骤：

1. 打开 `08-WBS-3.3.3-技能异常修复分批与状态回写.md`

观察点：

1. 文档明确写出：
   - `batch_structure_selection`
   - `batch_target_tags`
   - `batch_effect_tags`
2. 文档明确写出首批结构批次已关闭。
3. 文档明确写出后续两个批次仍有待修问题。
4. 文档不会把 `skill tags` 移除分析混进本节点，而是保持和独立 issue 解耦。
