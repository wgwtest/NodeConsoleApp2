## 1. 本清单是验证什么的

本清单不是验证 `Spine` 动画已经接进战斗页。

它只验证：

1. 主工程是否已经把与 `SpineAssets` 的协作规则写清
2. 主工程是否已经把“消费什么 / 不消费什么”写清
3. 主工程是否已经把 bundle 缺失、schema 不兼容、单角色 manifest 缺失时的降级规则写清
4. 双仓库协作入口是否已经在 GitHub 上留下稳定节点

如果这项通过，说明后续就算换一个 CLI 或多人并行推进，也不会先卡在“主工程到底该依赖什么”这个问题上。

## 3. 进入前说明

为了避免误解，先明确两点：

1. `D3.2` 是表现层对外协作基线，不是表现效果增强
2. 本页验证通过，并不意味着主工程已经开始真实加载 bundle，只意味着后续加载工作已经有清晰边界

## 4. 人工验收步骤

### 4.1 主工程职责与非职责是否清晰

操作：

1. 打开主工程协作文档
2. 阅读 `2. 主工程职责`
3. 阅读 `3. 主工程消费什么`

检查重点：

1. 主工程负责战斗规则、事件语义、导入适配层和降级策略
2. 主工程不负责 `Spine Editor` 源工程、Rig 和动画制作
3. 主工程只允许消费正式导出产物：
   - `bundle_manifest.json`
   - `character_manifest.json`
   - `*.skel|json`
   - `*.atlas`
   - `*.png`

为什么这能证明：

1. 如果主工程到底能依赖什么都没写清，后续导入适配层就无法低耦合实现

### 4.2 不可见边界和降级规则是否足够明确

操作：

1. 继续阅读主工程协作文档中的 `4. 运行时边界`
2. 阅读 `5. 降级规则`

检查重点：

1. 主工程不应感知 Rig 内部骨骼结构和素材制作临时命名
2. 当 bundle 缺失时，继续使用现有静态展示层
3. 当 schema 不兼容时，拒绝加载该 bundle，并记录错误
4. 当单角色 manifest 缺失时，只回退该角色，不中断整场战斗主流程
5. 文档明确写出：即使 `Spine` 完全不可用，主流程仍必须成立

为什么这能证明：

1. `D3.2` 的本质要求就是“动画系统未来失败时，也不能反向污染主流程”

【】

### 4.3 双仓库变更规则是否与对口仓库一致

操作：

1. 对照打开素材工程两份设计说明
2. 回看主工程文档中的 `6. 双仓库变更规则`

检查重点：

1. 非 breaking change 与 breaking change 的区分是一致的
2. breaking change 都要求双仓库分别建 issue
3. 主工程文档明确承担降级或迁移策略责任

为什么这能证明：

1. 如果主工程和素材工程对 breaking change 的理解不一致，后续独立仓库协同就一定会漂移

【】

### 4.4 GitHub 协作入口是否已经落地

操作：

1. 打开主工程 issue `#72`
2. 打开素材工程 issue `#4 / #5 / #6`
3. 如有需要，打开两个仓库的 Project

检查重点：

1. 主工程和素材工程都存在对应 issue
2. issue 主题与当前文档描述一致
3. 两边 Project 中都有相应 item，便于后续继续排程和状态同步

为什么这能证明：

1. `D3.2` 解决的不只是本地文档，还要把“协作入口”真正放到 GitHub 协作面上
