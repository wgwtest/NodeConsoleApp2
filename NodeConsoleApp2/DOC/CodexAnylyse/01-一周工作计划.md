# NodeConsoleApp2 一周工作计划

## 1. 计划目标

本周目标不是“把所有设计文档一次性做完”，而是完成一个可以验证的主流程闭环：

- 玩家技能按新 schema 在战斗主流程内真实结算
- Buff 能通过技能施加并在回合/攻击事件中生效
- 敌人能基于规则选择技能、目标和部位
- 时间轴执行、胜负判断、日志输出、测试页面能够完整演示战斗

交付标准建议定义为：

- 测试关卡 `level_1_1` 可以稳定打完
- 至少 6 类主要技能效果可执行
- 至少 4 类 Buff 触发链路可验证
- 至少 3 种敌人行为模式可跑通

## 2. 可行性判断

一周内可行，但必须按“先闭环、再扩展”的顺序推进。

如果本周试图同时做：

- 全量 Buff 枚举支持
- 全量敌人内容制作
- 编辑器大改
- UI 全部重做

那么计划会失控。

所以本计划默认采用以下范围控制：

- 先保证当前 `skills_melee_v4_5.json` 中实际出现的 effectType
- 先保证当前 `buffs_v2_7.json` 中核心 Buff 可以生效
- 先补齐当前敌人配置真实需要的技能，而不是做完整怪物库

## 3. 一周排期

### Day 1：运行时协议梳理与数据适配

目标：

- 消除技能/Buff 数据与引擎运行时之间的第一层协议断裂

任务：

- 梳理并固化 skill runtime view
  - 从 `actions[].effect` 解析出运行时可执行结构
- 修改 `CoreEngine` 的技能读取入口
  - 不再依赖旧字段 `type/value/targetType`
- 修正 BuffRegistry 的 definitions 来源
  - 使用 `gameConfig.buffs.buffs`
- 给 DataManager 增加轻量规范化层
  - 技能、Buff、敌人数据加载后进入统一 runtime shape

当日交付物：

- 技能与 Buff 数据进入运行时后结构一致
- 不再依赖旧字段结算

验收点：

- 控制台可打印规范化后的 skill/buff runtime 数据
- 旧字段缺失不再导致主流程直接失效

### Day 2：玩家技能执行器落地

目标：

- 让玩家技能在时间轴执行阶段真正按新 schema 生效

任务：

- 实现 skill action executor
  - 支持 `DMG_HP`
  - 支持 `DMG_ARMOR`
  - 支持 `HEAL`
  - 支持 `ARMOR_ADD`
  - 支持 `AP_GAIN`
  - 支持 `BUFF_REMOVE`
- 接入 `buffRefs.apply/remove`
- 统一 body part 命中与护甲扣减逻辑
- 统一 action result 输出结构，方便日志和测试

当日交付物：

- 玩家技能不再只是“能排队”，而是能正确执行

验收点：

- `skill_heal`
- `skill_heavy_swing`
- 至少一个破甲技能
- 至少一个加甲技能
- 至少一个带 buffRefs 的技能

以上都能在测试页验证

### Day 3：Buff 主流程接入

目标：

- 让 Buff 真正进入战斗事件链

任务：

- 统一 Buff action 名称映射
  - 兼容 `DAMAGE_HP`
  - `SKIP_TURN`
  - `ATTACK`
  - `PREVENT_DAMAGE_HP`
  - `PREVENT_DAMAGE_ARMOR`
  - `AP_COST_ADD`
  - `AP_COST_REDUCE`
- 统一 `effect.payload` 与运行时读取逻辑
- 修正 `statModifiers` 数组结构消费
- 在战斗流程中验证以下时机：
  - `TURN_START`
  - `TURN_END`
  - `BATTLE_ACTION_PRE`
  - `BATTLE_ATTACK_PRE`
  - `BATTLE_TAKE_DAMAGE_PRE`
  - `BATTLE_ATTACK_POST`

当日交付物：

- 关键 Buff 具备真实战斗影响

验收点：

- `buff_bleed` 可持续掉血
- `buff_stun` 可阻止行动
- `buff_slow` 或等价 stat buff 能影响速度/排序
- 至少一个减伤或护盾类 Buff 可改变最终伤害

### Day 4：敌人技能数据与行为规划器

目标：

- 让敌人从“占位动作”升级为“可解释行为”

任务：

- 补齐敌人技能数据
  - 对 `enemies.json` 当前引用的技能补定义，或改为映射到已存在技能
- 抽出 `EnemyActionPlanner` 或等价模块
- 实现最小行为策略：
  - Aggressive：优先输出伤害
  - Defensive：低血量时优先加甲/治疗/保命
  - Tactical：根据目标部位护甲和 Buff 状态选目标
- 实现目标部位选择策略：
  - 优先无甲部位
  - 优先弱点部位
  - 特定技能可固定部位

当日交付物：

- 敌人动作不再固定取 `skills[0]`

验收点：

- 至少 2 种敌人模板可表现出不同决策
- 敌人会根据局势选技能，而不是固定脚本

### Day 5：主流程联调与存档修正

目标：

- 把玩家、Buff、敌人行为三条链路真正合并成一个稳定战斗

任务：

- 联调 Timeline 与实际执行结果
- 检查回合切换后的 Buff tick 与持续时间
- 修复战斗中断、胜负结算、存档恢复问题
- 检查 `saveGame/loadGame/resumeBattle` 对 Buff 与队列的保存恢复
- 统一日志输出，增强测试可读性

当日交付物：

- 一个从进入关卡到战斗结束都稳定的战斗主流程

验收点：

- 战斗中不会因为某类技能/Buff 直接中断主流程
- 战斗结束后状态清理正确
- 存档恢复不会让战斗数据结构损坏

### Day 6：测试补齐与缺陷收敛

目标：

- 从“能跑”提升到“可验证”

任务：

- 为关键战斗链路补测试页用例或脚本化测试
- 按功能维度整理回归清单：
  - 单体伤害
  - 单体治疗
  - 护甲削减
  - Buff 施加
  - Buff 移除
  - 敌人控制
  - 敌人选择行为
- 修复联调阶段发现的高频问题

当日交付物：

- 一组明确的回归验证路径

验收点：

- 核心场景可重复验证
- 至少能快速复测主要技能/Buff/敌人逻辑

### Day 7：收尾、文档与演示整理

目标：

- 形成可交付版本与后续迭代边界

任务：

- 整理最终实现说明
- 更新 `DOC` 下阶段总结文档
- 标注本周已支持与未支持的 effect/buff/action 范围
- 给下一阶段留下明确 backlog

当日交付物：

- 可演示版本
- 实现说明
- 下一阶段待办清单

验收点：

- 外部阅读者能快速知道“已经支持什么、还没支持什么”

## 4. 本周建议优先级

优先级必须固定，否则会发散。

P0：

- 技能执行器适配新 schema
- BuffRegistry 与 BuffSystem 协议统一
- 敌人技能数据补齐

P1：

- 敌人行为规划器
- Buff 主流程联调
- Timeline 执行稳定性

P2：

- 存档恢复增强
- 测试页增强
- 文档补齐

P3：

- 编辑器体验优化
- 更复杂 AI
- 更复杂 Buff 枚举

## 5. 风险与应对

### 风险 1：新旧 schema 混用导致反复返工

应对：

- 第一天下午之前必须确定唯一 runtime shape
- 后续所有执行器只消费 runtime shape，不直接消费原始 JSON

### 风险 2：Buff 范围失控

应对：

- 只先支持当前数据真实用到的 action
- 不做未来枚举的超前实现

### 风险 3：敌人 AI 做成大工程

应对：

- 本周只做规则驱动决策，不做复杂行为树
- 先把“可解释且稳定”放在“复杂”前面

### 风险 4：测试不足导致最后两天集中爆雷

应对：

- 从 Day 2 起每落一个 effectType 就在测试页验证
- 不把测试全部压到 Day 6

## 6. 我对这个计划的可行性判断

这个计划是可行的，原因有三点：

1. 当前工程主骨架已经存在，不需要从零搭战斗循环
2. 当前技能数据虽然量不少，但真实 effectType 集中，适合先做 MVP
3. 当前 Buff 与敌人系统的问题主要是“协议没对齐”，不是“完全没有系统”

但这个计划成立的前提也很明确：

- 本周交付以“战斗主流程闭环”优先
- 不把范围扩展到完整内容生产和全编辑器重构

如果按这个边界执行，一周内做出一个可信、可玩的版本是现实的。
