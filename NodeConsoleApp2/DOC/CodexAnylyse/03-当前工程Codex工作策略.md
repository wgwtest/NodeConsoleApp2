## 3. 当前工程的执行规则

### 3.1 开始前

- 先读最近一次会话交接文档
- 先读当前阶段计划、验收清单、验收记录和自测报告
- 先检查工作区状态
- 先明确本轮目标、非目标和验证方式

### 3.2 开发中

- 优先修改运行时、契约和事件链路
- 页面与测试页只做必要兼容，不做无关大改
- 每轮只推进一个明确切片
- 不扩散到与当前目标无关的模块

### 3.3 开发后

- 至少执行静态检查和最小真实链路验证
- 有实际改动时默认新增验收清单
- 有真实测试时默认新增验收记录或自测报告
- 默认新增会话交接文档
- 在用户未明确反馈验收结果前，阶段状态一律记为：
  - `待人工验收`
  - 或 `待用户确认`
- `待人工验收 / 待用户确认` 不再自动阻断后续低耦合研发准备
- 但在用户未确认前，不得把当前阶段记为 `Accepted / Closed`

### 3.4 双门模型执行规则

本工程默认采用“两扇门”而不是“一扇门”：

1. `研发推进门`
   - 条件：
     - 本轮实现完成
     - 已完成最小自测
     - 已形成验收清单
     - 已形成自测报告或验收记录
     - 已形成会话交接
     - 已明确告知用户验收入口
   - 作用：
     - 允许继续推进后续低耦合研发任务
2. `阶段关闭门`
   - 条件：
     - 用户已明确回复验收结果
     - 用户已确认当前阶段可接受
   - 作用：
     - 允许把当前阶段记为 `Accepted / Closed`

### 3.5 Prep / Core 分类规则

为了减少研发进度对审核进度的依赖，后续任务默认分两类：

1. `Prep`
   - 低耦合准备工作
   - 包括：
     - 契约草案
     - 数据样本
     - 测试脚手架
     - 只读分析
     - 独立页面壳子
     - 文档和验收入口设计
2. `Core`
   - 依赖上一阶段稳定输出的正式实现
   - 包括：
     - 主链路实现
     - 真实运行时绑定
     - 依赖稳定契约的业务逻辑
     - 正式结算、成长、存档、恢复等核心功能

执行约束：

1. 当前阶段进入 `待人工验收` 后：
   - 可以启动下一阶段 `Prep`
   - 不可以启动下一阶段 `Core`
2. 只有当前阶段通过人工验收后：
   - 才可以把下一阶段 `Core` 作为正式主线推进
3. 如果某项任务原先被当成 `Prep`，但实现中发现其真实依赖上一阶段稳定输出：
   - 必须改判为 `Core`
   - 并停止提前推进

### 3.6 多人协同最小治理字段

从本轮开始，凡是准备多人并行推进的 issue / 切片，至少应补齐以下三个字段：

1. `Owner`
   - 当前谁负责该切片
2. `Write Scope`
   - 当前切片允许改哪些文件、模块或目录
3. `Depends On`
   - 当前切片依赖哪些上游 issue、契约或阶段输出

如果这三个字段未明确：

1. 可以继续分析
2. 可以继续写 `Prep`
3. 不应进入多人并行的 `Core` 实现阶段

## 4. 当前工程的验证顺序

推荐顺序：

1. 静态检查
2. 构建或启动验证
3. 浏览器链路验证
4. 对照组验证
5. 验收清单与交接文档更新

## 5. 当前工程的闭环判定

本工程内，只有同时满足以下条件才视为阶段闭环：

1. 本轮实现完成
2. 已形成验收清单
3. 已形成验收记录或自测报告
4. 已明确告知用户验收入口
5. 用户已明确回复验收结果

如果第 5 条未满足，则阶段只能标记为：

- `已完成实现`
- `待人工验收`
- `待用户确认`

但此时可以继续推进：

- 下一阶段 `Prep`
- 当前阶段剩余的非阻塞性文档、测试和收口工作

## 6. 让所有代码工程都沿用同一策略的办法

当前建议采用两层方案：

### 6.1 全局层

在 Codex 全局技能目录中维护一个可复用 skill：

- `~/.codex/skills/project-engineering-strategy/`

作用：

- 让新工程也能自动套用同一套工程策略
- 把“先理解、后修改；先契约、后页面；必须有验收清单和交接”固化成稳定工作流

### 6.2 项目层

每个项目再落一份本地策略映射文件，说明：

- 本项目文档目录在哪里
- 本项目验收清单放哪里
- 本项目自测报告放哪里
- 本项目当前阶段的特定约束是什么

也就是说：

- 全局 skill 负责跨项目统一方法
- 项目本地文档负责当前项目具体落点
