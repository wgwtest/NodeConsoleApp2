本文件用于把 `/home/wgw/CodexProject/BlueSim/ENGINEERING_WORK_STRATEGY.md` 落地到当前工程，作为当前项目后续开发、自测、验收和交接的执行基线。

## 2. 当前工程的默认落点

- 分析文档：`DOC/CodexAnylyse/`
- 验收清单：`DOC/CodexAnylyse/验收清单/`
- 验收记录：`DOC/CodexAnylyse/验收记录/`
- 自测报告：`DOC/CodexAnylyse/自测报告/`
- 会话交接：`DOC/CodexAnylyse/会话交接/`

## 3. 当前工程的执行规则

### 3.1 开始前

- 先读最近一次会话交接文档
- 先读当前阶段计划、验收清单、验收记录和自测报告
- 先检查工作区状态
- 先明确本轮目标、非目标和验证方式

### 3.2 开发中

- 优先修改运行时、契约和事件链路
- 页面与测试页只做必要兼容，不做无关大改
- 每轮只推进一个明确切片
- 不扩散到与当前目标无关的模块

### 3.3 开发后

- 至少执行静态检查和最小真实链路验证
- 有实际改动时默认新增验收清单
- 有真实测试时默认新增验收记录或自测报告
- 默认新增会话交接文档
- 在用户未明确反馈验收结果前，阶段状态一律记为：
  - `待人工验收`
  - 或 `待用户确认`

## 4. 当前工程的验证顺序

推荐顺序：

1. 静态检查
2. 构建或启动验证
3. 浏览器链路验证
4. 对照组验证
5. 验收清单与交接文档更新

## 5. 当前工程的闭环判定

本工程内，只有同时满足以下条件才视为阶段闭环：

1. 本轮实现完成
2. 已形成验收清单
3. 已形成验收记录或自测报告
4. 已明确告知用户验收入口
5. 用户已明确回复验收结果

如果第 5 条未满足，则阶段只能标记为：

- `已完成实现`
- `待人工验收`

## 6. 让所有代码工程都沿用同一策略的办法

当前建议采用两层方案：

### 6.1 全局层

在 Codex 全局技能目录中维护一个可复用 skill：

- `~/.codex/skills/project-engineering-strategy/`

作用：

- 让新工程也能自动套用同一套工程策略
- 把“先理解、后修改；先契约、后页面；必须有验收清单和交接”固化成稳定工作流

### 6.2 项目层

每个项目再落一份本地策略映射文件，说明：

- 本项目文档目录在哪里
- 本项目验收清单放哪里
- 本项目自测报告放哪里
- 本项目当前阶段的特定约束是什么

也就是说：

- 全局 skill 负责跨项目统一方法
- 项目本地文档负责当前项目具体落点

从本文件创建开始，当前工程默认继续沿用 BlueSim 工程策略，并以本文件作为当前项目落地版本。
