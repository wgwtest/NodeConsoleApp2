## 7. 本轮发现的问题

### F-UI-01 双引擎初始化与资源重复加载

严重性：高

现象：

- 控制台多次出现双份 `Engine initializing...`
- 核心数据资源存在双份请求

定位依据：

- 页面自身在 [mock_ui_v11.html](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/mock_ui_v11.html#L368) 到 [mock_ui_v11.html](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/mock_ui_v11.html#L412) 中显式 `new CoreEngine()`
- `CoreEngine` 文件底部又在 [CoreEngine.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/engine/CoreEngine.js#L1640) 到 [CoreEngine.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/engine/CoreEngine.js#L1650) 创建默认单例并挂到 `window.Engine`

影响：

- 资源重复加载
- UI 与运行态可能绑定到不同实例
- 后续敌人行为、Buff 事件链、存档同步都可能出现“看起来有值，实际实例不一致”的问题

### F-UI-02 进入战斗后关卡选择模态残留可见

严重性：高

现象：

- 选关进入战斗后，页面已进入 `BATTLE_LOOP / PLANNING`
- 但 `选择关卡` 模态内容仍可见
- 运行态辅助验证中 `levelModalVisible = true`

定位依据：

- 关卡卡片是在 [UI_SystemModal.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/ui/UI_SystemModal.js#L372) 到 [UI_SystemModal.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/ui/UI_SystemModal.js#L420) 中直接渲染
- 代码注释假设 `selectLevel` 触发状态切换后会自动隐藏，但本次实测宿主层仍残留可见内容

影响：

- 主流程页面被旧模态污染
- 后续人工验收容易误判页面状态

### F-UI-03 技能树关闭存在焦点管理问题

严重性：中

现象：

- 控制台出现 `Blocked aria-hidden ... descendant retained focus`
- 关闭技能树时，焦点仍停留在关闭按钮上

影响：

- 辅助功能与焦点管理异常
- 继续叠加模态时可能放大交互问题

### F-UI-04 技能目标部位约束未在槽位高亮中生效

严重性：中

现象：

- `裂颅一击` 配置为仅允许 `head`
- 但 UI 槽位高亮了所有敌方部位，而不是仅头部

定位依据：

- 技能配置在 [skills_melee_v4_5.json](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/assets/data/skills_melee_v4_5.json#L511) 到 [skills_melee_v4_5.json](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/assets/data/skills_melee_v4_5.json#L523) 中将 `candidateParts` 和 `selectedParts` 固定为 `head`
- 详情文案也显示“对头部造成伤害”
- 但槽位高亮逻辑在 [UI_SkillPanel.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/ui/UI_SkillPanel.js#L903) 到 [UI_SkillPanel.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/ui/UI_SkillPanel.js#L935) 只读取 `selection.part`，未消费 `candidateParts` / `selectedParts`

影响：

- UI 与技能契约不一致
- 玩家可能把限定头部技能错误部署到其它部位

### F-RUNTIME-05 技能树学习路径写出的是陈旧战斗存档

严重性：高

现象：

- 学习技能后存档中全局技能已更新
- 但运行态仍停留在旧的 `MAIN_MENU / turn=2 / phase=PLANNING`
- 与真实页面回合 `3` 不一致

定位依据：

- 技能树提交逻辑在 [UI_SkillTreeModal.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/ui/UI_SkillTreeModal.js#L540) 到 [UI_SkillTreeModal.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/ui/UI_SkillTreeModal.js#L545) 直接调用 `this.engine.data.saveGame()`
- 该路径绕过了 [CoreEngine.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/engine/CoreEngine.js#L696) 到 [CoreEngine.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/engine/CoreEngine.js#L715) 的 `saveGame()` 包装，因此没有先执行 `saveBattleState()`
- `DataManagerV2.saveGame()` 只直接序列化当前 `dataConfig`，见 [DataManagerV2.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/engine/DataManagerV2.js#L76) 到 [DataManagerV2.js](/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2/script/engine/DataManagerV2.js#L98)

影响：

- 学习技能后若立刻读档，可能回到过期战斗快照
- Buff、敌人队列、回合数、规划状态可能失真

### F-RUNTIME-06 多选技能进入时间线后疑似未完整执行

严重性：高

现象：

- `多攻测试` 已成功放入 4 个敌方槽位
- `提交规划` 后，时间线预览中明确出现 4 个我方动作
- 但执行阶段控制台仅记录 3 条 `skill_1771952147024 -> DONE`
- 4 个目标部位中，`腹部` 未观察到对应变化

当前证据链：

- 规划阶段：4 个槽位填充成功，第 5 个槽位被上限拦截
- 预览阶段：4 个我方动作进入时间线
- 执行阶段：只看到 3 条完成日志

影响：

- 多选技能当前不能被视为稳定可用
- 后续 Buff / 敌人策略若依赖多动作链路，结果可能继续失真
