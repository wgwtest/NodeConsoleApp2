# NodeConsoleApp2 工程总体分析

## 1. 结论先行

当前工程不是“从零开始”，而是一个已经具备主流程骨架、但核心结算层尚未完成新数据协议落地的战斗原型工程。

它已经具备的部分：

- 有可运行的浏览器端静态工程入口：`app.js`
- 有核心状态机与战斗主循环：`script/engine/CoreEngine.js`、`GameFSM.js`、`GameLoop.js`
- 有规划阶段与时间轴执行阶段：`TurnPlanner.js`、`TimelineManager.js`
- 有技能编辑器、Buff 编辑器、测试页面和较完整的设计文档
- 有玩家、敌人、关卡、技能、Buff、槽位布局等数据文件

它当前缺失或未对齐的部分：

- 技能执行层仍主要按旧字段 `skill.type / skill.value / skill.targetType` 结算
- 技能数据 `skills_melee_v4_5.json` 已切到新结构 `actions[].effect`
- Buff 系统、Buff 数据、Buff 执行器三者之间存在明显 schema 脱节
- 敌人行为目前仍是占位逻辑，只会取技能列表第一个技能
- 敌人配置引用的技能 ID 大量不存在于当前技能库

所以，这个工程的真实状态可以定义为：

`UI/流程原型 > 数据设计较完整 > 运行时主流程未闭环`

如果目标是“完整实现技能/buff 在游戏主流程的使用、敌人的行为操作”，一周内是有机会做出可玩的 MVP 闭环的，但前提是以“完成当前数据集的主战斗闭环”为范围，而不是同时做完整数值内容扩展与大型系统重构。

## 2. 工程定位与结构

项目实际代码目录在：

- `NodeConsoleApp2/`

核心目录职责：

- `app.js`
  - 一个极简静态文件服务器，用于本地启动浏览器端原型
- `script/engine/`
  - 核心引擎、数据加载、状态机、时间轴、Buff 运行时
- `script/ui/`
  - 战斗 UI、技能面板、时间轴面板、系统弹窗等
- `script/editor/skill/`
  - 技能编辑器逻辑
- `assets/data/`
  - 玩家、敌人、关卡、技能、Buff、槽位布局等配置数据
- `design/`
  - 设计文档，内容较多，方向明确
- `test/`
  - 原型页、编辑器测试页、Buff 测试页
- `DOC/`
  - 项目推进过程文档

工程形态判断：

- 这是一个浏览器端 ES Module 原型工程，不是 C# 控制台程序
- `NodeConsoleApp2.esproj` 和 `NodeConsoleApp2.slnx` 更像是 Visual Studio 对前端工程的封装
- 当前运行主方式仍然是静态服务器 + HTML 页面

## 3. 当前主流程现状

### 3.1 已实现的主流程骨架

从 `CoreEngine` 来看，当前流程是完整串起来的：

1. `init()`
   - 加载数据
   - 初始化 BuffSystem
   - 启动 GameLoop
2. `login()`
   - 进入主菜单
3. `selectLevel()`
   - 实例化关卡
4. `startBattle()`
   - 初始化玩家/敌人运行时数据
   - 建立 BuffManager
5. `startTurn()`
   - 进入规划阶段
   - 重置 AP
   - 初始化 AP 预算
6. `commitPlanning()`
   - 将玩家草稿转换为时间轴动作
   - 生成敌方动作
   - 载入 Timeline
7. `commitTurn()`
   - 开始执行阶段
8. `executeTurn()`
   - Timeline 逐条执行
9. `_executeTimelineEntry()`
   - 分派到 `executePlayerSkill()` 或 `executeEnemySkill()`
10. `checkBattleStatus()`
   - 判断胜败

从结构上说，战斗循环并不缺。

### 3.2 当前真正缺的是“执行层”

虽然回合规划与时间轴都有，但技能真正落地时，执行器仍然依赖旧版 skill schema：

- `executePlayerSkill()` 使用 `skillConfig.type`、`skillConfig.value`
- `executeEnemySkill()` 也使用 `skillConfig.value`
- 规划阶段的目标合法性判断使用 `skillConfig.type`、`skillConfig.targetType`

但 `skills_melee_v4_5.json` 中：

- 没有 `type`
- 没有 `value`
- 没有 `targetType`
- 统一改为 `target`、`costs`、`actions[].effect`

这意味着：

- 规划 UI 基本已经向新结构靠拢
- 运行时结算仍停留在旧结构
- 主流程的“中段和末段”协议不一致

这是本工程当前最大的断点。

## 4. 技能系统现状分析

### 4.1 数据层

`skills_melee_v4_5.json` 的结构已经相对清晰：

- `target.subject / scope / selection`
- `requirements`
- `costs.ap / costs.partSlot / perTurnLimit`
- `actions[]`
- `buffRefs.apply / remove`

对当前技能库做静态核对后的结论：

- 技能总数：35
- 带 `actions[]` 的技能：29
- 缺少 `costs.ap` 的技能：0
- 缺少 `target` 的技能：0
- 当前 `actions[]` 中实际使用到的 effectType 主要是：
  - `DMG_HP`
  - `DMG_ARMOR`
  - `HEAL`
  - `ARMOR_ADD`
  - `AP_GAIN`
  - `BUFF_REMOVE`

这反而是一个好消息：当前数据虽然多，但真正需要先支持的 effectType 并不多，适合一周内先打通 MVP。

### 4.2 运行时层

`TurnPlanner` 已经开始消费新结构的一部分：

- 读取 `target.selection.selectCount`
- 用 `costs.ap` 建 AP 预算
- 维护按槽位规划的动作

说明“规划层”已经向新 schema 迁移。

但 `CoreEngine` 结算层没有完成同样迁移。

### 4.3 UI 层

`UI_SkillPanel` 明显已经围绕新结构构建：

- 用学习技能列表渲染技能池
- 支持草稿式规划
- 支持提交规划再执行
- 支持按槽位摆放

所以现阶段最合理的工程策略不是推倒重写，而是补齐“技能解释器”和“执行器”。

## 5. Buff 系统现状分析

### 5.1 Buff 运行时代码已存在，但未真正闭环

当前已有：

- `BuffRegistry`
- `BuffManager`
- `BuffSystem`
- 事件接线：
  - `TURN_START`
  - `TURN_END`
  - `BATTLE_ATTACK_PRE`
  - `BATTLE_ATTACK_POST`
  - `BATTLE_TAKE_DAMAGE_PRE`
  - `BATTLE_TAKE_DAMAGE`
  - `BATTLE_ACTION_PRE`

从工程角度说，Buff 运行时框架已经不是空白。

### 5.2 当前有三层明显错位

#### 问题 A：BuffRegistry 读取的数据结构不对

`buffs_v2_7.json` 是包装结构：

```json
{
  "$schemaVersion": "...",
  "meta": {},
  "buffs": {
    "buff_bleed": { ... }
  }
}
```

但 `CoreEngine.init()` 传给 `BuffRegistry.setDefinitions()` 的是整个 `gameConfig.buffs`，不是其中的 `gameConfig.buffs.buffs`。

后果：

- `registry.getDefinition('buff_bleed')` 拿不到定义
- 任何 `buffs.add(buffId)` 都会失败

这属于主阻塞问题。

#### 问题 B：Buff 数据 action 命名与 BuffSystem actionLibrary 不一致

Buff 数据里实际使用的是：

- `DAMAGE_HP`
- `SKIP_TURN`
- `ATTACK`
- `HEAL_ARMOR`
- `PREVENT_DAMAGE_ARMOR`
- `PREVENT_DAMAGE_HP`
- `AP_COST_REDUCE`
- `AP_COST_ADD`

但 `BuffSystem` 里注册的是：

- `damage`
- `heal`
- `applyBuff`
- `skipTurn`
- `modifyAP`
- `absorbDamage`
- `modifyDamageTaken`
- `attack`

问题非常直接：

- 大部分 Buff action 根本匹配不上
- 即使 Buff 被正确加载，也不会正确执行

#### 问题 C：Buff effect 参数结构也不一致

Buff 数据使用：

- `effect.payload.value`
- `effect.payload.valueType`

但 `BuffSystem._resolveValue()` 读取的是：

- `effect.value`

这意味着：

- 就算 action 名字对上了
- 当前很多 effect 的数值仍然取不到

### 5.3 statModifiers 结构也存在偏差

`BuffManager.getEffectiveStat()` 预期的 `statModifiers` 更接近对象结构：

```json
{
  "damageTakenMult": { "type": "percent", "value": 0.2 }
}
```

而 `buffs_v2_7.json` 实际上大多是数组结构：

```json
[
  { "stat": "speed", "type": "flat", "value": -5 }
]
```

这会导致：

- statModifiers 汇总逻辑与数据格式不匹配
- 一部分 Buff 即使存在，也不会正确影响属性

### 5.4 实际结论

Buff 系统现在是“有框架，但还没有统一协议”。

如果你的目标是“完整实现技能/buff 在主流程使用”，那么这部分必须作为本周最高优先级之一，不能后置。

## 6. 敌人系统现状分析

### 6.1 已实现部分

当前已有：

- 敌人模板数据
- 关卡实例化时复制敌人模板为运行时实例
- 敌人参与 Timeline
- 敌人也有 BuffManager
- 敌人可执行 `executeEnemySkill()`

### 6.2 当前敌人行为几乎仍是占位实现

在 `commitPlanning()` 中，敌人动作生成逻辑是：

- 活着就加入队列
- 技能直接取 `enemy.skills[0]`
- 目标固定是玩家
- 没有 AP 校验
- 没有技能选择策略
- 没有目标部位选择策略

这不构成“敌人行为系统”，只是“让时间轴上有个敌方动作占位”。

### 6.3 更严重的问题：敌人引用技能大量不存在

对 `enemies.json` 与 `skills_melee_v4_5.json` 做核对后，发现敌人引用的技能几乎全部缺失：

- `skill_bite`
- `skill_throw_stone`
- `skill_smash`
- `skill_warcry`
- `skill_heavy_smash`
- `skill_rage`
- `skill_cleave`
- `skill_ankle_bite`
- `skill_escape`
- `skill_shield_bash`
- `skill_bone_repair`

这意味着：

- 即使今天补了敌人 AI 规划器
- 也没有足够的技能数据可供敌人正确执行

所以“敌人行为”这一周工作，不能只写 AI 代码，还必须补齐敌人技能数据，或者统一敌我技能库并建立映射。

## 7. 数据与运行时对齐情况

### 7.1 已对齐的部分

- `costs.ap` 已进入 `CoreEngine` AP 预算
- `target.selection.selectCount` 已进入 `TurnPlanner`
- 槽位布局 `slot_layouts.json` 已进入战斗规划层

### 7.2 未对齐的部分

- `skills.actions[].effect` 未进入技能执行器
- `skills.buffRefs` 未进入技能执行器
- `buffs_v2_7.json` 未与 `BuffRegistry / BuffSystem / BuffManager` 对齐
- 敌人技能数据与技能库未对齐
- 敌人行为规则未独立抽象成 Planner/AI 层

## 8. 风险判断

### 8.1 一周内可行的目标

可行目标是：

- 打通当前技能 schema 到运行时结算
- 打通 Buff 基础执行链路
- 完成一套可解释、可复现的敌人行为逻辑
- 让测试关卡从“能规划”变成“能完整打完”

### 8.2 一周内不建议承诺的目标

以下目标如果同时追求，风险会明显升高：

- 覆盖所有未来 Buff action 枚举
- 做完整复杂 AI，例如多层行为树、战术记忆、仇恨系统
- 同时重做 UI 视觉与工具链
- 同时推进编辑器 schema 全量重构

### 8.3 最合理的交付定义

建议把本周交付定义为：

“完成当前战斗主流程的可玩闭环：玩家技能、Buff 生效、敌人可决策并执行、回合可完整结算、关键技能与 Buff 有测试覆盖。”

这个目标是务实且可验证的。

## 9. 推荐实现范围

为了在一周内实现稳定交付，建议本周只保证以下技能 effectType 先完整支持：

- `DMG_HP`
- `DMG_ARMOR`
- `HEAL`
- `ARMOR_ADD`
- `AP_GAIN`
- `BUFF_REMOVE`
- 通过 `buffRefs.apply/remove` 实现 `BUFF_APPLY/BUFF_REMOVE`

建议本周优先支持以下 Buff 能力：

- `DAMAGE_HP`
- `SKIP_TURN`
- `ATTACK`
- `PREVENT_DAMAGE_HP`
- `PREVENT_DAMAGE_ARMOR`
- `AP_COST_ADD`
- `AP_COST_REDUCE`
- 基础 statModifiers：`flat`、`percent_base/percent`、`overwrite`

建议本周优先支持以下敌人行为能力：

- 技能合法性过滤
- 基于 AP 的技能选择
- 基于技能类型的行为倾向选择
- 基于玩家护甲/弱点的目标部位选择
- 低血量防御/治疗倾向
- 默认 fallback 行为

## 10. 关键判断

如果按当前代码基础推进，这个工程不需要大改架构，核心是补三块：

1. `技能解释器`
   - 把 `skills_melee_v4_5.json` 正式接入执行器
2. `Buff 协议统一层`
   - 统一数据 schema 与运行时代码
3. `敌人行动规划器`
   - 把敌人从“固定取第一个技能”升级成规则驱动行为

只要这三块落下去，工程就会从“设计原型”跃迁成“可玩的战斗原型”。
