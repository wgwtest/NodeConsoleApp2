# 基于修复后深测结论的后续研发计划

创建时间：2026-03-28 01:13

状态：`待用户审定`

## 1. 当前阶段判断

主流程底座已经从“可跑”提升到“可稳定回归”：

1. 宿主实例一致
2. 模态生命周期稳定
3. 技能树学习与战斗保存一致
4. 单选技能、多选技能、敌方自动行为都已有真实页面证据

因此下一轮可以从“修底座”切换到“扩行为覆盖”。

## 2. 下一轮目标

下一轮不再优先做页面宿主修补，而是围绕用户最终目标推进两件事：

1. 完整实现技能 / Buff 在游戏主流程中的稳定使用
2. 扩展敌人的行为操作，让敌人不再只有单一路径

## 3. 建议研发顺序

### 阶段 A：Buff 运行时专项

目标：

- 把当前 Buff 数据定义、触发事件、运行时落地结果逐项打通

实施重点：

1. 建立 Buff 效果类型清单与当前实现映射表
2. 按触发链路补测：
   - `BATTLE_ACTION_PRE`
   - `BATTLE_ATTACK_PRE`
   - `BATTLE_TAKE_DAMAGE_PRE`
   - `BATTLE_ATTACK_POST`
3. 覆盖高价值效果：
   - 反击
   - 跳过回合
   - AP 消耗增加 / 减少
   - 护甲恢复
   - 伤害拦截 / 免伤

完成判定：

- 至少 1 条主动技能 + 1 条 Buff 联动链路可以在主流程真实触发并被记录

### 阶段 B：敌人策略扩展专项

目标：

- 从“会自动攻击”扩展到“会根据状态选择技能和目标”

实施重点：

1. 扩展敌人候选技能评分逻辑
2. 增加以下决策倾向：
   - 低血量自保 / 恢复
   - 护甲修复
   - 针对玩家薄弱部位攻击
   - 对已破甲部位追击
3. 在多敌人场景中验证：
   - 不同敌人可选不同技能
   - 时间线排序仍然稳定

完成判定：

- 至少 3 类敌人技能选择策略有真实战斗证据

### 阶段 C：战斗恢复与回归扩面

目标：

- 让后续迭代具备更高回归效率和更可靠的读档验证

实施重点：

1. 补战斗中保存 -> 刷新 -> 继续战斗的读档回归
2. 扩大自动回归脚本覆盖：
   - 单选技能
   - 多选技能
   - 技能树学习保存
   - 战斗存档恢复
3. 视情况增加固定测试技能 / 固定测试敌人样本

完成判定：

- 至少 1 条战斗中读档恢复链路通过

## 4. 风险判断

### 高风险

1. Buff 数据定义和运行时事件语义可能仍有脱节
2. 敌人策略一旦扩展到恢复 / 防御，会暴露更多 AP / 目标契约边界问题

### 中风险

1. 读档恢复可能会暴露更多 runtime snapshot 缺口
2. 多敌人场景可能会引出新的时间线排序问题

## 5. 建议验收方式

建议按“Buff 专项 -> 敌人策略专项 -> 读档恢复专项”三段验收，不建议再混成一轮大包：

1. 每个阶段结束都产出新的：
   - 自测报告
   - 验收清单
   - 交接记录
2. 每段通过后再进下一段

## 6. 我对下一轮可行性的判断

现在进入 Buff / 敌人行为扩展是可行的，原因是：

1. 主流程底座已可重复验证
2. 多选技能的关键执行一致性已经收敛
3. 学习保存不再污染战斗态样本

因此下一轮工作不需要再把大量时间花在宿主修补上，可以直接切入行为层实现。
